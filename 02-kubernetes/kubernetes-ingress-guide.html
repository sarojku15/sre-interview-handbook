Kubernetes Ingress — Complete Guide

Ingress is a Kubernetes API object that defines HTTP/HTTPS routing rules. The Ingress Controller is the software that actually implements those rules and handles traffic.

Corrected from the original poster: the sample YAML's admin path had backend.service written as a list item (- name: admin) — that's invalid; service must be a plain mapping like the other two paths. Fixed below in section 10.

1. What is Ingress?

  • A Kubernetes resource that manages external access to Services in a cluster — typically HTTP/HTTPS.
  • Provides L7 (application-layer) routing based on hostnames, paths, or other rules.

2. Why do we need it?

  • One external entry point (Load Balancer) for many Services.
  • Cost saving — one LB instead of one-per-Service.
  • Centralized routing, TLS termination, security policies.
  • Advanced routing: host/path based, rewrites, rate limiting, auth, WAF.

If no controller is installed, Ingress objects sit in etcd and do nothing — the single most common beginner trap.

Traffic flow: Internet → Load Balancer → Ingress Controller → Service → Pods — TLS termination happens at the Ingress Controller.

6a. Routing type — host-based

host: shop.company.com  → Frontend Service
host: api.company.com   → API Service

Different hostnames route to entirely different Services — one Ingress, many domains.

6b. Routing type — path-based

Host: company.com
/shop     → Shop Service
/payment  → Payment Service
/admin    → Admin Service

One hostname, multiple paths fan out to different backend Services.

7. TLS / SSL termination

  • HTTPS traffic arrives from the internet.
  • The Ingress Controller holds the certificate (a Kubernetes Secret).
  • It terminates (decrypts) HTTPS and forwards plain HTTP to backend Services — re-encryption to the Pod is optional but preferred for regulated workloads.
Client — HTTPS→Ingress Controller
(TLS termination)→Service — HTTP

8. Default backend

  • If a request doesn't match any rule (host/path), it goes to the default backend.
  • Usually returns 404 Not Found.
Unknown host/path→Ingress→Default backend (404)

9. IngressClass

Specifies which Ingress Controller should process a given Ingress — required when more than one controller runs in a cluster.

spec:
  ingressClassName: nginx

Useful commands

kubectl get ingress -AList all Ingress objects
kubectl describe ingress <name>Rules, backend, events
kubectl get pods -n ingress-nginxController Pod status
kubectl logs -n ingress-nginx <pod>Controller logs
kubectl get svc -n ingress-nginxController's LB Service

10. Sample Ingress YAML corrected

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: ecommerce
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  tls:
    - hosts:
        - shop.company.com
      secretName: tls-secret
  rules:
    - host: shop.company.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: frontend
                port:
                  number: 80
          - path: /payment
            pathType: Prefix
            backend:
              service:
                name: payment
                port:
                  number: 80
          - path: /admin
            pathType: Prefix
            backend:
              service:
                name: admin
                port:
                  number: 80

Highlighted line: the original had - name: admin (a list item) instead of a plain name: admin mapping — that would fail schema validation.

11. SSL certificate (TLS Secret)

tls:
  - hosts:
      - shop.company.com
    secretName: tls-secret

The Secret must exist in the same namespace as the Ingress.

# Create the secret
kubectl create secret tls tls-secret \
  --cert=cert.pem --key=key.pem

15. Best practices

  • Use a reliable Ingress Controller (NGINX / ALB / AGIC).
  • Enable TLS for all public endpoints.
  • Store certificates in Kubernetes Secrets, or automate with cert-manager.
  • Use readiness & liveness probes so traffic only hits healthy Pods.
  • Prefer host-based routing for multi-domain applications.
  • Use path-based routing to consolidate related services.
  • Enable access logs and monitor 4xx/5xx response rates.
  • Implement rate limiting, authentication, and WAF for internet-facing apps.
  • Keep Ingress rules simple, maintainable, and version-controlled (GitOps).

Key points: Ingress = rules (YAML) · Ingress Controller = the implementation (Pods) · one LB, many Services · L7 routing, TLS, and security all live at this layer.
