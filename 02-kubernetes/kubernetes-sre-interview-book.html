Chapter 1

Kubernetes Fundamentals & Architecture

This chapter establishes the foundations: what Kubernetes is, the problem it solves, the declarative model, the control plane versus worker node split, and the high-level architecture a Senior SRE keeps in mind when reasoning about cluster behavior.

Question 1 — What is Kubernetes?

Short Answer

Kubernetes is an open-source container orchestration platform that automates deployment, scaling, networking, and lifecycle management of containerized workloads.

Detailed Explanation

Containers solved packaging, but running thousands of them across many machines by hand is impossible. Kubernetes lets you declare a desired state and continuously reconciles the cluster toward it, providing self-healing, automated rollouts, service discovery, load balancing, and horizontal scaling.

Interview Answer

Kubernetes is a container orchestrator where you declare the desired state of your workloads and the platform's control loops continuously make reality match that declaration, handling scheduling, scaling, healing, and networking automatically.

Question 2 — What problem does Kubernetes solve?

Short Answer

It automates the operational burden of running containers at scale across a fleet of machines.

Detailed Explanation

Without orchestration you must manually place containers, restart failures, scale up under load, wire up networking, and roll out new versions safely. Kubernetes provides all of this declaratively, so operations become repeatable and self-healing rather than manual and error-prone.

Interview Answer

Kubernetes removes the manual toil of scheduling, healing, scaling, networking, and rolling out containers, turning fragile hand-managed deployments into a declarative, self-healing system.

Question 3 — What is declarative versus imperative management?

Short Answer

Imperative commands tell Kubernetes the exact steps to perform now; declarative manifests describe the desired end state and let Kubernetes figure out the steps.

Detailed Explanation

Imperative commands like kubectl run or kubectl scale are good for quick fixes. Declarative management applies YAML with kubectl apply, which is idempotent and repeatable. Production uses declarative manifests stored in Git so the desired state is reviewable, auditable, and reproducible.

Comparison

Imperative is procedural and one-shot; declarative is idempotent and continuously reconciled. kubectl apply is declarative and safe to re-run; kubectl create is imperative and fails if the object already exists.

Interview Answer

Imperative tells Kubernetes how to act step by step; declarative describes the desired end state in manifests that Kubernetes reconciles toward. Production uses declarative plus GitOps for idempotency and auditability.

Question 4 — What are the main components of a Kubernetes cluster?

Short Answer

A control plane (API server, etcd, scheduler, controller manager) and worker nodes (kubelet, kube-proxy, container runtime), plus add-ons like CoreDNS and a CNI plugin.

Detailed Explanation

The control plane makes global decisions and stores cluster state. Worker nodes run the actual workloads. Add-ons provide DNS (CoreDNS), pod networking (CNI), and metrics (metrics-server).

Interview Answer

A cluster splits into a control plane that decides and stores state, and worker nodes that run workloads via the kubelet, kube-proxy, and a container runtime, with add-ons supplying DNS, networking, and metrics.

Question 5 — What is a Namespace?

Short Answer

A Namespace is a virtual cluster inside a physical cluster used to scope and isolate groups of resources.

Detailed Explanation

Namespaces separate teams or environments, allow per-tenant ResourceQuotas and LimitRanges, scope RBAC, and prevent name collisions. Not everything is namespaced — Nodes, PersistentVolumes, and ClusterRoles are cluster-scoped.

Common Mistake

Assuming everything is namespaced. Nodes, PersistentVolumes, ClusterRoles, and StorageClasses are cluster-scoped and not bound to any namespace.

Interview Answer

A Namespace logically partitions a cluster so resources can be isolated and scoped per team or environment, enabling quotas, RBAC boundaries, and collision-free naming, while cluster-scoped objects like Nodes live outside namespaces.

Question 6 — What is the difference between Kubernetes and Docker?

Short Answer

Docker builds and runs individual containers; Kubernetes orchestrates many containers across many hosts.

Detailed Explanation

Docker is a container runtime and image tooling for a single host. Kubernetes schedules, scales, networks, and heals containers across a cluster, and uses a container runtime such as containerd underneath to actually run them.

Interview Answer

Docker packages and runs a container on one host; Kubernetes orchestrates fleets of containers across many hosts, using a runtime like containerd to execute them.

Production Scenario 1 — A New Team Asks Why They Need Kubernetes

They run three containers on one VM with a restart script. The honest answer: at that scale Kubernetes may be overkill. The value appears when you need multi-host scheduling, self-healing across failures, zero-downtime rollouts, autoscaling, and consistent declarative operations — the toil Kubernetes removes grows with scale.

Production Scenario 2 — Choosing Declarative Over Imperative in Production

A team fixes incidents with imperative kubectl edit commands, then loses those changes on the next apply. Moving to declarative manifests in Git makes every change reviewed, reproducible, and recoverable, eliminating configuration drift between what is running and what is described.

  2. When is Kubernetes the wrong choice for a workload?
  3. Why is GitOps a natural fit for Kubernetes' declarative model?
  4. What cluster-scoped resources exist and why aren't they namespaced?

Quick Answer Bank

  1. What is Kubernetes? A container orchestration platform that reconciles workloads toward a declared desired state.
  2. What problem does it solve? The manual toil of scheduling, healing, scaling, and networking containers at scale.
  3. Declarative vs imperative? Declarative describes the end state; imperative lists the steps.
  4. What does kubectl apply do? Declaratively reconciles resources, idempotently.
  5. What is a Namespace? A virtual partition for scoping and isolating resources.
  6. Name three cluster-scoped objects. Nodes, PersistentVolumes, ClusterRoles.
  7. Control plane vs worker node? Control plane decides and stores state; worker nodes run workloads.
  8. What are the control plane components? API server, etcd, scheduler, controller manager.
  9. What are the worker node components? kubelet, kube-proxy, container runtime.
  10. Kubernetes vs Docker? Docker runs a container; Kubernetes orchestrates many across hosts.
  11. What is self-healing? Controllers recreate failed Pods to match desired state.
  12. What is GitOps? Storing desired state in Git as the source of truth.
  13. What provides cluster DNS? CoreDNS.
  14. What provides Pod networking? A CNI plugin.
  15. Why is declarative preferred in production? Idempotent, reviewable, and reproducible.

Best Practices

Store all manifests declaratively in Git and apply them through a pipeline. Use namespaces to separate environments and teams, with ResourceQuotas and RBAC per namespace. Avoid imperative edits in production; treat the cluster as reconciled from source control. Keep cluster-scoped changes (RBAC, StorageClasses, CRDs) under strict review.

Chapter 2

Control Plane Deep Dive

The control plane is the brain of Kubernetes. This chapter covers the API server, etcd, scheduler, controller manager, and the request lifecycle of authentication, authorization, and admission control — the components and flows interviewers probe most heavily.

Question 1 — What is the Kubernetes control plane?

Short Answer

The control plane is the set of components that maintain the desired state of the cluster and make all global decisions.

Detailed Explanation

It contains the API server, etcd, scheduler, and controller manager. Every decision — scheduling, scaling, reconciliation — originates here. Worker nodes execute workloads but make no global decisions.

Interview Answer

The control plane maintains desired state by handling API requests, scheduling, and reconciliation, and stores everything in etcd, while worker nodes simply run the workloads it assigns.

Question 2 — What is the API server and why is it central?

Short Answer

The API server is the front door of Kubernetes; every request passes through it, and it is the only component that talks to etcd.

Detailed Explanation

It authenticates, authorizes, runs admission control, validates objects, and persists desired state to etcd. Even the scheduler and controllers never touch etcd directly. It is stateless, so it runs as multiple replicas behind a load balancer for high availability.

Common Mistake

Saying the scheduler or kubelet reads etcd directly. Only the API server communicates with etcd; everything else goes through the API server.

Interview Answer

The API server is the central, stateless gateway that authenticates, authorizes, validates, and admits every request before persisting it to etcd, and it is the only component that reads or writes etcd.

Question 3 — What is etcd and why is it critical?

Short Answer

etcd is a distributed, strongly consistent key-value store that holds the entire cluster state and serves as the single source of truth.

Detailed Explanation

It stores Pods, Deployments, Secrets, ConfigMaps, Nodes, RBAC, and leases. It uses the Raft consensus algorithm, so it runs as an odd-numbered quorum of three or five members to tolerate failures. Losing etcd without a backup means rebuilding the cluster.

Interview Answer

etcd is the strongly consistent key-value store that is Kubernetes' source of truth, holding all cluster state, run as an odd-numbered Raft quorum, and accessed only by the API server, which is why regular snapshots are essential.

ETCDCTL_API=3 etcdctl snapshot save backup.db \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=ca.crt --cert=server.crt --key=server.key

Question 4 — What does the scheduler do and how does it decide?

Short Answer

The scheduler watches for unscheduled Pods and selects the best node using a filtering phase and a scoring phase.

Detailed Explanation

Filtering eliminates nodes that cannot fit the Pod — insufficient CPU or memory, taints without tolerations, unsatisfied node selectors. Scoring ranks the survivors by least load, affinity, and topology spread. The highest-scoring node is chosen and the Pod is bound to it.

Interview Answer

The scheduler decides where, not how, a Pod runs: it filters out nodes that can't satisfy the Pod's constraints, scores the rest for best fit using resources, affinity, and topology, and binds the Pod to the winner.

Question 5 — What is the controller manager and what is a control loop?

Short Answer

The controller manager runs control loops that continuously reconcile actual cluster state to the desired state stored in etcd.

Detailed Explanation

Each controller — Deployment, ReplicaSet, Node, Job — observes actual state, compares it to desired state, and acts to close the gap. If desired replicas are five and only four exist, the ReplicaSet controller creates one more. This continuous reconciliation is what makes Kubernetes self-healing.

Interview Answer

The controller manager runs the reconciliation loops that watch actual state, compare it to the desired state in etcd, and take corrective action when they diverge, which is the mechanism behind self-healing.

Question 6 — Explain authentication, authorization, and admission control.

Short Answer

Every API request passes three gates in order: authentication verifies identity, authorization checks permissions, and admission control can mutate or reject the request.

Detailed Explanation

Authentication uses client certs, bearer tokens, OIDC, or service account tokens. Authorization is usually RBAC. Admission runs mutating webhooks first (which can change objects, such as injecting sidecars) and then validating webhooks and policies, which can reject the request before it is persisted.

Interview Answer

A request is authenticated to confirm who you are, authorized to confirm what you may do, then passed through mutating and validating admission controllers before the object is stored in etcd.

Production Scenario 1 — The Control Plane Is Down but Apps Keep Serving

The API server is unavailable. Developers cannot deploy or scale, yet customer traffic is unaffected because running Pods are maintained locally by each kubelet. The lesson: control plane outages stop cluster management, not the data plane, so the priority is restoring the API server and etcd quorum without panicking about running workloads.

Production Scenario 2 — etcd Loses Quorum

Two of three etcd members fail. The cluster can no longer commit writes, so deployments hang. Because etcd is an odd-numbered Raft quorum, one failure is survivable but two is not. Recovery is restoring a member or, if quorum is unrecoverable, restoring from the latest snapshot — which is why frequent etcd backups are non-negotiable.

Senior Interview Questions

  1. Why must the API server be highly available, and what breaks if it isn't?
  2. Why does everything route through the API server instead of talking to etcd directly?
  3. What is the difference between a mutating and a validating admission webhook?
  4. Why is etcd run as an odd number of members?
  5. How do controllers detect drift between desired and actual state?

Quick Answer Bank

  1. What is the control plane? The components that maintain desired state and make global decisions.
  2. What is the API server? The central gateway for all requests and the only writer to etcd.
  3. What is etcd? The strongly consistent key-value store that is the source of truth.
  4. Which component talks to etcd? Only the API server.
  5. What does the scheduler decide? Which node a Pod runs on.
  6. What are the scheduler's two phases? Filtering and scoring.
  7. What is the controller manager? The runner of reconciliation control loops.
  8. What is a control loop? Observe, compare to desired, act to close the gap.
  9. What are the three request gates? Authentication, authorization, admission.
  10. What does a mutating webhook do? Modifies a request, e.g. injects a sidecar.
  11. What does a validating webhook do? Accepts or rejects a request.
  12. Why is the API server stateless? So it can run as HA replicas.
  13. How many etcd members for HA? An odd number, typically three or five.
  14. What consensus algorithm does etcd use? Raft.
  15. What happens to running Pods if the control plane fails? They keep running; you just can't manage them.

Best Practices

Run an HA control plane with at least three control plane nodes and an odd-numbered etcd quorum. Take frequent, monitored etcd snapshots and test restores. Watch etcd disk fsync latency, since etcd is extremely sensitive to slow disks. Front the API server with a load balancer and protect it with RBAC and admission policies.

Interview Cheat Sheet

API server: central gateway, only etcd client. etcd: source of truth, Raft quorum. Scheduler: filter then score, binds Pod to node. Controller manager: reconciliation loops. Request flow: authn → authz → admission (mutating then validating) → etcd.

Chapter 3

Worker Nodes & Container Runtime

Worker nodes are where workloads actually run. This chapter covers the kubelet, kube-proxy, and the container runtime interface, plus the common misconceptions interviewers use to separate surface knowledge from real understanding.

Question 1 — What is the kubelet?

Short Answer

The kubelet is the per-node agent that ensures the Pods assigned to its node are running and healthy.

Detailed Explanation

It registers the node, watches the API server for Pods bound to it, drives the container runtime through the CRI to start and stop containers, runs liveness, readiness, and startup probes, mounts volumes, and reports node and Pod status back to the control plane.

Common Mistake

Thinking the kubelet schedules Pods. It only runs Pods already assigned to its node by the scheduler; it does not make placement decisions.

Interview Answer

The kubelet is the node agent that watches the API server for its assigned Pods, drives the container runtime to run them, executes probes, mounts volumes, and reports status back to the control plane.

Question 2 — What is kube-proxy, and what is it not?

Short Answer

kube-proxy programs iptables or IPVS rules to implement Service networking; it does not proxy application HTTP traffic in normal mode.

Detailed Explanation

It watches Services and EndpointSlices and configures the node's networking so traffic to a Service's virtual IP is load-balanced to healthy backend Pods. Modern clusters may use IPVS for scale or replace kube-proxy entirely with an eBPF dataplane like Cilium.

Comparison

iptables mode uses sequential rule chains and scales poorly with thousands of Services; IPVS mode uses a hash table and scales far better; eBPF dataplanes bypass kube-proxy entirely.

Interview Answer

kube-proxy implements Services at the network layer by programming iptables or IPVS rules that load-balance traffic from a Service's virtual IP to healthy Pods, rather than proxying HTTP itself.

Question 3 — What is the container runtime and the CRI?

Short Answer

The container runtime pulls images and runs containers; the kubelet talks to it through the Container Runtime Interface, a standard gRPC API.

Detailed Explanation

Docker was the original default but dockershim was removed in Kubernetes 1.24. Modern clusters use containerd or CRI-O, both CRI-native. Docker-built images still run fine because they are OCI-compliant.

Common Mistake

Claiming Docker images stopped working after dockershim removal. Removing the Docker shim changed the runtime, not the image format; OCI images run unchanged on containerd.

Interview Answer

The container runtime downloads images and runs containers, driven by the kubelet over the CRI gRPC interface; since 1.24 clusters use containerd or CRI-O, and Docker images still work because they're OCI-compliant.

Question 4 — What is a static Pod?

Short Answer

A static Pod is a Pod the kubelet runs directly from a local manifest file, without the scheduler or API server creating it.

Detailed Explanation

The kubelet watches a configured directory and runs any Pod manifests found there. This is how control plane components are bootstrapped in kubeadm clusters. A mirror Pod is created in the API server so the static Pod is visible, but it is managed by the kubelet, not a controller.

Interview Answer

A static Pod is managed directly by the kubelet from a local manifest rather than by the control plane, which is how kubeadm bootstraps control plane components like the API server itself.

Production Scenario 1 — A Node Goes NotReady

A node stops reporting and shows NotReady. describe node reveals the kubelet's last heartbeat is stale. The cause is usually a stopped kubelet, a failed container runtime, a full disk, or lost API-server connectivity. After the eviction timeout the control plane reschedules the node's Pods elsewhere, so the fix is restoring the kubelet and runtime on the node.

Production Scenario 2 — Service Traffic Fails on One Node Only

Pods are healthy but requests through the Service fail on a single node. kube-proxy on that node is unhealthy or its iptables rules are stale, so traffic to the Service IP isn't being routed. Restarting kube-proxy and verifying the dataplane resolves it, illustrating that Service routing is a per-node concern.

Senior Interview Questions

  1. Why was dockershim removed and what replaced it?
  2. When would you choose IPVS over iptables mode for kube-proxy?
  3. How does a static Pod differ from a Pod managed by a Deployment?
  4. What happens to a node's Pods when it stays NotReady past the eviction timeout?

Quick Answer Bank

  1. What is the kubelet? The per-node agent that runs assigned Pods and reports status.
  2. Does the kubelet schedule Pods? No, it only runs ones the scheduler assigned.
  3. What is kube-proxy? The component that programs Service networking rules.
  4. Does kube-proxy proxy HTTP? No, it programs iptables or IPVS rules.
  5. iptables vs IPVS? IPVS scales better at large Service counts.
  6. What is the container runtime? The component that pulls images and runs containers.
  7. What is the CRI? The gRPC interface between kubelet and runtime.
  8. What replaced Docker as the runtime? containerd or CRI-O.
  9. Do Docker images still work? Yes, they're OCI-compliant.
  10. What is a static Pod? A Pod run directly by the kubelet from a local manifest.
  11. How does kubeadm bootstrap the control plane? As static Pods.
  12. What does describe node show for a down node? A stale kubelet heartbeat and conditions.
  13. What can replace kube-proxy? An eBPF dataplane like Cilium.

Best Practices

Keep kubelet and runtime versions aligned with the control plane version and within the supported skew. Monitor node conditions for MemoryPressure, DiskPressure, and PIDPressure. Prefer IPVS or an eBPF dataplane for large clusters. Treat NotReady nodes by checking kubelet, runtime, disk, and network before deleting them.

Interview Cheat Sheet

kubelet: node agent, runs assigned Pods, runs probes. kube-proxy: programs iptables/IPVS Service rules. Runtime: containerd/CRI-O via CRI. Docker removed in 1.24; OCI images unaffected. Static Pod: kubelet-managed from local manifest.

Chapter 4

Pods & Lifecycle

Pods are the atomic unit of Kubernetes. This chapter covers Pod structure, restart policies, probes, init and sidecar containers, QoS classes, and the failure states such as CrashLoopBackOff that dominate real interviews.

Question 1 — What is a Pod?

Short Answer

A Pod is the smallest deployable unit — one or more containers that share a network namespace, IP, and optionally storage.

Detailed Explanation

Containers in a Pod share the same IP and port space and communicate over localhost. Kubernetes schedules Pods, not bare containers, so it can attach networking, storage, lifecycle, and probes uniformly. Most Pods run a single main container; multi-container Pods use the sidecar pattern.

Interview Answer

A Pod is the smallest schedulable unit, a group of co-located containers sharing a network namespace and IP so Kubernetes can manage networking, storage, and lifecycle consistently rather than handling raw containers.

Question 2 — Explain liveness, readiness, and startup probes.

Short Answer

Liveness restarts a hung container, readiness gates whether a Pod receives traffic, and startup protects slow-starting apps until they're up.

Detailed Explanation

A failing liveness probe makes the kubelet restart the container. A failing readiness probe removes the Pod from Service endpoints without restarting it. A startup probe disables liveness and readiness until the app has finished starting, preventing slow boots from being killed. The classic outage is an aggressive liveness probe restarting healthy but busy Pods.

Interview Answer

Liveness detects a dead container and restarts it, readiness decides if a Pod should receive Service traffic without restarting it, and startup gives slow apps time before the others apply; misusing liveness where readiness was needed causes restart loops.

readinessProbe:
  httpGet: { path: /healthz, port: 8080 }
  periodSeconds: 10
livenessProbe:
  httpGet: { path: /livez, port: 8080 }
  failureThreshold: 3
  periodSeconds: 15

Question 3 — What are init containers and sidecar containers?

Short Answer

Init containers run to completion before the main containers start; sidecars run alongside the main container for the Pod's lifetime.

Detailed Explanation

Init containers run sequentially for setup such as migrations or waiting on a dependency, and the Pod won't start its main containers until they succeed. Sidecars provide ongoing support such as log shippers, mesh proxies, or exporters. Since Kubernetes 1.28, native sidecars are init containers with restartPolicy Always.

Interview Answer

Init containers run setup tasks to completion before the app starts, while sidecars run beside the app for its whole lifetime; native sidecars in 1.28+ are always-restarting init containers.

Question 4 — Explain Pod QoS classes.

Short Answer

Kubernetes assigns Guaranteed, Burstable, or BestEffort based on a Pod's requests and limits, which determines eviction order under node pressure.

Detailed Explanation

Guaranteed requires requests equal to limits for every container and is evicted last. Burstable has some requests set and is evicted in the middle. BestEffort sets nothing and is evicted first under memory pressure. QoS is derived automatically, not set directly.

Common Mistake

Thinking you set QoS directly. It is inferred from the requests and limits you specify on the Pod's containers.

Interview Answer

QoS is computed from requests and limits: Guaranteed (requests equal limits), Burstable (some requests), and BestEffort (none). Under memory pressure the kubelet evicts BestEffort first, so critical workloads should be Guaranteed.

Question 5 — What is restartPolicy and how does it relate to controllers?

Short Answer

restartPolicy controls how the kubelet restarts containers within a Pod; replacing a whole Pod is the controller's job.

Detailed Explanation

Values are Always, OnFailure, and Never. Deployments require Always; Jobs use OnFailure or Never. When a Pod is deleted, the controller creates a replacement — restartPolicy only governs container restarts inside an existing Pod. Repeated failed restarts under Always surface as CrashLoopBackOff.

Interview Answer

restartPolicy governs container restarts inside a Pod — Always for Deployments, OnFailure or Never for Jobs — while replacing a failed Pod is handled by its controller, and repeated container failures appear as CrashLoopBackOff.

Question 6 — What does CrashLoopBackOff mean?

Short Answer

CrashLoopBackOff means a container keeps starting and exiting, and the kubelet is applying an increasing back-off between restarts.

Detailed Explanation

It is a symptom, not a root cause. Common causes are missing config or secrets, a failing startup dependency, a bad entrypoint, an OOMKill, or an over-aggressive liveness probe. You diagnose with kubectl logs --previous and the container exit code from describe.

Interview Answer

CrashLoopBackOff is the kubelet backing off while a container repeatedly crashes on start; you find the cause in the previous container's logs and the exit code, then fix the underlying config, dependency, or resource issue.

kubectl logs <pod> --previous
kubectl describe pod <pod>   # exit code, events

Production Scenario 1 — Aggressive Liveness Probe Causes Restart Storms

Under peak load a service slows briefly, its liveness probe times out, and the kubelet restarts it — dropping in-flight requests and worsening load, which triggers more restarts. The fix is to gate traffic with a readiness probe and make liveness tolerant, since liveness should detect a dead process, not a busy one.

Production Scenario 2 — Exit Code 137 in Production

A container is repeatedly killed with exit code 137. That is SIGKILL from an OOMKill: the container exceeded its memory limit. The fix is to right-size the memory limit, fix the leak, or set requests and limits so the Pod becomes Guaranteed and is less likely to be evicted.

  1. Why should readiness, not liveness, gate traffic during warm-up?
  2. How do native sidecars in 1.28+ differ from a plain extra container?
  3. Which Pods are evicted first under node memory pressure and why?
  4. What does exit code 137 indicate and how do you remediate it?
  5. How do init containers affect Pod startup ordering?

Quick Answer Bank

  1. What is a Pod? The smallest unit, sharing network and IP among its containers.
  2. Do Pod containers share an IP? Yes.
  3. What does a liveness probe do? Restarts a hung container.
  4. What does a readiness probe do? Gates Service traffic without restarting.
  5. What does a startup probe do? Delays liveness/readiness for slow starters.
  6. What is an init container? A setup container that runs to completion first.
  7. What is a sidecar? A container running alongside the app for its lifetime.
  8. What are the QoS classes? Guaranteed, Burstable, BestEffort.
  9. Which QoS is evicted first? BestEffort.
  10. How is QoS set? Inferred from requests and limits.
  11. What are restartPolicy values? Always, OnFailure, Never.
  12. Which restartPolicy do Deployments need? Always.
  13. What is CrashLoopBackOff? Repeated container crashes with kubelet back-off.
  14. What does exit code 137 mean? OOMKilled by SIGKILL.
  15. How do you debug a crashing Pod? logs --previous and the exit code.

Best Practices

Always set resource requests and limits to control scheduling and QoS. Use readiness probes to gate traffic and keep liveness probes tolerant. Aim for Guaranteed QoS on critical workloads. Use init containers for ordered setup and native sidecars for supporting processes. Diagnose crashes with previous-instance logs before changing config.

Chapter 5

Labels, Selectors & Annotations

Labels and selectors are how Kubernetes groups and connects objects. This short but high-yield chapter covers the difference between labels and annotations and how Services use selectors to find Pods.

Question 1 — What is the difference between labels and annotations?

Short Answer

Labels are identifying metadata used by selectors to group objects; annotations are non-identifying metadata for tools and humans and cannot be selected on.

Detailed Explanation

Selectors in Services, ReplicaSets, and Deployments match Pods by label. Annotations hold arbitrary data such as build info, change-cause, checksums, or ingress configuration. If something needs to select or group an object it must be a label; otherwise it is an annotation.

Comparison

Labels are queryable and bounded; annotations are free-form and can be large. Services select Pods by label, never by annotation.

Interview Answer

Labels identify and group objects and are matched by selectors, while annotations attach arbitrary non-identifying data for tooling and can't be used in selectors.

Question 2 — How does a Service know which Pods to route to?

Short Answer

A Service uses a label selector; the endpoints controller adds ready Pods whose labels match to the Service's EndpointSlices.

Detailed Explanation

kube-proxy then load-balances traffic to those endpoints. If the Pod template's labels don't match the Service selector, the Service has no endpoints and traffic fails. This selector-versus-label mismatch is one of the most common networking bugs.

Interview Answer

A Service's label selector is matched against ready Pods by the endpoints controller, which fills the Service's EndpointSlices that kube-proxy load-balances to; a label mismatch leaves the Service with no endpoints.

kubectl get endpoints <svc>
kubectl get pods -l app=payment,tier=backend

Production Scenario 1 — A Service Returns No Response

Requests to a Service time out though the Pods are running. kubectl get endpoints shows the Service has zero endpoints. The Pod template labels don't match the Service selector, or the Pods aren't passing readiness. Aligning the labels and fixing readiness restores routing — a reminder that Services connect to Pods purely through labels.

Senior Interview Questions

  1. Why can't a Service select Pods by annotation?
  2. What are the first two things to check when a Service has no endpoints?

ReplicaSets & Deployments

Deployments are the workhorse controller for stateless applications. This chapter covers ReplicaSets, Deployments, and how the two relate, setting up the rollout and rollback material in the next chapter.

Question 1 — What is a ReplicaSet?

Short Answer

A ReplicaSet ensures a specified number of identical Pod replicas are running at all times.

Detailed Explanation

It is the self-healing replica controller: if a Pod dies, it creates a replacement to maintain the desired count. You rarely create one directly because Deployments manage ReplicaSets and add rollouts and rollbacks on top.

Interview Answer

A ReplicaSet keeps a desired number of identical Pods running and recreates any that fail, but it's normally managed by a Deployment rather than created directly.

Question 2 — What is a Deployment and what does it add over a ReplicaSet?

Short Answer

A Deployment manages stateless apps declaratively, creating and controlling ReplicaSets to provide rolling updates, rollbacks, and scaling.

Detailed Explanation

On a spec change such as a new image, the Deployment creates a new ReplicaSet and gradually shifts Pods from the old to the new one per the rollout strategy. It tracks revision history so you can roll back. Each Deployment revision corresponds to a ReplicaSet.

Interview Answer

A Deployment is the controller for stateless workloads that manages ReplicaSets and layers on declarative rolling updates, rollbacks, and scaling, creating a new ReplicaSet per revision and shifting Pods gradually.

Question 3 — How do a Deployment, ReplicaSet, and Pod relate?

Short Answer

A Deployment owns ReplicaSets, and each ReplicaSet owns Pods; the Deployment is the top-level controller.

Detailed Explanation

The Deployment creates a ReplicaSet for the current template; the ReplicaSet creates and maintains the Pods. On an update, a new ReplicaSet is created and scaled up while the old one scales down. revisionHistoryLimit controls how many old ReplicaSets are kept for rollback.

Interview Answer

The hierarchy is Deployment owns ReplicaSet owns Pod: the Deployment manages revisions as ReplicaSets, each ReplicaSet maintains its Pods, and old ReplicaSets are retained per revisionHistoryLimit for rollback.

kubectl get deploy,rs,pods -l app=web
kubectl scale deployment/web --replicas=5

Chapter 7

Rollouts, Rollbacks & Deployment Strategies

Safe delivery is core to SRE. This chapter covers rolling updates, the maxSurge and maxUnavailable knobs, rollbacks, and the higher-level strategies of recreate, blue-green, and canary.

Question 1 — How does a rolling update work?

Short Answer

A rolling update gradually replaces old Pods with new ones, keeping the app available throughout.

Detailed Explanation

The Deployment scales up the new ReplicaSet and scales down the old one in increments bounded by maxSurge and maxUnavailable, waiting for new Pods to pass readiness before continuing. Old and new versions briefly run together, so the app must tolerate mixed versions.

Interview Answer

A rolling update shifts Pods from the old ReplicaSet to the new one incrementally, gated by readiness and bounded by maxSurge and maxUnavailable, giving zero-downtime delivery as long as versions can coexist.

Question 2 — Explain maxSurge and maxUnavailable.

Short Answer

maxSurge is how many Pods can exist above the desired count during an update; maxUnavailable is how many can be missing below it.

Detailed Explanation

Both accept absolute numbers or percentages and default to 25%. Setting maxUnavailable to 0 with maxSurge of 1 yields a fully available, one-at-a-time rollout that needs spare capacity. Setting maxSurge to 0 removes before adding, reducing capacity but avoiding extra Pods.

Interview Answer

maxSurge caps extra Pods created above desired during a rollout and maxUnavailable caps how many can be unavailable; maxUnavailable 0 with maxSurge 1 gives a zero-downtime rollout at the cost of temporary extra capacity.

strategy:
  type: RollingUpdate
  rollingUpdate:
    maxSurge: 1
    maxUnavailable: 0

Question 3 — How do you roll back a bad deployment?

Short Answer

Use kubectl rollout undo to return to the previous revision, or to a specific revision with --to-revision.

Detailed Explanation

Deployments keep revision history backed by old ReplicaSets, so a rollback is itself a controlled rollout back to a prior template. revisionHistoryLimit governs how far back you can go. Recording a change-cause makes the history readable.

Interview Answer

Rollback is kubectl rollout undo, which performs a controlled rollout back to a prior revision's ReplicaSet; how far back you can go depends on revisionHistoryLimit.

kubectl rollout history deployment/web
kubectl rollout undo deployment/web --to-revision=3
kubectl rollout status deployment/web

Question 4 — Compare rolling update, recreate, blue-green, and canary.

Short Answer

Rolling update replaces gradually, recreate replaces all at once, blue-green runs both versions and flips traffic, and canary shifts a small traffic percentage first.

Detailed Explanation

Recreate causes downtime and is used when versions can't coexist. Blue-green gives instant rollback at double the resources by switching the Service from blue to green. Canary sends a small fraction of traffic to the new version and ramps based on metrics, usually via a mesh, ingress weights, or Argo Rollouts.

Comparison

Rolling: built-in, cheap, mixed versions. Recreate: downtime, no mixing. Blue-green: instant rollback, double cost. Canary: safest, needs traffic-splitting tooling.

Interview Answer

Rolling update is gradual and zero-downtime, recreate is all-at-once with downtime for incompatible versions, blue-green flips traffic between two full environments for instant rollback, and canary ramps traffic to the new version based on metrics.

Production Scenario 1 — Guaranteeing Zero Downtime

A payments service must never drop requests during deploys. The team sets maxUnavailable to 0, adds proper readiness probes so only ready Pods receive traffic, and protects against node drains with a PodDisruptionBudget. Together these ensure capacity never dips below the required level during a rollout.

Production Scenario 2 — A Canary Catches a Regression

A new version is rolled out to five percent of traffic via a canary. Error-rate and latency metrics on the canary spike while the stable version stays healthy. Automated analysis halts the rollout and routes traffic back, containing the blast radius to a fraction of users instead of everyone.

  4. What tooling enables automated canary analysis?

Quick Answer Bank

  1. What is a rolling update? Gradual Pod replacement with zero downtime.
  2. What is maxSurge? Extra Pods allowed above desired during rollout.
  3. What is maxUnavailable? Pods allowed to be missing during rollout.
  4. What gives a zero-downtime rollout? maxUnavailable 0 plus readiness probes.
  5. How do you roll back? kubectl rollout undo.
  6. How do you roll back to a specific revision? --to-revision=N.

  8. What is recreate? Kill all old Pods, then start new ones.

Services & kube-proxy

Services give workloads stable networking. This chapter covers the Service types, how ClusterIP routing actually works, and headless Services, which underpin stateful workloads.

Question 1 — What are the Service types and when do you use each?

Short Answer

ClusterIP for internal traffic, NodePort for a static port on every node, LoadBalancer for an external cloud load balancer, and ExternalName for a DNS alias.

Detailed Explanation

ClusterIP is the default and is reachable only inside the cluster. NodePort exposes the Service on a fixed port on each node, mostly for dev or behind an external load balancer. LoadBalancer provisions a cloud load balancer with an external IP. ExternalName maps the Service to an external DNS name via a CNAME with no proxying.

Interview Answer

ClusterIP gives an internal virtual IP, NodePort opens a static port on every node, LoadBalancer provisions an external cloud load balancer, and ExternalName aliases an external DNS name; internal traffic uses ClusterIP and external cloud exposure uses LoadBalancer, often behind Ingress.

Question 2 — How does a ClusterIP Service route traffic to Pods?

Short Answer

The Service has a stable virtual IP; the endpoints controller fills EndpointSlices with ready Pods, and kube-proxy load-balances traffic to them.

Detailed Explanation

CoreDNS resolves the Service name to the ClusterIP so apps connect by name. kube-proxy programs iptables or IPVS rules that DNAT traffic from the ClusterIP to a backend Pod IP. The ClusterIP is virtual and stable even as Pods come and go.

Interview Answer

A ClusterIP Service has a stable virtual IP and a set of EndpointSlices of ready Pods; CoreDNS resolves the name to that IP and kube-proxy load-balances and DNATs traffic to a backend Pod, so clients use a stable name while Pods churn.

# inside the cluster
curl http://payment.default.svc.cluster.local:8080

Question 3 — What is a headless Service and when do you need one?

Short Answer

A headless Service sets clusterIP to None, so DNS returns the individual Pod IPs instead of a single virtual IP.

Detailed Explanation

It is used when clients need to address specific Pods — StatefulSets needing stable per-Pod DNS, databases, peer-to-peer systems, or client-side load balancing. With a StatefulSet it yields stable identities like pod-0.svc.namespace.svc.cluster.local.

Interview Answer

A headless Service (clusterIP None) returns Pod IPs directly via DNS rather than one virtual IP, which is what StatefulSets and stateful systems need for stable per-Pod addressing.

spec:
  clusterIP: None
  selector:
    app: cassandra

A team needs to expose an HTTP app externally in the cloud. A bare LoadBalancer Service per app gets expensive and lacks host or path routing. The better pattern is one LoadBalancer fronting an Ingress controller that routes by host and path to many internal ClusterIP Services, keeping cost and complexity low.

Production Scenario 2 — A Database Needs Stable Pod Addresses

A clustered database requires peers to reach each other by stable name. A normal ClusterIP load-balances across Pods, which breaks peer addressing. Switching to a headless Service gives each StatefulSet Pod a stable DNS name, allowing the cluster members to form a quorum reliably.

Ingress & Gateway API

Ingress and the newer Gateway API provide L7 HTTP routing into the cluster. This chapter clarifies how they differ from Services and why Gateway API is the modern direction.

Question 1 — What is Ingress and how does it differ from a Service?

Short Answer

A Service does L4 load balancing; Ingress provides L7 HTTP and HTTPS routing with host and path rules, TLS termination, and a single entry point for many Services.

Detailed Explanation

Ingress is only an API object describing routing rules; it needs an Ingress controller such as NGINX, Traefik, or a cloud ALB controller actually running to fulfill them. A typical setup is one external load balancer feeding the controller, which routes by host and path to many ClusterIP Services.

Common Mistake

Expecting an Ingress object to route traffic on its own. Without a running Ingress controller, the Ingress object does nothing.

Interview Answer

A Service load-balances at L4, while Ingress provides L7 host and path routing, TLS termination, and a shared entry point; Ingress is just rules and requires an Ingress controller to enforce them.

Question 2 — What is the Gateway API and why is it replacing Ingress?

Short Answer

Gateway API is a more expressive, role-oriented successor to Ingress that standardizes advanced routing without vendor-specific annotations.

Detailed Explanation

Ingress pushes vendor behavior into annotations and is HTTP-centric with weak role separation. Gateway API separates GatewayClass (infra provider), Gateway (cluster operator), and route objects like HTTPRoute (app teams), and natively supports header-based routing, traffic splitting, and cross-namespace routing.

Interview Answer

Gateway API is the portable, role-separated successor to Ingress: platform teams own GatewayClass and Gateway while app teams own HTTPRoutes, with native support for traffic splitting and header routing instead of annotation hacks.

Production Scenario 1 — Consolidating Dozens of Services Behind One Entry Point

A cluster has thirty microservices each needing external HTTPS. Thirty LoadBalancer Services would be costly and unmanageable. One Ingress controller behind a single load balancer terminates TLS centrally and routes by host and path to each ClusterIP Service, simplifying certificates and cost.

Production Scenario 2 — Multi-Team Routing Governance

Platform and app teams clash over a shared Ingress because all routing lives in one object with vendor annotations. Adopting Gateway API lets the platform team own the Gateway and TLS while each app team manages its own HTTPRoute, giving clean role separation and safer self-service.

  2. Which Gateway API roles map to GatewayClass, Gateway, and HTTPRoute?
  3. How does Gateway API handle traffic splitting compared to Ingress?
  4. Where is TLS typically terminated in an Ingress setup?

Quick Answer Bank

  1. What is Ingress? L7 HTTP routing into the cluster.
  2. Service vs Ingress? Service is L4; Ingress is L7 host/path routing.
  3. Does Ingress work without a controller? No.
  4. Name an Ingress controller. NGINX, Traefik, or a cloud ALB controller.
  5. What does Ingress terminate? TLS, at the controller.
  6. What is Gateway API? The role-separated successor to Ingress.
  7. Who owns a Gateway? The platform or cluster operator.
  8. Who owns an HTTPRoute? The application team.
  9. What does Gateway API add? Native traffic splitting and header routing.
  10. Why move off Ingress? Vendor annotation sprawl and weak role separation.

Best Practices

Run one Ingress controller behind a single external load balancer and terminate TLS centrally with automated certificates. Keep routing rules in version control. For multi-team clusters, adopt Gateway API to separate platform-owned Gateways from team-owned routes. Always confirm a controller is installed before relying on Ingress objects.

Interview Cheat Sheet

Ingress: L7 host/path routing, needs a controller, terminates TLS. Gateway API: GatewayClass (infra) → Gateway (platform) → HTTPRoute (app), native traffic splitting and header routing.

StatefulSets

StatefulSets run workloads that need stable identity and storage, such as databases and message brokers. This chapter covers how they differ from Deployments and what happens to their storage.

Question 1 — What is a StatefulSet and how does it differ from a Deployment?

Short Answer

A StatefulSet gives each Pod a stable ordinal identity, stable network name, and its own persistent storage, unlike a Deployment where Pods are interchangeable.

Detailed Explanation

Each Pod gets a stable name like app-0, app-1, stable DNS via a headless Service, and a dedicated PersistentVolumeClaim from volumeClaimTemplates that persists across rescheduling. Pods are created, scaled, and updated in order, with updates rolling in reverse ordinal order.

Interview Answer

A StatefulSet runs stateful apps that need stable identity and storage: each Pod has a fixed ordinal name, stable DNS, and its own PVC that survives rescheduling, and Pods are managed in a deterministic order, unlike a Deployment's interchangeable Pods.

Question 2 — What happens to storage when a StatefulSet Pod is rescheduled or deleted?

Short Answer

Each Pod's PVC is tied to its identity, so a rescheduled Pod reattaches to the same volume; deleting the StatefulSet does not delete PVCs by default.

Detailed Explanation

PVCs created from volumeClaimTemplates are bound to each Pod's stable identity, so data persists across rescheduling. Scaling down leaves PVCs behind to protect data. Deleting the StatefulSet retains PVCs unless you set a persistentVolumeClaimRetentionPolicy or remove them manually.

Common Mistake

Assuming deleting a StatefulSet cleans up its volumes. PVCs are intentionally retained to prevent accidental data loss.

Interview Answer

PVCs from volumeClaimTemplates are bound to each Pod's stable identity, so a rescheduled Pod keeps its data, and scaling down or deleting the StatefulSet retains the PVCs by default to protect data unless you opt into deletion.

Production Scenario 1 — Scaling a Database StatefulSet Down and Back Up

A three-node database StatefulSet is scaled to one to save cost, then back to three. Because PVCs are retained on scale-down, the returning Pods reattach to their original volumes and recover their data, rather than starting empty — exactly the behavior a stateful system depends on.

Production Scenario 2 — A StatefulSet Pod Won't Reschedule

A StatefulSet Pod is stuck Pending after a node failure. Its PVC uses a zonal volume that can only attach in the original zone, but no node there has capacity. The fix is topology-aware provisioning and spreading capacity across zones, since StatefulSet storage is tied to a Pod's identity and often a zone.

Senior Interview Questions

  1. What gives each StatefulSet Pod a stable DNS name?
  2. How do you intentionally clean up StatefulSet PVCs?
  3. Why do StatefulSet updates roll in reverse ordinal order?
  4. How does zonal storage constrain StatefulSet rescheduling?

Quick Answer Bank

  1. What is a StatefulSet? A controller for stateful Pods with stable identity and storage.
  2. How are StatefulSet Pods named? With stable ordinals like app-0.
  3. What gives stable DNS? A headless Service.
  4. Where does per-Pod storage come from? volumeClaimTemplates.
  5. Does a rescheduled Pod keep its data? Yes, it reattaches its PVC.
  6. Are PVCs deleted with the StatefulSet? No, retained by default.
  7. How are StatefulSet Pods created? In order, lowest ordinal first.
  8. How do updates roll? In reverse ordinal order.
  9. StatefulSet vs Deployment? Stable identity and storage vs interchangeable Pods.

Best Practices

Use StatefulSets only for workloads that truly need stable identity and storage. Always pair them with a headless Service. Use topology-aware volume provisioning and spread capacity across zones so Pods can reschedule. Set an explicit persistentVolumeClaimRetentionPolicy so cleanup behavior is intentional, and back up the underlying data independently.

DaemonSets, Jobs & CronJobs

Beyond long-running services, Kubernetes runs node-level agents and batch work. This chapter covers DaemonSets, Jobs, and CronJobs and their key configuration knobs.

Question 1 — What is a DaemonSet and what is it used for?

Short Answer

A DaemonSet runs exactly one Pod per node, automatically adding Pods as nodes join and removing them as nodes leave.

Detailed Explanation

It is used for node-level agents such as log collectors, node-exporter, CNI plugins, storage daemons, and security agents. DaemonSet Pods often tolerate taints so they can run even on control plane or specialized nodes. You can target a subset of nodes with a nodeSelector or affinity.

Interview Answer

A DaemonSet ensures one Pod runs on every node (or a matching subset), scheduling onto new nodes automatically; it's used for node-level agents like log shippers, node-exporter, and CNI plugins, and usually tolerates taints to run everywhere.

Question 2 — What is the difference between a Job and a CronJob?

Short Answer

A Job runs Pods to completion for batch work; a CronJob creates Jobs on a recurring schedule.

Detailed Explanation

A Job retries failures up to backoffLimit and supports parallelism and completion counts for batch processing. A CronJob uses cron syntax to create Jobs on a schedule and adds concurrencyPolicy, startingDeadlineSeconds, and history limits. Use ttlSecondsAfterFinished to clean up finished Jobs.

Interview Answer

A Job runs Pods to successful completion for one-off batch work with retries and optional parallelism, while a CronJob schedules Jobs on a recurring cron schedule with concurrency and history controls.

spec:
  schedule: "0 2 * * *"
  concurrencyPolicy: Forbid
  jobTemplate:
    spec:
      backoffLimit: 3

Production Scenario 1 — Rolling Out a Logging Agent Everywhere

Every node must run a log-forwarding agent, including new nodes added by autoscaling. A DaemonSet guarantees the agent runs on each node and automatically appears on new nodes, with tolerations so it also runs on tainted control plane nodes — the standard pattern for node-level observability.

Production Scenario 2 — Overlapping Backup Jobs

A nightly backup CronJob occasionally runs long and a second run starts before the first finishes, corrupting the backup. Setting concurrencyPolicy to Forbid prevents overlapping runs, and startingDeadlineSeconds bounds how late a missed run may start, making the schedule safe.

  2. What does concurrencyPolicy control on a CronJob?
  3. How do you automatically clean up finished Jobs?
  4. How do parallelism and completions shape a batch Job?

Quick Answer Bank

  1. What is a DaemonSet? One Pod per node, added automatically to new nodes.
  2. What runs as a DaemonSet? Log agents, node-exporter, CNI plugins.
  3. How does a DaemonSet run on tainted nodes? With tolerations.
  4. What is a Job? A controller running Pods to completion.
  5. What is backoffLimit? The number of retry attempts for a Job.
  6. What is a CronJob? A controller creating Jobs on a schedule.
  7. What is concurrencyPolicy? Allow, Forbid, or Replace overlapping runs.
  8. How do you clean up finished Jobs? ttlSecondsAfterFinished.

For CronJobs, set concurrencyPolicy, startingDeadlineSeconds, and history limits explicitly, and use ttlSecondsAfterFinished to avoid Job buildup.

ConfigMaps & Secrets

Configuration and sensitive data are decoupled from images via ConfigMaps and Secrets. This chapter covers their differences, security posture, and the subtle update behavior that surprises people in production.

Question 1 — What is the difference between a ConfigMap and a Secret?

Short Answer

ConfigMaps hold non-sensitive configuration; Secrets hold sensitive data and are only base64-encoded, not encrypted, by default.

Detailed Explanation

Both can be consumed as environment variables or mounted files. Secrets are base64-encoded, which is not security, so you should enable encryption at rest, tighten RBAC, and ideally source secrets from an external manager like Vault or a cloud secrets service via an operator.

Common Mistake

Believing Secrets are encrypted by default. They are only base64-encoded unless you enable encryption at rest in the API server.

Interview Answer

ConfigMaps store non-sensitive config and Secrets store sensitive data, but Secrets are merely base64-encoded by default, so production hardening means encryption at rest, strict RBAC, and external secret managers.

Question 2 — If you update a ConfigMap, do running Pods see the change?

Short Answer

Environment variables from a ConfigMap are fixed at Pod start and don't update live; volume-mounted ConfigMaps update in the filesystem but the app must re-read them.

Detailed Explanation

Env-injected values require a Pod restart to change. Mounted ConfigMaps eventually update after the kubelet sync, but the application must reload the file. A common pattern is annotating the Pod template with a checksum of the ConfigMap so a change triggers a rolling restart automatically.

Interview Answer

ConfigMap values injected as environment variables are frozen at Pod start and need a restart, while volume-mounted values update on the filesystem but require the app to re-read them; teams often add a ConfigMap checksum annotation so changes trigger a rollout.

kubectl rollout restart deployment/web

Production Scenario 1 — A Config Change That Never Took Effect

An engineer edits a ConfigMap consumed as environment variables and is confused when nothing changes. Env vars are set only at Pod start, so the running Pods keep the old values. Rolling the Deployment, or wiring a checksum annotation to do so automatically, applies the new configuration.

Production Scenario 2 — A Secret Leaked Through Git

A base64-encoded Secret manifest was committed to Git, exposing a database password since base64 is trivially reversible. The remediation is to rotate the credential, remove it from history, enable encryption at rest, and move secrets to an external manager injected at runtime rather than stored in manifests.

  1. What is a ConfigMap? Non-sensitive key/value configuration.

  4. How do you encrypt Secrets at rest? EncryptionConfiguration on the API server.

Question 1 — Explain PersistentVolume, PersistentVolumeClaim, and StorageClass.

Short Answer

A PV is a storage resource in the cluster, a PVC is a request that binds to a PV, and a StorageClass enables dynamic provisioning of PVs on demand.

Detailed Explanation

A PV is an actual piece of storage such as an EBS volume or NFS share. A PVC is a user's request for storage by size and access mode and binds to a matching PV. A StorageClass defines a storage type and provisioner, so a PVC referencing it triggers automatic creation of a backing PV. Pods always reference PVCs, never PVs directly.

Interview Answer

A PersistentVolume is real storage in the cluster, a PersistentVolumeClaim is a user's request that binds to a PV, and a StorageClass with a provisioner enables dynamic creation of PVs when a PVC asks for it; Pods reference PVCs.

# PVC referencing a StorageClass triggers dynamic provisioning

  storageClassName: gp3
  accessModes: ["ReadWriteOnce"]

Question 2 — Explain the access modes: ReadWriteOnce, ReadOnlyMany, ReadWriteMany, ReadWriteOncePod.

Short Answer

RWO mounts read-write to one node, ROX read-only to many nodes, RWX read-write to many nodes, and ReadWriteOncePod read-write to exactly one Pod.

Detailed Explanation

Block storage like EBS is typically RWO, mounted read-write by a single node. RWX needs a shared filesystem such as NFS, EFS, or CephFS. ReadWriteOncePod is stricter than RWO. The common error is scheduling two Pods that need the same RWO volume onto different nodes, which causes a multi-attach error.

Common Mistake

Expecting an RWO EBS volume to be shared by Pods on different nodes. RWO is single-node, so this produces a multi-attach error; use RWX storage for shared access.

Interview Answer

RWO is single-node read-write, ROX is multi-node read-only, RWX is multi-node read-write requiring a shared filesystem, and ReadWriteOncePod restricts to one Pod; because block storage is RWO, spreading Pods needing it across nodes causes multi-attach errors.

Production Scenario 1 — A Multi-Attach Error During Failover

A Pod with an RWO EBS volume fails over to a new node while the old Pod is still terminating, and the new Pod can't attach the volume. Because EBS is single-node read-write, the volume must detach from the old node first. Proper Pod termination and waiting for detachment, or topology-aware scheduling, resolves it.

Production Scenario 2 — Sharing Files Across Replicas

A team wants three replicas to share a writable directory but uses an RWO volume, so only one Pod can mount it. The correct approach is RWX storage such as EFS or NFS, which supports concurrent read-write mounts across nodes, since RWO can never provide shared write access.

Networking: CNI, CoreDNS & NetworkPolicy

Kubernetes networking is where many candidates are weakest. This chapter covers the networking model and CNI, CoreDNS service discovery, and NetworkPolicy for east-west segmentation.

Question 1 — Explain the Kubernetes networking model and the role of CNI.

Short Answer

Every Pod gets its own IP and can reach every other Pod without NAT; a CNI plugin implements this addressing and routing.

Detailed Explanation

Containers in a Pod share the Pod's network namespace. Kubernetes itself doesn't implement Pod networking — a CNI plugin such as Calico, Cilium, Flannel, or the AWS VPC CNI assigns Pod IPs and programs routes or overlays. Cilium uses eBPF for high performance and can replace kube-proxy.

Interview Answer

The model requires each Pod to have a unique IP reachable by every other Pod without NAT, with containers in a Pod sharing one namespace; Kubernetes delegates this to a CNI plugin like Calico or Cilium that assigns IPs and programs the dataplane.

Question 2 — What is CoreDNS and how does service discovery work?

Short Answer

CoreDNS provides in-cluster DNS so Services resolve by name; every Pod is configured to query it.

Detailed Explanation

Services resolve as service.namespace.svc.cluster.local, or just the service name within the same namespace via search domains. Headless Services resolve to individual Pod IPs, and StatefulSet Pods get stable per-Pod names. DNS issues are a frequent cause of connectivity failures.

Interview Answer

CoreDNS runs in kube-system and serves cluster DNS so apps reach Services by name like service.namespace.svc.cluster.local, with headless Services and StatefulSets resolving to specific Pod IPs.

kubectl -n kube-system get pods -l k8s-app=kube-dns
kubectl run -it --rm dns --image=busybox -- nslookup my-svc

Question 3 — What is a NetworkPolicy and what is the default behavior?

Short Answer

By default all Pods can talk to all Pods; a NetworkPolicy restricts ingress and egress using label selectors.

Detailed Explanation

Once any NetworkPolicy selects a Pod, that Pod becomes default-deny for the selected direction except what the policy allows. NetworkPolicies require a CNI that enforces them, such as Calico or Cilium; some CNIs ignore them. The recommended baseline is default-deny plus explicit allows.

Common Mistake

Applying a NetworkPolicy and assuming traffic is blocked even though the CNI doesn't enforce policies. Confirm the CNI supports NetworkPolicy, or nothing is enforced.

Interview Answer

Without a NetworkPolicy all Pod-to-Pod traffic is allowed; a NetworkPolicy uses selectors to restrict traffic, and once a Pod is selected everything in that direction is denied except explicit allows, but only with a policy-enforcing CNI.

Production Scenario 1 — DNS Resolution Fails Intermittently

Pods sometimes can't resolve Service names. CoreDNS Pods are under-resourced and dropping queries under load. Scaling CoreDNS, tuning its cache, and checking node DNS configuration restores reliable resolution — a reminder that DNS is a critical, easily overlooked dependency.

Production Scenario 2 — Enforcing Default-Deny Segmentation

A compliance audit requires that services only talk to approved peers. The team applies a default-deny NetworkPolicy per namespace and then explicit allow rules for required flows, with a CNI like Calico enforcing them. East-west traffic is now segmented, satisfying the audit.

Scheduling: Affinity, Taints & Tolerations

Placement controls reliability and performance. This chapter covers taints and tolerations, node affinity, and pod affinity and anti-affinity with topology spread, which are the levers a Senior SRE uses to design for availability.

Question 1 — What are taints and tolerations, and how do they differ from node affinity?

Short Answer

Taints on nodes repel Pods that don't tolerate them; node affinity on Pods attracts them to nodes with matching labels.

Detailed Explanation

Taints are applied to nodes with effects NoSchedule, PreferNoSchedule, or NoExecute and keep Pods off unless they carry a matching toleration. Node affinity is a Pod-side preference or requirement that pulls Pods toward labeled nodes. They are complementary: taint a GPU node to keep generic workloads off, and add affinity so GPU workloads target it.

Comparison

Taints push Pods away from nodes; affinity pulls Pods toward nodes. NoExecute even evicts already-running Pods that lack a matching toleration.

Interview Answer

Taints are a node-side mechanism that repels Pods unless they tolerate the taint, while node affinity is a Pod-side rule that attracts Pods to matching nodes; together they reserve specialized nodes for the right workloads.

Question 2 — Explain pod affinity, anti-affinity, and topology spread constraints.

Short Answer

Pod affinity co-locates Pods, anti-affinity separates them, and topology spread distributes Pods evenly across domains like zones.

Detailed Explanation

Pod affinity schedules Pods near related Pods, such as a cache beside its app. Anti-affinity keeps replicas apart so one node or zone failure doesn't take them all. Topology spread constraints use maxSkew to balance Pods across zones or nodes, a cleaner modern way to achieve high availability than anti-affinity alone.

Interview Answer

Pod affinity co-locates related Pods, anti-affinity spreads replicas to survive failures, and topology spread constraints distribute Pods evenly across zones or nodes via maxSkew, which together let you design for fault isolation and availability.

topologySpreadConstraints:
- maxSkew: 1
  topologyKey: topology.kubernetes.io/zone
  whenUnsatisfiable: DoNotSchedule
  labelSelector:
    matchLabels: { app: web }

Production Scenario 1 — Reserving GPU Nodes

Expensive GPU nodes were being filled by ordinary Pods, starving ML jobs. Tainting the GPU nodes with NoSchedule keeps generic workloads off, while adding a matching toleration and node affinity to the ML Pods ensures only they land there — the canonical taint-plus-affinity pattern for specialized hardware.

Production Scenario 2 — Surviving a Zone Failure

All replicas of a service happened to land in one availability zone, so a zone outage took the service down. Adding topology spread constraints on the zone topology key forces replicas to distribute across zones, so a single zone failure now removes only a fraction of capacity.

  4. Why are topology spread constraints often preferred over anti-affinity?

Quick Answer Bank

  2. What is a toleration? A Pod allowance to schedule onto a tainted node.

Autoscaling: HPA, VPA & Cluster Autoscaler

Scaling is a core SRE and platform competency. This chapter covers the three autoscalers, how they compose, and the pitfalls of running them together.

Question 1 — Explain HPA, VPA, and the Cluster Autoscaler and how they relate.

Short Answer

HPA scales the number of Pods, VPA right-sizes Pod resource requests, and the Cluster Autoscaler adds or removes nodes.

Detailed Explanation

HPA scales replicas on CPU, memory, or custom and external metrics via metrics-server. VPA adjusts CPU and memory requests and often recreates Pods to apply them. The Cluster Autoscaler adds nodes when Pods can't schedule and removes underutilized ones. They compose: HPA adds Pods, and if no node has room the Cluster Autoscaler adds a node.

Common Mistake

Running HPA and VPA on the same resource metric like CPU. They conflict, since one changes replica count and the other changes requests based on the same signal.

Interview Answer

HPA scales replica count on metrics, VPA right-sizes per-Pod requests, and the Cluster Autoscaler scales the node pool; HPA and the Cluster Autoscaler work together to handle load, while VPA handles right-sizing.

kubectl autoscale deployment web --cpu-percent=70 --min=3 --max=20

Production Scenario 1 — A Traffic Spike Without Node Capacity

HPA scales a Deployment from five to twenty Pods during a spike, but the extra Pods stay Pending because no node has room. The Cluster Autoscaler observes the unschedulable Pods and provisions new nodes, after which the Pods schedule. The two autoscalers together absorb the load — neither alone would suffice.

Production Scenario 2 — HPA and VPA Fighting

A team enables both HPA and VPA on CPU for the same Deployment. VPA raises requests while HPA adds replicas off the same signal, causing oscillation and instability. The fix is to let HPA own CPU-based replica scaling and use VPA only in recommendation mode or for non-overlapping resources like memory.

  1. Why shouldn't HPA and VPA both act on CPU?
  2. What component supplies CPU and memory metrics to the HPA?
  3. How does the Cluster Autoscaler decide to remove a node?
  4. How would you scale on a custom application metric like queue depth?

Quick Answer Bank

  1. What does HPA scale? The number of Pod replicas.
  2. What does VPA scale? Per-Pod CPU and memory requests.
  3. What does the Cluster Autoscaler scale? The node pool.
  4. What feeds HPA metrics? metrics-server.
  5. Can HPA use custom metrics? Yes, custom and external metrics.
  6. Why not run HPA and VPA on CPU together? They conflict.
  7. When does the Cluster Autoscaler add nodes? When Pods can't schedule.
  8. When does it remove nodes? When they're underutilized.
  9. How do HPA and Cluster Autoscaler compose? HPA adds Pods; CA adds nodes.
  10. How does VPA apply new requests? Often by recreating Pods.

Best Practices

Use HPA for load-driven replica scaling with sensible min and max replicas and proper requests. Pair it with the Cluster Autoscaler so node capacity follows Pod demand. Use VPA for right-sizing in recommendation mode, avoiding overlap with HPA on the same metric. Scale on the metric that reflects real load, such as queue depth or request rate, not just CPU.

Interview Cheat Sheet

HPA: replica count on metrics (metrics-server). VPA: right-sizes requests, recreates Pods. Cluster Autoscaler: node pool. HPA + CA compose. Don't run HPA + VPA on the same metric.

RBAC & Security

Security is a top topic for fintech and regulated roles. This chapter covers RBAC, Pod hardening, and NetworkPolicy posture that demonstrate a defense-in-depth mindset.

Question 1 — Explain RBAC: Role, ClusterRole, RoleBinding, and ClusterRoleBinding.

Short Answer

Roles grant namespaced permissions and ClusterRoles grant cluster-wide ones; bindings grant those roles to subjects, in a namespace or cluster-wide.

Detailed Explanation

A Role defines verbs on resources within one namespace; a ClusterRole does the same cluster-wide or for cluster-scoped resources. Pending means unscheduled; run kubectl describe pod and read the Events for the reason.

Detailed Explanation

Common causes are insufficient CPU or memory on all nodes, node selectors or affinity that match nothing, taints without tolerations, an unbound PVC, or a ResourceQuota limit. Cross-check node capacity with kubectl get nodes and PVC status with kubectl get pvc.

Interview Answer

Pending means the scheduler couldn't place the Pod; describe pod's Events state why — insufficient resources, unsatisfiable affinity or taints, an unbound PVC, or a quota — and you confirm against node capacity and PVC status.

kubectl describe pod <pod>
kubectl get nodes
kubectl get events --sort-by=.lastTimestamp

Question 2 — A Pod is in CrashLoopBackOff. Walk through your debugging.

Short Answer

Read the previous container's logs and the exit code, then fix the underlying cause.

Detailed Explanation

Use kubectl logs --previous to see why the last instance died and kubectl describe pod for the exit code and events. Causes include missing config or secrets, a failing startup dependency, a bad entrypoint, an OOMKill at exit code 137, or an over-aggressive liveness probe. Fix the root cause and redeploy.

Interview Answer

CrashLoopBackOff is repeated crashing with kubelet back-off; I check the previous instance's logs and the exit code, identify whether it's config, a dependency, the entrypoint, an OOMKill, or a probe, fix that, and redeploy.

kubectl logs <pod> --previous
kubectl describe pod <pod>

Question 3 — A Service returns connection errors though Pods are running. Diagnose.

Short Answer

Check whether the Service has endpoints; empty endpoints usually mean a label mismatch or failing readiness.

Detailed Explanation

Verify endpoints with kubectl get endpoints, confirm the selector matches the Pod labels, check targetPort against the container port, and ensure the app binds 0.0.0.0 rather than localhost. Then test DNS resolution, kube-proxy, and any NetworkPolicy that could block the path.

Interview Answer

I check the Service's endpoints first; if empty it's usually a selector or label mismatch or readiness failing, then I verify ports, that the app binds all interfaces, DNS resolution, and any blocking NetworkPolicy.

kubectl get endpoints <svc>
kubectl get pods -l <selector> -o wide

Question 4 — A node shows NotReady. How do you investigate?

Short Answer

Run kubectl describe node and check conditions and the kubelet heartbeat; the cause is usually kubelet, runtime, disk, or network.

Detailed Explanation

Check conditions for MemoryPressure, DiskPressure, and PIDPressure and when the kubelet last reported. On the node, inspect kubelet and runtime status, disk space, and kubelet logs. Pods on a node that stays NotReady are evicted and rescheduled after the eviction timeout.

Interview Answer

I describe the node to read its conditions and heartbeat; NotReady usually means a stopped kubelet, failed runtime, full disk, or lost connectivity, which I confirm on the node, knowing its Pods reschedule after the eviction timeout.

kubectl describe node <node>
# on node: systemctl status kubelet; journalctl -u kubelet -n100

Question 5 — etcd is unhealthy or a control plane node is down. What's the impact and response?

Short Answer

Losing etcd quorum stops all writes, though running Pods continue; restore the member or recover from a snapshot.

Detailed Explanation

With a three-member etcd you can lose one and keep quorum; losing two stops deployments and scaling because the API server can't write. Existing Pods keep running via the kubelets. Recovery is restoring the failed member or, if quorum is lost, restoring from the latest snapshot. Prevent this with an HA control plane and frequent backups.

Interview Answer

If etcd loses quorum the API server can't write so management stops, but the data plane keeps serving; with three members one failure is survivable, otherwise you restore a member or recover from the latest snapshot, which is why frequent etcd backups matter.

ETCDCTL_API=3 etcdctl snapshot status backup.db
ETCDCTL_API=3 etcdctl snapshot restore backup.db

Production Scenario 1 — Insufficient CPU Across the Cluster

Several Pods sit Pending with Insufficient cpu events. Every node is at capacity. The options are lowering requests if they're inflated, adding nodes manually or via the Cluster Autoscaler, or rescheduling lower-priority work. The events point straight to the bottleneck, avoiding guesswork.

Production Scenario 2 — A Cascading Failure From a Bad Probe

During a load spike an aggressive liveness probe restarts busy Pods, reducing capacity and increasing load on the rest, which then also fail their probes. The cascade is broken by making liveness tolerant and using readiness to shed traffic gracefully, then adding capacity — a reminder that probes are a reliability control, not just a health check.

Production Scenario 3 — Image Pull Failures Block a Rollout

New Pods are stuck in ImagePullBackOff after a deploy. describe pod shows an authentication error pulling from a private registry. The imagePullSecret is missing or wrong. Fixing the registry credential and re-rolling restores the deployment, showing how the same describe-and-read-events method resolves a different failure class.

Best Practices

Troubleshoot evidence-first: describe the object, read the events, form a hypothesis, and verify against the interacting components rather than guessing. Set requests, limits, and sane probes so failure signals are meaningful. Keep an HA control plane with etcd backups. Standardize on logs, metrics, and traces so root cause is reachable quickly under pressure.
