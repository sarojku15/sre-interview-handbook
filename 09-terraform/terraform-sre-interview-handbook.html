Chapter 1

Terraform Fundamentals & Infrastructure as Code

Interviewers open here to confirm you understand why Terraform exists before drilling into HCL syntax. The strongest answers connect IaC directly to the reliability problems it solves — configuration drift, unreviewed changes, non-reproducible environments.

Core Concepts

Infrastructure as Code means defining infrastructure in versioned, declarative files instead of clicking through a console or running ad-hoc CLI commands. Terraform is HashiCorp's IaC tool: you declare the desired end state in HCL (HashiCorp Configuration Language), and Terraform computes a plan to reconcile real infrastructure toward it, across any provider (AWS, Azure, GCP, Kubernetes, and hundreds more) through a single, consistent workflow.

Question 1 — What problem does Terraform solve that clicking in the AWS/Azure console doesn't?

Answer

Console changes are unaudited, unreviewed, and easy to forget or misremember. Terraform makes infrastructure changes reviewable (via pull request), reproducible (the same config produces the same environment), and diffable (terraform plan shows exactly what will change before it happens). It also gives you a single declarative source of truth instead of tribal knowledge of "how prod was actually set up."

Question 2 — Declarative vs imperative — where does Terraform sit?

Answer

Terraform is declarative: you describe the desired end state (resource "aws_instance" "web" { ... }) and Terraform figures out the steps to get there — create, update in place, or destroy-and-recreate. This differs from imperative tools (a Bash script that runs aws ec2 run-instances step by step), which describe the actions, not the outcome, and don't know how to reconcile drift.

Question 3 — Terraform vs Ansible — when would you use which?

Answer

Terraform is provisioning-focused: it creates and manages the lifecycle of infrastructure resources (VPCs, VMs, load balancers, DNS records) and tracks their state. Ansible is configuration-focused: it installs packages, manages files, and configures software on machines that already exist, and is typically agentless and idempotent per-run rather than state-tracked. A common production pattern: Terraform provisions the VM, then hands off to Ansible (or cloud-init/user-data) to configure it.

Question 4 — Terraform vs CloudFormation / ARM / Bicep?

Answer

CloudFormation and ARM/Bicep are cloud-native IaC tools locked to one provider (AWS, Azure respectively). Terraform is provider-agnostic — the same tool and workflow manage AWS, Azure, GCP, Kubernetes, and on-prem resources through provider plugins, which matters heavily for multi-cloud shops or teams that don't want tooling fragmented per cloud.

Practice Exercise

Task: Explain in one paragraph, as if to a skeptical engineer, why a team currently doing manual console changes should adopt Terraform.

Model answer: "Right now, if prod breaks because of a manual change, nobody can say for certain what changed or when. With Terraform, every infrastructure change goes through a pull request, gets a `plan` showing the exact diff before anything happens, and lives in Git history — so we get code review, rollback, and an audit trail for infrastructure the same way we already have it for application code."
Common mistake: describing Terraform purely as "a scripting tool for creating cloud resources." That undersells it and misses what interviewers are listening for — the state model and declarative reconciliation, not just resource creation.
Interview tip: if asked "what is Terraform" cold, lead with declarative + state-tracked + provider-agnostic in one sentence, then expand. That ordering signals you know what actually distinguishes it from a shell script.
Chapter 2

Installation & CLI Workflow

The core workflow — init, plan, apply, destroy — is assumed knowledge, but interviewers probe what happens under the hood at each step, since that's where real understanding shows.

Core Workflow

terraform init      # downloads providers/modules, configures the backend
terraform fmt        # auto-formats .tf files to canonical style
terraform validate   # checks syntax and internal consistency (no API calls)
terraform plan       # computes and shows the diff between state and config
terraform apply      # executes the plan, asks for confirmation
terraform destroy    # tears down everything Terraform manages in this state
terraform show        # inspects current state or a saved plan file
terraform output      # prints output values

Question 1 — What exactly does terraform init do?

Answer

It downloads and installs the provider plugins referenced in the config (into .terraform/), initializes the configured backend (local or remote, e.g. S3+DynamoDB), and downloads any referenced modules. It's safe to re-run and is required any time providers, modules, or the backend configuration change.

Question 2 — What does terraform plan actually compute?

Answer

It refreshes state against real infrastructure (unless -refresh=false), then diffs the desired config against that refreshed state, producing a list of resources to create, update in place, or destroy-and-recreate. Nothing is changed on real infrastructure during plan — it's read-only against the providers, aside from the state refresh reads.

Question 3 — How do you save and later apply an exact plan (important for CI/CD)?

terraform plan -out=tfplan
# ... review, get approval ...
terraform apply tfplan   # applies EXACTLY what was planned, no re-diffing

Answer

This two-step pattern is the production-safe way to run Terraform in a pipeline: it guarantees what gets approved is exactly what gets applied, with no risk of the underlying infrastructure changing between review and apply causing a different outcome.

Question 4 — What's the difference between terraform destroy and removing a resource block from config?

Answer

terraform destroy tears down everything tracked in the state file. Removing a single resource block from your .tf files and running apply destroys only that resource — Terraform sees it's no longer declared and plans its deletion. Confusing these two is a common source of "why did it delete everything" incidents.

Practice Exercise

Task: Write the safe CI/CD sequence for a pull-request-triggered Terraform pipeline (plan on PR, apply on merge).

# On pull request:
terraform init -input=false
terraform validate
terraform plan -out=tfplan -input=false
# post plan output as a PR comment for review

# On merge to main:
terraform init -input=false
terraform apply -input=false tfplan   # apply the exact reviewed plan
Common mistake: running terraform apply directly in CI without a saved plan file, or with -auto-approve on a fresh plan — this risks applying a different diff than whatever was reviewed if infrastructure drifted between plan and apply.
Interview tip: mention -input=false and -lock-timeout for CI usage unprompted — it signals you've actually wired Terraform into a pipeline, not just run it locally.
Chapter 3

HCL Syntax & Core Language Concepts

HCL (HashiCorp Configuration Language) fluency is checked quickly and then left behind — but stumbling on basic block syntax under pressure reads badly, so it's worth being fast here.

Core Syntax

# Block syntax: BLOCK_TYPE "LABEL" "LABEL2" { ARGUMENTS }
resource "aws_instance" "web" {
  ami           = "ami-0abcd1234"
  instance_type = "t3.micro"

  tags = {
    Name        = "web-server"
    Environment = "production"
  }
}

# Comments
# single line
# also single line
/* multi
   line */

# Interpolation / references
resource "aws_eip" "web_ip" {
  instance = aws_instance.web.id     # references another resource's attribute
}

# String interpolation
locals {
  full_name = "${var.project}-${var.environment}"
}

Question 1 — What are the top-level block types in Terraform?

Answer

terraform (settings, required providers, backend), provider (configures a provider), resource (declares a managed infrastructure object), data (reads existing infrastructure without managing it), variable (input), output (exposes a value), locals (computed internal values), and module (calls a reusable module).

Question 2 — variable vs locals — when do you use each?

Answer

variable blocks are inputs supplied by the caller (CLI flag, .tfvars file, environment variable, or module caller). locals are computed values derived inside the config itself — for naming conventions, combining variables, or avoiding repetition — and cannot be set from outside. If a value should be configurable by whoever runs the config, it's a variable; if it's derived logic internal to the module, it's a local.

Question 3 — What is the terraform block used for?

terraform {
  required_version = ">= 1.6.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"
      version = "~> 5.0"
    }
  }

  backend "s3" {
    bucket = "my-tfstate-bucket"
    key    = "prod/network.tfstate"
    region = "ap-south-1"
  }
}

Answer

It pins the Terraform CLI version, declares which providers (and version constraints) the config needs, and configures the state backend. Version pinning here is what prevents "works on my machine" drift between team members and CI.

Practice Exercise

Task: Write an HCL snippet that defines a local combining a project name and environment variable, then uses it as a resource tag.

variable "project" {
  type    = string
  default = "checkout"
}

variable "environment" {
  type = string
}

locals {
  name_prefix = "${var.project}-${var.environment}"
}

resource "aws_s3_bucket" "logs" {
  bucket = "${local.name_prefix}-logs"

  tags = {
    Name = local.name_prefix
  }
}
Common mistake: confusing resource type and name in the block header (resource "aws_instance" "web" — aws_instance is the type, web is your local name) and then referencing it wrong elsewhere (aws_instance.web.id vs the reversed order).
Interview tip: version-pin providers with ~> (pessimistic constraint operator, e.g. ~> 5.0 allows 5.x but not 6.0) unprompted when writing any terraform block — unpinned providers are a real, common cause of "it broke after a routine terraform init" incidents.
Chapter 4

Providers

Providers are the plugins that translate HCL into actual API calls against AWS, Azure, Kubernetes, etc. Interviewers use this chapter to check you understand multi-provider and multi-region/account setups, which come up constantly in real multi-account AWS/Azure environments.

Core Concepts

provider "aws" {
  region = "ap-south-1"
}

# Multiple configurations of the same provider, via alias
provider "aws" {
  alias  = "us_east"
  region = "us-east-1"
}

resource "aws_instance" "primary" {
  provider = aws              # default provider (ap-south-1)
  # ...
}

resource "aws_acm_certificate" "cdn_cert" {
  provider = aws.us_east      # explicitly use the aliased provider — ACM for CloudFront must be us-east-1
  # ...
}

Question 1 — What is a provider, mechanically?

Answer

A provider is a plugin binary that Terraform Core talks to over an internal RPC protocol. It knows how to translate resource/data blocks of its types into actual API calls (e.g. the AWS provider knows how aws_instance maps to EC2 API calls), and how to read the resulting state back. This plugin architecture is what makes Terraform provider-agnostic at its core.

Question 2 — How do you manage resources across multiple AWS accounts or regions in one config?

Answer

Declare multiple provider "aws" blocks with different alias values (each with its own region and/or assume_role for cross-account access), then reference the specific one per resource with the provider = aws.alias_name meta-argument. This is the standard pattern for hub-and-spoke or multi-region architectures in one root module.

Question 3 — How do you authenticate the AWS/Azure provider without hardcoding credentials?

Answer

AWS: rely on the default credential chain — environment variables, shared credentials file, or (best practice in CI/CD and on EC2/EKS) an IAM role via instance profile or IRSA (IAM Roles for Service Accounts), never hardcoded access_key/secret_key in HCL. Azure: use azurerm provider with a Service Principal via environment variables or Managed Identity, again never inline. Both patterns keep secrets out of version control entirely.

Practice Exercise

Task: Configure the AWS provider to assume a cross-account role for managing resources in a separate "prod" AWS account from a shared tooling account.

provider "aws" {
  region = "ap-south-1"

  assume_role {
    role_arn     = "arn:aws:iam::111122223333:role/TerraformDeployRole"
    session_name = "terraform-prod-deploy"
  }
}
Common mistake: hardcoding access_key and secret_key directly in a provider block. Even if the file isn't committed, this pattern encourages copy-pasting credentials between machines and CI systems — use the default credential chain and IAM roles instead.
Interview tip: know that provider version constraints matter as much as Terraform core version constraints — an unpinned aws provider can introduce breaking changes (renamed arguments, changed defaults) on a routine init in CI with no code change on your end.
Chapter 5

Resources

The resource block is the fundamental unit of Terraform — anything Terraform creates and manages the lifecycle of. This chapter covers the mechanics interviewers probe: dependency resolution, resource addressing, and the create/update/replace decision.

Core Concepts

resource "aws_security_group" "web" {
  name        = "web-sg"
  description = "Allow HTTP/HTTPS"
  vpc_id      = aws_vpc.main.id      # implicit dependency — Terraform infers ordering

  ingress {
    from_port   = 443
    to_port     = 443
    protocol    = "tcp"
    cidr_blocks = ["0.0.0.0/0"]
  }
}

resource "aws_instance" "web" {
  ami                    = data.aws_ami.latest.id
  instance_type          = "t3.micro"
  vpc_security_group_ids = [aws_security_group.web.id]

  depends_on = [aws_internet_gateway.main]   # explicit dependency when no attribute reference exists
}

Question 1 — How does Terraform know the order to create resources in?

Answer

By default, Terraform builds a dependency graph from implicit references — any time one resource's argument references another resource's attribute (e.g. vpc_id = aws_vpc.main.id), Terraform knows to create the VPC first. Resources with no relationship to each other are created in parallel for speed. Use depends_on only when a dependency exists that isn't expressed through an attribute reference (e.g. IAM policy propagation timing).

Question 2 — What triggers an in-place update vs a destroy-and-recreate?

Answer

It depends on the specific argument and provider — each resource schema marks each argument as either updatable in place or ForceNew (requires replacement). Changing an EC2 instance's tags is an in-place update; changing its availability_zone forces replacement, since AZ can't be changed on a running instance. terraform plan always shows which: ~ for update in place, -/+ for destroy-and-recreate.

Question 3 — What is a resource address, and why does it matter?

Answer

The address is resource_type.resource_name (or module.name.resource_type.resource_name inside a module) — it's the unique key Terraform uses to track a resource in state. It matters for targeted operations (terraform apply -target=aws_instance.web), for terraform state mv during refactors, and because renaming a resource block changes its address, which Terraform sees as destroy-old + create-new unless you use a moved block.

Question 4 — What does the lifecycle block's create_before_destroy do, and why does it matter for zero-downtime changes?

resource "aws_launch_template" "app" {
  name_prefix = "app-"
  image_id    = var.ami_id

  lifecycle {
    create_before_destroy = true
  }
}

Answer

By default, when a resource must be replaced, Terraform destroys the old one first, then creates the new one — causing downtime for anything depending on it. create_before_destroy = true reverses that: the new resource is created first, dependent resources are re-pointed, and only then is the old one destroyed. Essential for things like launch templates behind an ASG, or any resource fronting live traffic.

Practice Exercise

Task: Write a security group and an EC2 instance that references it, using implicit dependency (no explicit depends_on).

resource "aws_security_group" "app" {
  name   = "app-sg"
  vpc_id = aws_vpc.main.id
}

resource "aws_instance" "app" {
  ami                    = "ami-0abcd1234"
  instance_type          = "t3.small"
  vpc_security_group_ids = [aws_security_group.app.id]   # implicit dependency
}
Common mistake: reaching for depends_on by default instead of letting Terraform infer dependencies from attribute references. Excessive explicit depends_on makes the graph harder to reason about and can hide real bugs in how resources actually relate.
Interview tip: when asked about zero-downtime infrastructure changes, create_before_destroy is the answer interviewers are fishing for — mention it even if the question is phrased generically ("how do you avoid downtime during a Terraform change").
Chapter 6

Variables & Types

Variables are how configs stay reusable across environments. Interviewers check whether you know the precedence order and use validation/types deliberately, not just type = any everywhere.

Core Concepts

variable "instance_type" {
  type        = string
  default     = "t3.micro"
  description = "EC2 instance type for the app tier"
}

variable "allowed_cidrs" {
  type    = list(string)
  default = ["10.0.0.0/8"]
}

variable "tags" {
  type    = map(string)
  default = {}
}

variable "environment" {
  type = string

  validation {
    condition     = contains(["dev", "staging", "prod"], var.environment)
    error_message = "environment must be one of: dev, staging, prod."
  }
}

variable "db_password" {
  type      = string
  sensitive = true    # hides value from plan/apply output and logs
}

Question 1 — What is the variable precedence order (highest to lowest)?

Answer

  1. Command-line -var or -var-file flags
  2. *.auto.tfvars / *.auto.tfvars.json files (alphabetical order)
  3. terraform.tfvars / terraform.tfvars.json
  4. TF_VAR_name environment variables
  5. The default value in the variable block (lowest priority)

In practice: CLI flags win, environment variables are a convenient CI mechanism, and .tfvars files are the standard way to separate per-environment values (prod.tfvars, staging.tfvars).

Question 2 — What does marking a variable sensitive = true actually do (and not do)?

Answer

It redacts the value from CLI output (plan/apply show (sensitive value)) and from being displayed in error messages. It does not encrypt the value in the state file — sensitive values are still stored in plaintext in terraform.tfstate, which is why state must be stored in an encrypted, access-controlled backend regardless of variable sensitivity settings.

Question 3 — Why use validation blocks instead of just documenting constraints?

Answer

Validation blocks fail fast at plan time with a clear error message, before any API calls happen — catching a typo'd environment name or an out-of-range value immediately instead of after a partial apply or a downstream resource failure. This is a cheap, high-value guardrail, especially in shared/team modules.

Practice Exercise

Task: Define a variable for an instance count that must be between 1 and 10, with a validation block.

variable "instance_count" {
  type    = number
  default = 2

  validation {
    condition     = var.instance_count >= 1 && var.instance_count <= 10
    error_message = "instance_count must be between 1 and 10."
  }
}
Common mistake: assuming sensitive = true protects a secret end-to-end. It only redacts CLI output — the value is still in plaintext state. Real secrets belong in a secrets manager (Vault, AWS Secrets Manager, Azure Key Vault) referenced via data source, not passed as a plain Terraform variable at all where avoidable.

Interview tip: if asked how you manage per-environment configuration, describe *.tfvars files per environment (prod.tfvars, staging.tfvars) passed with -var-file, combined with workspaces or separate state files — this is the standard, expected answer.
Chapter 7

Outputs & Data Sources

Outputs expose values for humans or other configs to consume; data sources read existing infrastructure Terraform doesn't manage. Both come up constantly in module design and cross-stack references.

Core Concepts

output "instance_public_ip" {
  value       = aws_instance.web.public_ip
  description = "Public IP of the web server"
}

output "db_connection_string" {
  value     = "postgres://${aws_db_instance.main.endpoint}/app"
  sensitive = true    # hides the value in CLI output
}

# Data source — reads existing infrastructure, doesn't manage it
data "aws_ami" "latest_amazon_linux" {
  most_recent = true
  owners      = ["amazon"]

  filter {
    name   = "name"
    values = ["al2023-ami-*-x86_64"]
  }
}

resource "aws_instance" "web" {
  ami = data.aws_ami.latest_amazon_linux.id   # use the looked-up value
}

Question 1 — Resource vs data source — what's the difference?

Answer

A resource block is created, updated, and destroyed by Terraform — it's in the state and Terraform owns its lifecycle. A data block only reads information about something that already exists (looked up via provider API), and is never created or destroyed by Terraform. Use data sources for referencing shared/pre-existing infrastructure (an existing VPC, the latest AMI, an existing Route53 zone) that this config shouldn't own.

Question 2 — How do you share outputs between separate Terraform states (e.g. a network stack and an app stack)?

Answer

Two common patterns: (1) terraform_remote_state data source, which reads another state file's outputs directly (requires read access to that backend); (2) writing outputs to a shared parameter store (AWS SSM Parameter Store, Azure App Configuration) that the consuming stack reads via a data source. Pattern 2 is generally preferred at scale — it decouples the stacks so the consumer doesn't need direct backend access to the producer's state.

data "terraform_remote_state" "network" {
  backend = "s3"
  config = {
    bucket = "my-tfstate-bucket"
    key    = "prod/network.tfstate"
    region = "ap-south-1"
  }
}

resource "aws_instance" "app" {
  subnet_id = data.terraform_remote_state.network.outputs.private_subnet_id
}

Question 3 — Why mark an output sensitive, and what does it actually protect?

Answer

Same caveat as sensitive variables: it hides the value from terraform output and apply CLI output, but the value is still stored in plaintext in state. It's a UX/accidental-exposure guard (stops a value leaking into CI logs on screen), not encryption.

Practice Exercise

Task: Look up the default VPC via data source and output its ID and CIDR block.

data "aws_vpc" "default" {
  default = true
}

output "default_vpc_id" {
  value = data.aws_vpc.default.id
}

output "default_vpc_cidr" {
  value = data.aws_vpc.default.cidr_block
}
Common mistake: using a resource block to "import" infrastructure you don't actually want Terraform to manage or destroy — e.g. accidentally re-declaring an existing production VPC as a managed resource, risking a future destroy tearing it down. If you only need to reference something, use a data source.
Interview tip: know terraform_remote_state by name — it's the standard mechanism for cross-stack references and comes up in almost every "how do you structure multiple Terraform configs" question.
Chapter 8

State Management Fundamentals

State is the single most-tested Terraform topic in senior interviews. It's the thing that, when misunderstood, causes real production incidents — so interviewers spend real time here.

Core Concepts

Terraform state (terraform.tfstate) is a JSON file that maps every resource in your config to the real-world object it represents (e.g. aws_instance.web → EC2 instance i-0abc123) and caches all of that resource's attributes. State is what makes Terraform fast (it doesn't need to query every provider API on every plan) and is what allows Terraform to detect drift and compute diffs at all. Without state, Terraform would have no way to know a resource it previously created still exists, or what its current values are.

Question 1 — Why does Terraform need state at all — why not just query the cloud provider live every time?

Answer

Three reasons: performance (querying every resource live on every command would be slow at scale), mapping (some resources have no unique, discoverable identifier that ties them back to a specific config block without a stored reference), and metadata (state stores things not present in the live resource, like the dependency graph and resource addresses). State is Terraform's source of truth for "what do I currently manage and what does it look like."

Question 2 — What is configuration drift, and how does Terraform detect it?

Answer

Drift is when real infrastructure no longer matches what's in state — typically caused by a manual console change, another tool, or an out-of-band API call. On terraform plan (unless -refresh=false), Terraform refreshes state by querying the actual provider APIs and compares that against both the stored state and the desired config, surfacing drift as part of the diff.

Question 3 — Why is storing state locally (terraform.tfstate in the repo or on a laptop) dangerous for a team?

Answer

Local state means no locking (two people running apply simultaneously can corrupt state or race), no shared source of truth (everyone's local copy can drift from each other), it often contains secrets in plaintext that shouldn't be committed to Git, and there's no backup/versioning if a laptop is lost. This is why any real team setup uses a remote backend (Chapter 9) from day one.

Question 4 — What's the difference between terraform refresh, plan, and apply with respect to state?

Answer

refresh (now folded into plan/apply by default, or run standalone as terraform apply -refresh-only) updates state to match real infrastructure without changing any config or infrastructure. plan refreshes, then diffs against config. apply refreshes, diffs, then executes changes and writes the new state.

Practice Exercise

Task: Explain what happens, step by step, if someone manually deletes an EC2 instance in the AWS console that Terraform manages, and then someone runs terraform plan.

Model answer: "On plan, Terraform refreshes state and discovers the instance no longer exists in AWS. It updates its in-memory understanding to reflect that, then diffs against the config, which still declares the instance — so the plan shows the instance will be created (not updated), since as far as Terraform's refreshed state is concerned, it doesn't currently exist. Applying that plan recreates it."
Common mistake: treating the state file as unimportant or disposable ("we can always re-import"). Losing state without a backup means Terraform loses all mapping to existing resources — re-importing everything by hand at scale is a serious, error-prone recovery effort, not a quick fix.
Interview tip: if asked "what is Terraform state," don't just define it — immediately connect it to why remote backends and locking exist. That's the follow-up question coming next anyway, and answering it pre-emptively signals depth.
Chapter 9

Remote Backends & State Locking

This is the practical, production half of the state story — how a real team shares state safely. Expect a live-whiteboard question here in almost any senior interview.

Core Concepts

terraform {
  backend "s3" {
    bucket         = "my-company-tfstate"
    key            = "prod/network/terraform.tfstate"
    region         = "ap-south-1"
    dynamodb_table = "terraform-locks"   # state locking via DynamoDB
    encrypt        = true
  }
}

# Azure equivalent
terraform {
  backend "azurerm" {
    resource_group_name  = "tfstate-rg"
    storage_account_name = "tfstatestorage"
    container_name        = "tfstate"
    key                    = "prod/network.terraform.tfstate"
  }
}

Question 1 — What does a remote backend give you that local state doesn't?

Answer

A single shared source of truth the whole team and CI read from, state locking (preventing concurrent apply operations from corrupting state), versioning/history if the backend supports it (S3 bucket versioning), and encryption at rest for a file that may contain secrets. It turns Terraform from a single-operator tool into something a team can safely run concurrently.

Question 2 — How does state locking actually work with S3 + DynamoDB?

Answer

Before any operation that could write state (plan acquires a read-ish lock in newer versions; apply always locks), Terraform writes a lock record to the DynamoDB table keyed by the state file's path. If another operation tries to run concurrently, it sees the existing lock and fails fast with a "state locked" error instead of racing. The lock is released when the operation completes or is interrupted cleanly.

Question 3 — A teammate's terraform apply got killed (Ctrl-C, CI timeout) and now every apply fails with "Error acquiring the state lock." What do you do?

Answer

First confirm no other apply is actually still running (check CI, ask the team) — force-unlocking a genuinely active operation corrupts state. Once confirmed safe, run terraform force-unlock <LOCK_ID> (the lock ID is printed in the error message). This is a manual, deliberate action precisely because it bypasses the safety mechanism — never script it to run automatically.

terraform force-unlock 8f14e45f-ceea-4d16-8f1a-example1234

Question 4 — What backends are commonly used per cloud, and is DynamoDB locking still needed with S3?

Answer

AWS: S3 for state storage + DynamoDB for locking (as of Terraform 1.x, native S3 locking via conditional writes is being introduced but DynamoDB is still the widely-deployed standard). Azure: azurerm backend using Blob Storage, which has built-in lease-based locking — no separate lock table needed. GCP: Google Cloud Storage backend, similarly with built-in locking. Terraform Cloud/Enterprise is a fully managed backend with locking, a UI, and policy enforcement (Sentinel) built in.

State Commands & Troubleshooting

Hands-on state surgery — the commands you reach for when state and reality have diverged, or when refactoring config without wanting to destroy and recreate real resources.

Core Commands

terraform state list                          # list all resources tracked in state
terraform state show aws_instance.web         # show one resource's full attributes
terraform state mv aws_instance.web aws_instance.app   # rename/move within state (no infra change)
terraform state rm aws_instance.old           # stop tracking a resource (does NOT destroy it)
terraform import aws_instance.web i-0abc123   # bring an existing resource under management
terraform state pull > state.json             # download current state
terraform state push state.json                # upload modified state (dangerous — last resort)

Question 1 — You renamed a resource block in your .tf files (e.g. aws_instance.web → aws_instance.app). What happens if you just run apply, and how do you avoid it?

Answer

Terraform sees the old address gone and a new address appeared, so it plans to destroy the old resource and create a new one — even though nothing about the actual infrastructure needs to change. To avoid the unnecessary destroy/recreate (and potential downtime/data loss), use terraform state mv aws_instance.web aws_instance.app first, which updates the state mapping without touching real infrastructure. Modern Terraform (1.1+) also supports a declarative moved block in config for the same purpose, which is preferred since it's version-controlled and self-documenting.

moved {
  from = aws_instance.web
  to   = aws_instance.app
}

Question 2 — What does terraform import do, and what's its main limitation?

Answer

It brings an already-existing resource (created outside Terraform, e.g. manually in the console) under Terraform's management by adding it to state, mapped to a resource block you must already have written in config. Its main limitation: import only populates state — it does not generate the matching HCL for you (older versions at least; newer Terraform has experimental -generate-config-out to help). You must write a resource block whose arguments closely match the real resource's current values, or the next plan shows a large diff.

Question 3 — What's the difference between terraform state rm and terraform destroy -target?

Answer

state rm removes a resource from Terraform's tracking without touching the real infrastructure at all — the resource keeps running, Terraform just "forgets" about it. destroy -target actually deletes the real resource. Use state rm when you want Terraform to stop managing something (e.g. handing it off to another team/tool) but keep it running.

Question 4 — State file appears corrupted or has a manual bad edit. Recovery steps?

Answer

First, don't panic-apply. Pull the current state (terraform state pull), and if the backend has versioning (S3 bucket versioning is the standard setup), restore the last known-good version. If no backup exists, reconstruct via terraform import for each resource, one at a time, verifying with plan after each that the diff is empty (or expected). This is exactly the pain remote backend versioning exists to prevent — a good answer here loops back to "this is why we always enable state versioning."

Chapter 11

Modules

Modules are how Terraform scales beyond a single team's single config — reusable, versioned building blocks. Interviewers use this chapter to check design judgment, not just syntax.

Core Concepts

# modules/vpc/main.tf — a reusable module
variable "cidr_block" { type = string }
variable "name"       { type = string }

resource "aws_vpc" "this" {
  cidr_block = var.cidr_block
  tags       = { Name = var.name }
}

output "vpc_id" {
  value = aws_vpc.this.id
}

# root module — calling it
module "prod_network" {
  source     = "./modules/vpc"
  cidr_block = "10.0.0.0/16"
  name       = "prod"
}

module "staging_network" {
  source     = "./modules/vpc"
  cidr_block = "10.1.0.0/16"
  name       = "staging"
}

resource "aws_instance" "app" {
  subnet_id = module.prod_network.vpc_id   # reference a module's output
}

Question 1 — What makes a well-designed reusable module?

Answer

Clear, minimal required inputs with sensible defaults for optional ones; outputs that expose everything a caller is likely to need to reference (IDs, ARNs) without over-exposing internals; no hardcoded values that should be configurable (region, naming, environment); and no assumptions about what calls it (avoid baking in a specific provider alias or backend inside a reusable module — that belongs in the root). A good module is boring and predictable to use.

Question 2 — Where can module sources come from?

Answer

Local paths (./modules/vpc), the public Terraform Registry (terraform-aws-modules/vpc/aws), a private registry (Terraform Cloud/Enterprise), or directly from Git (git::https://github.com/org/repo.git//path?ref=v1.2.0). Production setups typically pin Git or registry modules to a specific tagged version — never an unpinned branch — for the same reproducibility reason providers are version-pinned.

Question 3 — When would you use a public registry module (e.g. terraform-aws-modules/vpc/aws) vs writing your own?

Answer

Well-maintained public modules for common, well-understood patterns (VPC, EKS cluster, RDS) save significant time and encode a lot of hard-won edge-case handling — generally worth using for standard infrastructure. Write your own when the pattern is specific to your org's conventions (naming, tagging policy, security baselines) or when you need tight control over exactly what gets created for compliance reasons. A common middle ground: wrap a public module inside a thin internal module that enforces your org's required tags/naming.

Question 4 — How do you structure a repo with multiple environments sharing common modules?

repo/
├── modules/
│   ├── vpc/
│   ├── eks/
│   └── rds/
├── environments/
│   ├── dev/
│   │   ├── main.tf        # calls modules with dev-specific inputs
│   │   └── backend.tf     # dev's own state file
│   ├── staging/
│   └── prod/
│       ├── main.tf
│       └── backend.tf     # separate state — blast radius isolation

Answer

Shared logic lives once in modules/; each environment is its own root module with its own state file and backend key, calling the shared modules with different inputs. This isolates blast radius — a mistake in a dev apply can't touch prod state — while avoiding duplicated resource logic.

Chapter 12

Meta-Arguments: count, for_each, depends_on, lifecycle

These four meta-arguments show up in nearly every real config. The count vs for_each choice specifically is one of the most common "gotcha" interview questions because getting it wrong causes real destroy/recreate incidents.

Core Concepts

# count — index-based, good for identical N copies
resource "aws_instance" "web" {
  count         = 3
  ami           = var.ami_id
  instance_type = "t3.micro"
  tags          = { Name = "web-${count.index}" }
}
# addresses: aws_instance.web[0], aws_instance.web[1], aws_instance.web[2]

# for_each — key-based, good for a map/set of distinct items
resource "aws_instance" "web" {
  for_each      = toset(["web-a", "web-b", "web-c"])
  ami           = var.ami_id
  instance_type = "t3.micro"
  tags          = { Name = each.key }
}
# addresses: aws_instance.web["web-a"], aws_instance.web["web-b"], ...

Question 1 — Why is for_each generally preferred over count for a list of distinct named resources?

Answer

With count, resources are tracked by numeric index. If you remove an item from the middle of the list (or the list's order changes), every subsequent index shifts — Terraform sees this as destroying and recreating every resource after that point, even though most of them didn't logically change. With for_each, resources are tracked by a stable key (from a map or set), so removing one item only affects that one resource; everything else is untouched. This difference is a real, common cause of unintended production destroy/recreate.

Question 2 — When is count still the right choice?

Answer

When you genuinely want N identical, interchangeable copies of something and don't need to reference them individually by a meaningful name — or for the common conditional-resource pattern (count = var.create_resource ? 1 : 0) to optionally create a single resource.

resource "aws_eip" "nat" {
  count = var.enable_nat_gateway ? 1 : 0
  vpc   = true
}

Question 3 — What does the lifecycle block's prevent_destroy do?

resource "aws_db_instance" "prod" {
  # ...
  lifecycle {
    prevent_destroy = true
  }
}

Answer

It makes Terraform refuse to destroy that specific resource — any plan that would destroy it (including via terraform destroy or a forced replacement) fails with an error instead of proceeding. It's a guardrail for genuinely critical, stateful resources like production databases, where an accidental destroy would be catastrophic and hard to reverse.

Question 4 — What does ignore_changes do, and when is it appropriate?

resource "aws_instance" "app" {
  # ...
  lifecycle {
    ignore_changes = [tags["LastDeployedBy"]]   # or ignore_changes = all
  }
}

Answer

It tells Terraform to stop showing/planning changes for specific arguments even if they drift from config — useful when something else legitimately manages that field outside Terraform (e.g. an autoscaler adjusting desired_capacity, or a CI pipeline tagging deploys). Overusing it is a smell: it means either Terraform shouldn't own that resource fully, or two systems are fighting over the same field.

Interview tip: if a question describes "we removed one item from a list of resources and Terraform wanted to recreate everything after it" — that's a count vs for_each question in disguise. Naming that pattern immediately is a strong signal.

Expressions, Functions & Dynamic Blocks

HCL's built-in functions and dynamic blocks are what let modules stay DRY and flexible. Interviewers use these to check comfort with the language beyond static resource declarations.

Core Concepts — Common Functions

length(["a", "b", "c"])              # 3
concat(["a"], ["b", "c"])              # ["a", "b", "c"]
merge({a=1}, {b=2})                     # {a=1, b=2}
lookup(var.map, "key", "default")       # safe map lookup with fallback
coalesce(var.a, var.b, "fallback")      # first non-null value
join(",", ["a", "b", "c"])              # "a,b,c"
split(",", "a,b,c")                     # ["a", "b", "c"]
for s in var.subnets : s.id             # for expression (list comprehension style)
{ for k, v in var.map : k => upper(v) } # for expression producing a map
try(var.optional.value, "default")      # safe access, avoids errors on nulls
templatefile("user-data.tpl", { name = var.name })  # render a template file

Question 1 — What is a for expression, and give a practical example

variable "subnets" {
  type = list(object({ id = string, az = string }))
}

# Produce just the list of subnet IDs
locals {
  subnet_ids = [for s in var.subnets : s.id]
}

# Produce a map keyed by AZ
locals {
  subnets_by_az = { for s in var.subnets : s.az => s.id }
}

Answer

A for expression transforms a list or map into another list or map inline, similar to a Python list/dict comprehension. Extremely common for reshaping module inputs/outputs without a separate resource or data source.

Question 2 — What is a dynamic block, and when do you need one?

resource "aws_security_group" "app" {
  name = "app-sg"

  dynamic "ingress" {
    for_each = var.allowed_ports
    content {
      from_port   = ingress.value
      to_port     = ingress.value
      protocol    = "tcp"
      cidr_blocks = ["0.0.0.0/0"]
    }
  }
}

Answer

Normal nested blocks (like ingress {} inside a security group) are static — you can't loop over them directly. A dynamic block generates a variable number of nested blocks from a list or map, which is essential when the number of ingress rules, tags, or similar repeated nested blocks is only known from a variable, not hardcoded.

Question 3 — What does try() protect against, and why not just use a ternary?

Answer

try() evaluates a sequence of expressions and returns the first one that doesn't produce an error — useful for safely accessing optional/nested attributes that might not exist without the whole plan erroring out. A ternary (var.x != null ? var.x.value : "default") still errors if var.x.value itself is invalid to evaluate in some branch; try() is specifically built to swallow evaluation errors, which a ternary's untaken branch doesn't protect against in all cases.

Practice Exercise

Task: Given a variable map of environment → instance count, write a for expression producing a list of "environment: count" strings.

  type    = map(number)
  default = { dev = 1, staging = 2, prod = 5 }
}

  summary = [for env, count in var.env_sizes : "${env}: ${count}"]
}
# ["dev: 1", "staging: 2", "prod: 5"]
Common mistake: hardcoding repeated nested blocks (multiple ingress {} blocks copy-pasted with small tweaks) instead of using a dynamic block driven by a variable — makes the module inflexible and violates DRY for something that's inherently a list.
Interview tip: know lookup() and try() by name for "safely handle an optional/missing value" questions — these come up constantly in module design discussions.

Workspaces & Environment Strategies

Workspaces are a genuinely contested topic — interviewers use this chapter to see if you have an informed opinion, not just textbook knowledge, since teams disagree on this in practice.

Core Concepts

terraform workspace new staging
terraform workspace select staging
terraform workspace list
terraform workspace show

# Referencing the current workspace in config
resource "aws_instance" "app" {
  instance_type = terraform.workspace == "prod" ? "m5.large" : "t3.micro"
  tags          = { Environment = terraform.workspace }
}

Question 1 — What do Terraform workspaces actually do, mechanically?

Answer

Workspaces let one set of .tf config files manage multiple, separate state files (all under the same backend configuration, differentiated by a workspace-suffixed key). Switching workspace (terraform workspace select) changes which state file subsequent commands read/write, without changing the config on disk.

Question 2 — What's the main risk of using workspaces for dev/staging/prod?

Answer

Because all environments share the exact same config, a mistake in terraform.workspace conditional logic, or simply forgetting which workspace is currently selected, can apply changes intended for staging directly to prod — there's no structural separation, only a runtime switch. This is why many teams consider workspaces risky for genuinely separate environments with different blast-radius requirements.

Question 3 — What's the alternative, and when is each approach right?

Answer

The alternative is separate directories/root modules per environment (environments/dev, environments/prod), each with its own backend config and .tfvars, calling shared modules — see Chapter 11's structure. This gives true isolation (different AWS accounts, different state files, different IAM permissions per environment are trivial to enforce) at the cost of some duplication. Workspaces are genuinely well-suited for short-lived, ephemeral, structurally-identical environments (e.g. a workspace per feature-branch preview environment, or per-PR test infrastructure) where the risk profile is low and the convenience of one config is worth it. Separate directories are the standard, safer choice for dev/staging/prod with real blast-radius differences.

Common mistake: using workspaces for dev/staging/prod and then relying purely on discipline ("just remember to check terraform workspace show before applying") instead of structural safeguards. Discipline fails under pressure; separate state/accounts don't.
Interview tip: this is a "do you have real opinions from experience" question. A flat "workspaces are good" or "workspaces are bad" answer reads as textbook; naming the specific risk (shared config = no structural blast-radius isolation) reads as earned.
Chapter 15

Provisioners (and Why to Avoid Them)

HashiCorp's own docs call provisioners a "last resort." Interviewers ask about them specifically to see if you know that and can explain why — not to see if you can write one.

Core Concepts

resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = "t3.micro"

  provisioner "remote-exec" {
    inline = [
      "sudo apt-get update",
      "sudo apt-get install -y nginx",
    ]
    connection {
      type = "ssh"
      user = "ubuntu"
      host = self.public_ip
    }
  }

  provisioner "local-exec" {
    command = "echo ${self.public_ip} >> inventory.txt"
  }
}

Question 1 — Why does HashiCorp recommend avoiding provisioners where possible?

Answer

Provisioners step outside Terraform's declarative model — they run imperative scripts with no idempotency guarantee, no drift detection (Terraform can't know if the remote-exec commands' effects still hold later), and failures mid-provisioning can leave a resource in an inconsistent "tainted" state that's awkward to recover. They also create a hard runtime dependency on network/SSH connectivity to the resource at apply time, which is fragile in CI.

Question 2 — What should you use instead of a remote-exec provisioner to configure a new VM?

Answer

Cloud-init / user_data (AWS/Azure both support this natively) baked into the instance launch itself — it's part of the resource's own lifecycle, idempotent by cloud-init design, and doesn't need live SSH connectivity from wherever terraform apply runs. For more complex configuration, build a custom AMI/image with Packer that already has software installed, so the instance boots ready rather than being configured after the fact. For ongoing configuration management, hand off to Ansible/Chef/Puppet as a separate step after provisioning, not as a Terraform provisioner.

resource "aws_instance" "web" {
  ami           = var.ami_id
  instance_type = "t3.micro"
  user_data     = templatefile("user-data.tpl", { app_version = var.app_version })
}

Question 3 — What is local-exec still reasonably used for?

Answer

Narrow, genuinely one-off local actions that have no Terraform-native equivalent — e.g. invalidating a CDN cache after a deploy, or triggering an external webhook notification. Even here, many teams prefer to do this in the CI/CD pipeline step after terraform apply rather than inside the Terraform run itself, keeping Terraform's job strictly to infrastructure state.

Question 4 — What does when = destroy do on a provisioner?

provisioner "local-exec" {
  when    = destroy
  command = "curl -X POST https://hooks.example.com/instance-terminated"
}

Answer

Runs the provisioner during terraform destroy instead of on creation — useful for cleanup notifications or deregistration steps, but subject to the same "last resort" caution: if destroy fails partway, this may not run reliably.

Question 2 — How do you manage IAM least-privilege for the Terraform execution role itself?

Answer

Scope the CI/CD deploy role's IAM policy to exactly the resource types and actions the config actually manages — not AdministratorAccess. Use separate roles per environment (a prod deploy role distinct from staging), enforce MFA/short-lived credentials via assume_role from CI's OIDC identity rather than long-lived access keys, and review the policy whenever new resource types are added to config. This is a common senior interview follow-up: "the plan looks right, but what CAN this pipeline actually do if compromised?"

Question 3 — How do you handle secrets (DB passwords, API keys) needed by AWS resources, without putting them in Terraform state as plain variables?

data "aws_secretsmanager_secret_version" "db_password" {
  secret_id = "prod/db/password"
}

resource "aws_db_instance" "main" {
  # ...
  password = data.aws_secretsmanager_secret_version.db_password.secret_string
}

Answer

Reference AWS Secrets Manager or SSM Parameter Store via a data source at apply time, rather than passing the raw secret as a Terraform variable. This keeps the secret's lifecycle (rotation, access policy) managed by the secrets service itself. Note the caveat from Chapter 8: the resolved value still ends up in state, so state encryption and access control remain mandatory regardless.

Question 4 — How would you structure Terraform for a multi-account AWS Organization (e.g. separate accounts per environment)?

Answer

One state per account/environment (per Chapter 14's separate-directories pattern), a shared "tooling" account hosting the S3 state bucket and DynamoDB lock table (with cross-account bucket policies granting each environment's deploy role access to only its own state key), and per-account provider blocks using assume_role from a central CI identity. This gives IAM-enforced blast-radius isolation between environments, which is the actual goal.

Common mistake: giving the Terraform CI/CD execution role broad AdministratorAccess "to avoid permission errors while iterating," then never tightening it before production use — a significant blast-radius and security-audit finding.
Interview tip: when designing any AWS resource live, narrate security defaults unprompted (encryption at rest, no public access, least-privilege IAM) — interviewers evaluating an 11-year SRE expect security-by-default thinking without being asked.
Chapter 17

Terraform with Azure

The azurerm provider's resource model differs from AWS in a few structural ways interviewers like to probe — particularly resource groups and the identity/auth story.

Question 1 — What is a Resource Group, and how does it change how you structure Terraform on Azure vs AWS?

Answer

Every Azure resource must belong to a Resource Group — a logical container that also scopes lifecycle (deleting the RG deletes everything in it) and often RBAC. Unlike AWS, where resources exist somewhat independently in a region/VPC, Azure configs almost always start with an azurerm_resource_group resource, and most other resources reference its name and location. This makes RG design (one per environment? per application? per team?) an early, consequential decision.

resource "azurerm_resource_group" "main" {
  name     = "rg-checkout-prod"
  location = "Central India"
}

resource "azurerm_virtual_network" "main" {
  name                = "vnet-checkout-prod"
  address_space       = ["10.0.0.0/16"]
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
}

Question 2 — How do you authenticate the azurerm provider in CI/CD without long-lived secrets?

Answer

Use a Service Principal with Federated Identity Credentials (Workload Identity Federation via OIDC) from the CI system (GitHub Actions, Azure DevOps) — this issues short-lived tokens with no stored secret at all. The older pattern (a Service Principal with a stored client_secret in a pipeline variable) still works but is a long-lived credential that needs rotation and is a bigger blast radius if leaked.

provider "azurerm" {
  features {}
  use_oidc        = true
  subscription_id = var.subscription_id
  tenant_id       = var.tenant_id
  client_id       = var.client_id   # federated identity — no client_secret
}

Question 3 — Deploy an AKS cluster's core shape — what are the essential arguments?

resource "azurerm_kubernetes_cluster" "main" {
  name                = "aks-checkout-prod"
  location            = azurerm_resource_group.main.location
  resource_group_name = azurerm_resource_group.main.name
  dns_prefix          = "checkout-prod"

  default_node_pool {
    name       = "system"
    node_count = 3
    vm_size    = "Standard_D2s_v5"
  }

  identity {
    type = "SystemAssigned"   # managed identity — no stored credentials for AKS itself
  }
}

Answer

Resource group + location, dns_prefix (used in the cluster's generated FQDN), a default_node_pool block (the system node pool — additional pools are separate azurerm_kubernetes_cluster_node_pool resources), and an identity block — SystemAssigned managed identity is the standard, credential-free way for the cluster to authenticate to other Azure resources.

Question 4 — How do you read a secret from Azure Key Vault into a Terraform-managed resource?

data "azurerm_key_vault" "main" {
  name                = "kv-checkout-prod"
  resource_group_name = azurerm_resource_group.main.name
}

data "azurerm_key_vault_secret" "db_password" {
  name         = "db-password"
  key_vault_id = data.azurerm_key_vault.main.id
}

resource "azurerm_postgresql_flexible_server" "main" {
  administrator_password = data.azurerm_key_vault_secret.db_password.value
  # ...
}

Answer

Same pattern as AWS Secrets Manager (Chapter 16): reference the vault and secret via data sources rather than a plain variable. The Terraform identity running this needs a Key Vault access policy or RBAC role (Key Vault Secrets User) granting Get permission on secrets.

Terraform with Kubernetes & Helm

Using Terraform to manage Kubernetes objects (versus kubectl/GitOps) is a genuinely debated pattern. Interviewers use this chapter to see if you know the trade-off, not just the provider syntax.

Core Concepts

provider "kubernetes" {
  host                   = module.eks.cluster_endpoint
  cluster_ca_certificate = base64decode(module.eks.cluster_ca_certificate)
  token                  = data.aws_eks_cluster_auth.main.token
}

provider "helm" {
  kubernetes {
    host                   = module.eks.cluster_endpoint
    cluster_ca_certificate = base64decode(module.eks.cluster_ca_certificate)
    token                  = data.aws_eks_cluster_auth.main.token
  }
}

resource "kubernetes_namespace" "app" {
  metadata { name = "checkout" }
}

resource "helm_release" "nginx_ingress" {
  name       = "nginx-ingress"
  repository = "https://kubernetes.github.io/ingress-nginx"
  chart      = "ingress-nginx"
  version    = "4.10.0"
  namespace  = "ingress-nginx"

  set {
    name  = "controller.service.type"
    value = "LoadBalancer"
  }
}

Question 1 — Should Terraform manage application workloads (Deployments, Services) inside Kubernetes, or just the cluster itself?

Answer

The strong convention in most production setups: Terraform provisions the cluster (EKS/AKS control plane, node pools, networking) and cluster-level add-ons that change rarely (ingress controller, cert-manager, cluster autoscaler via the helm provider) — not day-to-day application workloads. Application Deployments/Services are better owned by GitOps (ArgoCD/Flux) or a CI/CD pipeline applying manifests directly, because those change far more frequently than infrastructure and have a different release cadence, review process, and team ownership than the cluster itself. Mixing both in one Terraform state creates a slow, high-blast-radius apply cycle for what should be a fast app deploy.

Question 2 — What's the practical risk of managing Helm releases via the helm_release resource at scale?

Answer

The helm_release resource's plan/diff visibility into what a chart actually changes is weaker than native Helm/GitOps tooling — large or complex chart upgrades can behave unexpectedly under Terraform's model, and a failed helm_release apply can leave Terraform state and the actual Helm release out of sync, requiring manual helm CLI intervention to reconcile. It works well for a handful of stable, infrequently-changed cluster add-ons; it's a poor fit for frequently-deployed application charts.

Question 3 — How do you authenticate the kubernetes/helm providers to a cluster Terraform itself just created (a chicken-and-egg problem)?

Answer

Reference the cluster module's outputs (endpoint, CA cert) and a data source for a short-lived auth token (aws_eks_cluster_auth for EKS, azurerm_kubernetes_cluster's kube_config block for AKS) directly in the provider block, as shown above. This works within a single apply because Terraform resolves the dependency graph — the cluster is created first, then the token is fetched, then Kubernetes/Helm resources are applied. Some teams deliberately split cluster creation and in-cluster resources into two separate Terraform states/applies specifically to avoid this coupling and reduce blast radius.

Chapter 19

CI/CD, Security & Best Practices

The capstone-adjacent chapter on running Terraform safely as a team practice, not just a tool — policy enforcement, drift detection, and the pipeline shape interviewers expect a senior candidate to have opinions on.

Question 1 — Design a safe Terraform CI/CD pipeline, end to end

# .github/workflows/terraform.yml (conceptual shape)
on:
  pull_request:
    paths: ["environments/**"]
  push:
    branches: [main]

jobs:
  plan:
    if: github.event_name == 'pull_request'
    steps:
      - terraform init -input=false
      - terraform validate
      - tflint                          # linting
      - checkov -d .                     # security/policy scan
      - terraform plan -out=tfplan -input=false
      - post plan as PR comment

  apply:
    if: github.ref == 'refs/heads/main'
    steps:
      - terraform init -input=false
      - terraform apply -input=false tfplan   # exact plan from the merged PR

Answer

Plan runs on every PR against protected environment paths, posts the diff for human review as a PR comment, and includes automated linting (tflint) and security scanning (checkov/tfsec) as required checks. Apply only runs on merge to main, using the exact plan file approved in the PR — never a fresh apply that could diff differently. Production environments typically add a manual approval gate before the apply step regardless of automated checks passing.

Question 2 — What is policy-as-code (Sentinel/OPA), and what problem does it solve that code review doesn't?

Answer

Sentinel (Terraform Cloud/Enterprise) and Open Policy Agent (via conftest against plan JSON, cloud-agnostic) let you codify and automatically enforce organizational rules against every plan — e.g. "no S3 bucket may be public," "all resources must have a CostCenter tag," "no instance type larger than X without an exception." Unlike human code review, policy-as-code is consistent, can't be missed by a tired reviewer, and blocks the apply automatically rather than relying on someone catching it.

# Example: run OPA policy checks against a plan in CI
terraform show -json tfplan > plan.json
conftest test plan.json --policy policies/

Question 3 — How do you detect and handle drift proactively, rather than discovering it during the next planned change?

Answer

Run a scheduled (nightly/hourly) CI job that runs terraform plan -detailed-exitcode against production state and alerts (Slack/PagerDuty) if it exits 2 (meaning a non-empty diff exists) without applying anything. This surfaces manual console changes or out-of-band API calls quickly, before they compound or get discovered accidentally during an unrelated legitimate change.

terraform plan -detailed-exitcode
# exit code 0 = no changes; 1 = error; 2 = changes present (drift detected)

Question 4 — What are your top production best practices for a Terraform-managed org, summarized?

  • Remote backend with locking and versioning, from day one — never local state for anything shared.
  • Pin provider and module versions; upgrade deliberately, not automatically.
  • Separate state per environment with real blast-radius isolation (accounts/subscriptions, not just workspaces, for prod).
  • Plan on every PR, apply only on merge, using the exact reviewed plan file.
  • Least-privilege IAM for the CI execution identity, scoped per environment.
  • Secrets via a secrets manager referenced by data source, never as plain committed variables.
  • Policy-as-code (Sentinel/OPA) enforcing security/cost/tagging baselines automatically.
  • Scheduled drift detection with alerting, not just reactive discovery.
  • prevent_destroy on genuinely critical stateful resources (prod databases).

Practice Exercise

Task: A teammate proposes running terraform apply -auto-approve directly on every merge to main, skipping the plan-review-comment step "to move faster." How do you respond?

Model answer: "I'd push back — the plan-then-review step is what makes infrastructure changes as reviewable as code changes; skipping it means merged PRs can apply a different diff than what was actually reviewed if anything drifted in between. If speed is the concern, I'd rather speed up the plan step itself (caching providers, narrowing the plan's scope) than remove the safety gate. For genuinely low-risk paths (e.g. a sandbox environment) I could see auto-apply being reasonable, but not for anything touching shared or production infrastructure."
Common mistake: treating Terraform security as "get the plan to succeed" rather than building in policy enforcement, least-privilege CI identities, and drift detection as first-class parts of the pipeline — the plan succeeding says nothing about whether what it's about to do is safe or compliant.
Interview tip: if asked to design a pipeline, mention the manual approval gate before prod apply even if not asked — it's the detail that separates "runs Terraform" from "runs Terraform safely at a company that would get audited."
Chapter 20

Troubleshooting & Real-World Production Scenarios (Capstone)

The capstone chapter — the scenario-style questions a senior interview panel actually leans on to differentiate candidates. Each follows the same shape: symptoms, investigation, root cause, resolution, prevention.

Scenario 1 — "State locked" and nobody knows why

Symptoms: every terraform plan/apply fails immediately with Error acquiring the state lock.

Investigation: check CI for any currently-running or recently-killed Terraform job; check the DynamoDB lock table directly for the lock record and its age/timestamp.

Root cause: a previous CI job was killed (timeout, cancelled pipeline) mid-apply and never released the lock.

Resolution: confirm nothing is genuinely still running, then terraform force-unlock <LOCK_ID>.

Prevention: set a reasonable CI job timeout that's longer than any expected apply, and alert on force-unlock usage so it's never silent.

Scenario 2 — A routine apply wants to destroy and recreate 40 unrelated resources

Symptoms: plan shows dozens of -/+ replacements for resources nobody touched.

Investigation: diff the actual config change — it's a one-line removal from a count-based list in the middle.

Root cause: classic count index-shift (Chapter 12) — removing an item mid-list shifted every subsequent index.

Resolution: abort the apply; migrate the resource to for_each with terraform state mv for each affected address to avoid the destroy/recreate, then remove the intended item cleanly.

Prevention: default to for_each over count for any list of distinct, individually-referenced resources going forward.

Scenario 3 — Terraform wants to recreate a database because someone renamed a resource block

Symptoms: plan shows aws_db_instance.old_name being destroyed and aws_db_instance.new_name being created — for a live production database.

Investigation: confirm it's purely a naming refactor in the PR, not an intentional replacement.

Root cause: renaming a resource's local name changes its state address; Terraform has no way to know it's "the same" resource without being told.

Resolution: terraform state mv aws_db_instance.old_name aws_db_instance.new_name (or a moved block) before applying, so state is remapped without touching the real database.

Prevention: add prevent_destroy = true on the database resource as a hard backstop, and require moved blocks as a matter of course whenever a PR renames a resource.

Scenario 4 — CI's Terraform apply fails after a routine terraform init with no code changes

Symptoms: pipeline that worked yesterday fails today; error references an argument that "doesn't exist" on a resource that hasn't changed.

Investigation: check the provider version actually downloaded — terraform init re-resolves within an unpinned or loosely-pinned version constraint.

Root cause: an unpinned (or >=-only) provider version constraint let CI pull a new major version overnight with breaking schema changes.

Resolution: pin the provider to the last known-good version, verify plan is clean, then deliberately upgrade and test in a lower environment first.

Prevention: always use ~> pessimistic constraints (or exact pins) on provider versions in the terraform block, never open-ended >=.

Scenario 5 — A secret value shows up in a CI log despite being marked sensitive

Symptoms: a database password appears in plaintext in a build log, even though the variable has sensitive = true.

Investigation: find where the value flows — it's being interpolated into a non-sensitive local_exec command or an unmarked output that references it indirectly.

Root cause: sensitivity doesn't automatically propagate through every expression in older Terraform versions the same way, and any place the value is echoed outside a properly-marked sensitive context can leak it. (Modern Terraform propagates sensitivity through most expressions, but provisioner commands and external scripts are common escape hatches.)

Resolution: remove the value from the offending command/output; rotate the leaked secret immediately regardless of the fix (assume anything that touched a log is compromised).

Prevention: avoid passing real secrets through Terraform variables at all where possible — reference secrets managers via data source at the point of use, and treat sensitive = true as a UX safeguard, not a security boundary.

Scenario 6 — State file appears to be missing several resources that definitely exist in AWS

Symptoms: terraform plan wants to create resources that are already running in AWS — applying would create duplicates.

Investigation: terraform state list confirms those resources aren't tracked; check if someone ran terraform state rm previously, or if state was restored from an older backup.

Root cause: a stale state restore, or resources created manually outside Terraform that were never imported.

Resolution: do not apply. terraform import each missing resource individually, verifying an empty (or expected) diff via plan after each import before moving to the next.

Prevention: enable state file versioning on the backend (S3 bucket versioning) so any bad state write can be rolled back precisely, and restrict who can run state rm/state push in production.

Scenario 7 — Two engineers ran apply at nearly the same time against a local state file, and now state is corrupted

Symptoms: terraform plan errors on invalid JSON in the state file, or shows resources with clearly wrong/duplicated attributes.

Investigation: confirm there's no remote backend (this only happens with local or unlocked state) and that two concurrent applies did in fact race.

Root cause: no state locking mechanism existed to prevent the concurrent write.

Resolution: restore the last valid state (from a manual backup, or reconstruct via careful import if none exists), verify against real infrastructure resource by resource.

Prevention: this incident is the canonical argument for a remote backend with locking (Chapter 9) from day one — it's precisely the failure mode locking exists to prevent, and is a strong real-world story to have ready for "tell me about a Terraform incident" interview questions.

Scenario 8 — A module upgrade silently changes behavior in production

Symptoms: after a routine terraform init -upgrade, plan shows unexpected changes to resources managed by a third-party module.

Investigation: check the module source reference — it's pointed at a Git branch (ref=main) or an unpinned registry version, not a tagged release.

Root cause: the module's upstream maintainer pushed a breaking change to the branch/latest version being referenced.

Resolution: pin the module to the last known-good tagged version immediately; evaluate the new version's changelog deliberately before upgrading again.

Prevention: always pin module sources to specific tagged versions (ref=v2.3.1, or an exact registry version constraint), exactly as you would a provider.

Quick Answer Bank (Rapid Review)

  1. What is Terraform state? A JSON mapping of config resources to real-world infrastructure, used to compute diffs and detect drift.
  2. Why remote backend over local state? Locking, shared source of truth, encryption, versioning/backup.
  3. count vs for_each? count is index-based (fragile on reordering); for_each is key-based (stable per-item tracking).
  4. What does terraform plan -out do? Saves an exact plan to apply later, guaranteeing apply matches what was reviewed.
  5. How to rename a resource without destroy/recreate? terraform state mv, or a moved block.
  6. What does prevent_destroy do? Blocks any plan from destroying that specific resource.
  7. What does terraform import not do? Generate matching HCL automatically (in most versions) — only updates state.
  8. Why avoid provisioners? No idempotency, no drift detection, fragile runtime SSH dependency; prefer user_data/cloud-init.
  9. What enforces org policy automatically on every plan? Policy-as-code — Sentinel or OPA/conftest.
  10. How to detect drift proactively? Scheduled terraform plan -detailed-exitcode with alerting on exit code 2.
  11. Workspaces vs separate directories for prod? Separate directories/accounts give real blast-radius isolation; workspaces share one config and are riskier for high-stakes environments.
  12. Where do secrets belong instead of plain variables? A secrets manager (Secrets Manager, Key Vault) referenced via data source.
  13. What causes "Error acquiring the state lock"? A previous operation still holds the lock, or was killed without releasing it.
  14. How to fix a stuck lock safely? Confirm nothing is actually running, then terraform force-unlock.
  15. Why pin provider and module versions? Prevents breaking changes from silently landing on a routine init/upgrade.

Interview Cheat Sheet

State = source of truth mapping config to reality. Remote backend + locking = team safety. count = index (fragile); for_each = key (stable). plan -out + apply tfplan = safe CI pattern. state mv / moved block = rename without destroy. prevent_destroy = hard stop on critical resources. Provisioners = last resort; prefer user_data. Policy-as-code = automated guardrails. Pin providers and modules, always.
