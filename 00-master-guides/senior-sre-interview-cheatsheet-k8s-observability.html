Kubernetes

architecture · troubleshooting · workloads · networking · release · gitops

K1 Cluster Architecture

SAYControl plane (API server, scheduler, controllers, etcd) holds desired state; worker nodes (kubelet, kube-proxy, runtime) run it. K8s constantly reconciles actual → desired.

K2 kubectl apply Flow

apply→API server→ authN→authZ (RBAC)→ admission→etcd→ controller→ReplicaSet→scheduler→ kubelet→runtime pulls+starts→Running
SAYThe request hits the API server, passes authentication, RBAC, and admission, is stored in etcd as desired state, then controllers, scheduler, and kubelet drive the pod to Running.

K6 Storage

StorageClass→PVC (claim)→PV (volume)→Pod

Access modes: RWO (one node), ROX (read-many), RWX (read-write-many). CSI = pluggable driver. Snapshots for point-in-time.

SAYA PVC requests storage; a StorageClass dynamically provisions a PV via a CSI driver; the pod mounts the PVC.

K7 Scheduling & Resources

  • requests = reserved (scheduling); limits = cap (throttle/OOM)
  • Taints/Tolerations repel; Affinity attracts; nodeSelector pins
  • QoS: Guaranteed > Burstable > BestEffort (evict order reversed)
SAYRequests drive scheduling and limits cap usage; CPU over-limit throttles, memory over-limit gets OOMKilled.

K10 Security

  • RBAC = who can do what (Role + RoleBinding)
  • ServiceAccount = pod identity
  • Secrets (encrypt at rest), Pod Security standards
  • OPA/Kyverno policy · Falco runtime threat detection
SAYLeast privilege via RBAC and ServiceAccounts, policy enforcement with Kyverno/OPA, and runtime detection with Falco.

K11 GitOps / ArgoCD

Git = single source of truth. Controller compares desired (Git) vs live (cluster) and reconciles.

  • Drift = live ≠ Git · selfHeal reverts manual changes
  • prune deletes orphans · ApplicationSet = many clusters from one template
SAYGitOps makes Git the source of truth; ArgoCD continuously reconciles the cluster to match Git, with self-heal and pruning to kill drift.

K12 Performance Traps

CPU throttleOOMKilled
Causeover CPU limitover mem limit

etcd latency slows the whole API. Small images = fast startup. JVM heap ≤ container mem (MaxRAMPercentage).

SAYCPU over-limit throttles silently; memory over-limit is OOMKilled and restarted — so I size limits from real usage, not guesses.

Monitoring & Observability

pillars · promql · slo · alerting · logging · tracing · incidents · dr

O1 ★ The Three Pillars — what each one answers

PillarAnswersExamplesTool
MetricsIs something wrong? (symptom)latency, error rate, CPUPrometheus
TracesWhere is it wrong? (location)request path across servicesOTel + Tempo/Jaeger
LogsWhy is it wrong? (evidence)stack trace, SQL timeoutLoki / OpenSearch
SAYMonitoring tells me something is wrong; observability tells me why. Metrics detect, traces locate, logs explain — I correlate all three with a shared trace/request ID.

O2 Observability Stack

SAYStandardize collection on OpenTelemetry, fan out to Prometheus/Loki/Tempo, visualize in Grafana, and alert through Alertmanager.

O3 Metric Types

TypeMeaning
Counteronly goes up (requests) → use rate()
Gaugeup/down (memory)
Histogrambuckets → P95/P99, aggregates
Summaryclient-side quantiles, hard to aggregate
SAYCounters for cumulative events with rate(), gauges for current state, histograms for latency percentiles because they aggregate across instances.

O4 ★ PromQL Essentials

# req/sec (Counter only!) rate(http_requests_total[5m]) # error rate % sum(rate(..._total{status=~"5.."}[5m])) / sum(rate(..._total[5m])) # P95 latency histogram_quantile(0.95, sum(rate(..._bucket[5m])) by(le)) # top 10 CPU pods topk(10, sum(rate(cpu_..._total[5m])) by(pod)) # forecast disk full predict_linear(fs_free[6h], 86400)

rate vs irate: rate = stable (alerts); irate = reactive (debug). Recording rules precompute heavy queries.

SAYrate() only on counters, histogram_quantile for percentiles, sum by() to aggregate replicas, recording rules for expensive queries.

O5 ★ SLI · SLO · SLA · Budget

  • SLI measures (actual 99.96%)
  • SLO targets (goal 99.9%)
  • SLA promises (contract + penalty)
  • Error budget = 100% − SLO · burn rate = how fast it's spent
SLODowntime / month
99%~7h 18m
99.9%~43m
99.95%~22m
99.99%~4m 23s
99.999%~26s
SAYSLI measures, SLO targets, SLA promises. Error budget = 1 − SLO; if it's exhausted we freeze risky releases and fix reliability.

O6 Golden Signals

L-T-E-S: Latency · Traffic · Errors · Saturation.

ModelUse
REDservices: Rate, Errors, Duration
USEresources: Utilization, Saturation, Errors
SAYLatency, traffic, errors, saturation on every dashboard; RED for request-driven services, USE for resources.

O9 Grafana

  • Does NOT store metrics — queries Prometheus/Loki/Tempo
  • Variables = one dashboard, all clusters
  • Annotations overlay deploys on graphs
  • Exec = business KPIs · Ops = technical metrics
SAYGrafana is the visualization layer — if it's down, Prometheus still scrapes and Alertmanager still pages; only dashboards are gone.

O10 Logging

stdout→Fluent Bit→parse/enrich→Loki/OpenSearch→Grafana
  • Loki indexes labels only → cheap; ELK full-text → costly
  • Fluent Bit = light DaemonSet; Fluentd = heavy aggregator
  • Correlation ID ties a request across services
  • Log JSON to stdout; never log secrets/PII
SAYStructured JSON to stdout, collected by Fluent Bit, enriched with k8s metadata, into Loki, correlated to traces by request ID.

O11 Distributed Tracing

  • Trace = whole request · Span = one operation
  • Trace ID shared across all services; Span ID per span
  • Context propagation via W3C traceparent header
  • Tail sampling keeps slow/failed traces (prod)
  • Tempo = cheap, Grafana-native · Jaeger = rich UI
SAYOne trace ID flows through every service via the traceparent header; I find the slowest span, then pivot to logs by that ID.

O12 ★ Incident Response

detect→classify SEV→IC + war room→mitigate→recover→RCA→postmortem
MetricMeaning
MTTDtime to detect
MTTAtime to acknowledge

SEV1 = full outage · IC coordinates, doesn't debug · update every 15–30 min · blameless postmortem.

SAYI scope customer impact, declare severity, assign an Incident Commander, mitigate first, then run a blameless RCA with tracked actions.

O13 HA & Disaster Recovery

HADR
Scopenode/podregion
Recoversecondsmin–hrs
Howmulti-AZmulti-region
  • RTO = max downtime · RPO = max data loss
  • Active-Active = both serve · Active-Passive = standby
  • Replicas ≠ backups; snapshots = backups. Test restores!
SAYHA handles local failures in seconds via multi-AZ; DR handles a region loss in minutes via multi-region. RTO is downtime, RPO is data loss.

⚡ Rapid-Fire Mnemonics — say these verbatim

SLI/SLO/SLA — "measures · targets · promises."
rate() — counters only; gauges go up and down.
P95/P99 — histograms, never averages (averages hide outliers).
Cluster health — Green all assigned · Yellow replica missing · Red primary missing.
Throttle vs OOM — CPU over-limit throttles; memory over-limit is killed.
HA vs DR — seconds/multi-AZ vs minutes/multi-region.
RTO vs RPO — Time to recover vs amount of data lost.
JVM/ES heap — keep under ~31 GB (compressed OOPs), ≤50% RAM.
Replicas ≠ backups — snapshots are backups; test the restore.
Grafana — doesn't store data; it queries backends.
Alert quality — page on symptoms (SLO), not CPU; group + inhibit to cut noise.
The nines — 99.9% ≈ 43 min/mo · 99.99% ≈ 4 min · 99.999% ≈ 26 sec.
GitOps — Git is truth; controller reconciles live → desired.
Incident first move — scope impact before touching anything.
Three pillars — metrics detect, traces locate, logs explain.
Golden signals — Latency, Traffic, Errors, Saturation.

The senior posture (use it on every scenario question)

1 scope impact→ 2 check dashboards/SLOs→ 3 gather evidence (metrics→traces→logs)→ 4 form & test hypothesis→ 5 mitigate / rollback→ 6 verify recovery→ 7 blameless RCA + prevent

Interviewers reward methodology over the first guess. Always say what you'd check and how you'd prevent recurrence — that's the difference between mid-level and senior.
