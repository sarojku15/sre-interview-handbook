Docker Mental Model
Docker is a standardised shipping container for software.

Before Docker: works on my machine! The app had specific library versions on the dev laptop. On the server: different versions, crashes.

After Docker: the application, ALL its dependencies, the runtime, and the config are packed into one image. That image runs identically on developer laptop, test server, and production.
Image vs Container vs Registry

Image

A read-only blueprint. Built from a Dockerfile. Stored in layers. Immutable once built.


Analogy: A class definition in code.


In production: ALWAYS use immutable tags like v1.2.3 or git SHA. Never use :latest

Container

A running instance of an image. Has its own filesystem, network, and process space. Shares the host OS kernel.


Analogy: An object instantiated from a class.


Containers are ephemeral. Never store important state inside a container.

Registry

A server that stores and distributes images. Docker Hub is public. ECR is AWS. ACR is Azure. Artifactory is enterprise.


Analogy: npm registry for container images.


In enterprise: private registry with vulnerability scanning before images reach production.

Production Dockerfile - Multi-Stage Build

Best Practices in One File

# Stage 1: Build - fat image with all build tools
FROM maven:3.9-openjdk-17 AS builder
WORKDIR /app
# Copy dependency manifest FIRST to preserve layer cache
COPY pom.xml .
RUN mvn dependency:go-offline -q
# Copy source only after dependencies are cached
COPY src ./src
RUN mvn package -DskipTests -q

# Stage 2: Runtime - minimal distroless image
FROM gcr.io/distroless/java17-debian11
WORKDIR /app
COPY --from=builder /app/target/myapp.jar .
# Run as non-root user - security requirement
USER 1001
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "myapp.jar"]

5 Dockerfile Rules to Remember

1. Multi-stage builds: build image is fat, runtime image is tiny
2. Copy dependency manifests before source code to preserve cache
3. Non-root user: USER 1001 before ENTRYPOINT
4. Use .dockerignore to exclude node_modules, .git, test files
5. Use distroless or Alpine base images: smaller means less attack surface

Kubernetes Mental Model
Kubernetes is an automated operations team for your containers.

Without K8s: you SSH into servers, manually start containers, they crash at 3am and nobody restarts them.

With K8s: you declare I want 5 copies of my app running at all times. Kubernetes makes that true and keeps it true automatically. You express DESIRED STATE. K8s does RECONCILIATION to reach it.
Core Objects

Workload Objects

Pod - smallest unit. One or more containers. Ephemeral. Never create Pods directly.


Deployment - manages stateless apps. Rolling updates and rollbacks. Use for web services and APIs.


StatefulSet - manages stateful apps. Stable pod names pod-0, pod-1. Ordered startup. Use for databases and Kafka.


DaemonSet - one pod per node. Use for Fluent Bit log collectors and Prometheus node-exporter.


CronJob - runs on schedule. Use for DB migrations and cleanup scripts.

Networking Objects

Service - stable virtual IP and DNS in front of dynamic pods.
ClusterIP = internal only default
NodePort = expose on node IP
LoadBalancer = cloud load balancer


Ingress - HTTP routing rules. One load balancer, many services. Route by path or hostname. Needs Ingress Controller like nginx or AWS ALB Controller.


NetworkPolicy - firewall rules for pods. Default: all pods can talk to all pods. Apply default-deny-all first, then whitelist required ports.

The Reconciliation Loop - Core K8s Concept

Every controller runs this loop forever:
1. Watch: what is the CURRENT state?
2. Compare: what is the DESIRED state from etcd?
3. Act: take actions to make current equal desired

Pod crashes: current=4, desired=5, controller immediately starts a new pod. This is why K8s is self-healing.

Deployment Strategies

Rolling Update - Default

Gradually replace old pods with new. Set maxUnavailable=0 and maxSurge=1 for zero downtime. Old and new run simultaneously briefly so the app must be backward compatible.

Blue-Green

Two identical environments. Switch traffic at load balancer level. Instant rollback by switching back. Double the infra cost during transition.

Canary

Send 5-10% traffic to new version. Monitor error rate and latency. Gradually increase if healthy. Roll back by removing canary pods. Tools: Argo Rollouts, Flagger, Istio.

Recreate

Kill ALL old pods then start ALL new pods. Causes downtime. Only use when old and new versions cannot run simultaneously. Never for HA production services.

Terraform Mental Model
Terraform is a blueprint system for cloud infrastructure.

Without Terraform: an engineer clicks through the AWS console for 4 hours. It cannot be reproduced exactly. When production and staging differ, nobody knows why.

With Terraform: the entire infrastructure is described in code files. Running terraform apply creates it identically every time. Version-controlled, peer-reviewed, and auditable.
The Four Core Actions

terraform init and plan

init - Downloads provider plugins and modules. Sets up the backend for state storage. Run once when starting a project.


plan - Compares desired state in .tf files against current state. Shows what WILL change. Plus create, tilde modify, minus destroy. ALWAYS review before applying. A plan showing unexpected destroys is a red flag.


terraform plan = git diff before you commit

terraform apply and destroy

apply - Executes the plan. Makes actual API calls. Updates the state file. In CI/CD: apply only after plan is reviewed and approved. Never run locally in prod.


destroy - Destroys ALL resources in the state file. Use with extreme caution. Add lifecycle prevent_destroy = true to protect critical resources like RDS.


terraform destroy = rm -rf on your infrastructure

RED Method Queries for SRE Dashboards

# Request Rate - RPS per service
sum(rate(http_requests_total[5m])) by (service)

# Error Rate - percent of 5xx responses
sum(rate(http_requests_total{status=~"5.."}[5m])) by (service)
/
sum(rate(http_requests_total[5m])) by (service)
* 100

# P99 Latency - using histogram
histogram_quantile(0.99,
  sum(rate(http_request_duration_seconds_bucket[5m])) by (le, service)
)

# Pod memory usage
container_memory_working_set_bytes{container!="", pod!=""}

# Pod CPU usage
sum(rate(container_cpu_usage_seconds_total{container!=""}[5m])) by (pod)

# SLO burn rate alert - fires when burning 14.4x faster than normal
(1 - sum(rate(http_requests_total{status!~"5.."}[1h]))
     / sum(rate(http_requests_total[1h])))
/ (1 - 0.999) > 14.4
SLO-Based Alerting vs Symptom-Based
Symptom-Based - Junior SRE

Alert when CPU above 80 percent.
Alert when memory above 75 percent.
Alert when disk above 70 percent.

Result: 50 alerts per day. Engineers ignore them. Alert fatigue. Critical alerts missed in the noise. CPU high does not always mean users are affected.

SLO Burn Rate - Senior SRE

Fast burn alert via PagerDuty: burning budget 14.4x faster than normal. Will exhaust in 1 hour. Wake someone up NOW.

Slow burn alert via Slack: burning 3x faster. Will exhaust in 5 days. Fix this today.

Result: 2 meaningful alerts instead of 50 noisy ones. Every alert equals user impact.

New Relic Mental Model
Prometheus and Grafana watches your servers. New Relic watches your USERS.

Prometheus tells you the server has high CPU. New Relic tells you a specific user clicked Pay Now, the request took 8 seconds, failed at the payment-gateway service on line 247, and 342 other users experienced the same issue in the last hour.

New Relic is Full Stack Observability: from the user browser through your API gateway, through your microservices, down to the database query - all in one trace.
Four Key Components

APM - Application Performance Monitoring

Auto-instruments your code. Just add the agent - Java, Python, Node. No code changes needed for basic setup.


Tracks:

  • Response time per transaction P95 and P99
  • Error rate with full stack traces
  • External service calls HTTP and DB queries
  • JVM metrics heap, GC, thread counts

Key Differentiator

New Relic APM shows you WHICH database query is slow and HOW MANY times it is called per transaction. Prometheus cannot do this without manual instrumentation.

Distributed Tracing

Follows a single user request across ALL microservices. Each service adds a trace span. You see the complete waterfall.


POST /api/pay       450ms total
  Auth service       12ms  OK
  Fraud check        38ms  OK
  DB: SELECT user   220ms  SLOW
  Payment gateway   165ms  OK
  Notification       15ms  OK

Without tracing: know the request took 450ms. With tracing: know the DB query caused most of the delay and can fix it.

Browser - Real User Monitoring

JavaScript snippet in your HTML measures what REAL users experience in their browsers.


  • Page load time full not server-only
  • Core Web Vitals LCP, FID, CLS
  • JavaScript errors in browser
  • AJAX call performance

Why this matters: Server says 200ms. RUM says 3 seconds because of slow JS. Two very different problems requiring different fixes.

Synthetic Monitoring

Runs scripted tests 24/7 from data centres worldwide even when no real users are active.


  • Ping: is the endpoint up?
  • API test: does login return 200?
  • Scripted browser: simulate checkout flow

SRE use case: At 3am no real users are on the system. Synthetic tests still run every minute. If the payment flow breaks, PagerDuty fires before the morning rush discovers it.

In Practice - Many Companies Use Both

Prometheus and Grafana: infrastructure metrics, Kubernetes health, SLO tracking, on-call dashboards.
New Relic: APM traces for application debugging, RUM for user experience, synthetics for proactive checks.

Interview answer: We used Prometheus and Grafana for infrastructure and SLO monitoring, and New Relic APM for application-level tracing and root cause analysis. Grafana shows the metric spike. New Relic shows which code caused it.

SRE Mental Model
SRE is what happens when a software engineer designs an operations team.

Traditional Ops: manual processes, tribal knowledge, resistance to change. If it ain't broke, don't touch it.

SRE: treat operations as a software engineering problem. Automate everything repeatable. Measure reliability mathematically with SLOs. Use postmortems to learn from failures without blame.
SLO, SLI, Error Budget

SLI - Service Level Indicator

A measurable signal of service health. A ratio of good events to total events.


Good SLIs:

  • Request success rate
  • Latency: percent of requests under 200ms
  • Availability: uptime minutes divided by total minutes

Bad SLIs: CPU usage, memory usage. These are infrastructure health, not user experience.

SLO - Service Level Objective

The TARGET value for an SLI. Agreed between engineering and the business.


Example SLOs:

  • 99.9% of payment requests succeed
  • 95% of requests complete under 200ms
  • 99.95% availability over 30 days

Rule: Set SLO slightly below what users actually need. This gives you headroom before user impact.

Error Budget

The allowable unreliability = 1 minus SLO


99.9% SLO
0.1% budget
43.8 minutes per month


Budget runs out: freeze feature releases, focus only on reliability


Budget healthy: ship features fast, experiment freely

Toil and How to Eliminate It

What is Toil?

Manual, repetitive operational work that scales with service load, has no lasting value, and could be automated.


Examples:

  • Manually restarting a service that crashes every day
  • Copying from a dashboard into a ticket
  • SSH-ing into servers to check disk space
  • Manually creating incident tickets from alerts

Google SRE rule: keep toil below 50% of work. Above that, engineers burn out.

Elimination Process

  • Track toil: log every manual task for 2 weeks
  • Prioritise: high frequency plus high time cost = automate first
  • Automate: Python scripts, Lambda, K8s operators
  • Measure: track time saved, report to leadership
  • Prevent: make toil impossible with auto-scaling and self-healing

Script 1 - Health Check and Auto-Restart

Monitor a service and restart if unresponsive

Deployed as: Cron job every 5 minutes
#!/bin/bash
# Health check and auto-restart script
# Cron: */5 * * * * /opt/scripts/health_check.sh

set -euo pipefail

SERVICE_NAME="payments-api"
HEALTH_URL="http://localhost:8080/health"
SLACK_WEBHOOK="${SLACK_WEBHOOK_URL}"
LOG_FILE="/var/log/health_check.log"

log() {
    echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1" | tee -a "$LOG_FILE"
}

alert_slack() {
    curl -s -X POST -H "Content-type: application/json" \
        --data "{\"text\":\":rotating_light: $1\"}" \
        "$SLACK_WEBHOOK" > /dev/null
}

check_health() {
    curl -s -o /dev/null -w "%{http_code}" --max-time 10 "$HEALTH_URL"
}

restart_service() {
    log "Restarting $SERVICE_NAME"
    systemctl restart "$SERVICE_NAME"
    sleep 10
    local status
    status=$(check_health)
    if [ "$status" -eq 200 ]; then
        log "Restart successful"
        alert_slack "$SERVICE_NAME restarted successfully on $(hostname)"
    else
        log "ERROR: Restart FAILED. HTTP: $status"
        alert_slack "CRITICAL: $SERVICE_NAME restart FAILED on $(hostname)!"
        exit 1
    fi
}

main() {
    local status
    status=$(check_health)
    if [ "$status" -eq 200 ]; then
        log "OK: $SERVICE_NAME healthy (HTTP $status)"
    else
        log "WARN: HTTP $status - restarting"
        alert_slack "WARNING: $SERVICE_NAME unhealthy on $(hostname). HTTP: $status"
        restart_service
    fi
}

main

Interview Talking Points

Uses curl to check HTTP status code. Sends Slack alert before and after restart. Logs everything with timestamps. set -euo pipefail ensures script fails fast on any error. In production we also push a custom metric to CloudWatch on every run so we can alert if the script itself stops running.

Script 2 - Disk Usage Monitor and Auto-Cleanup

Two-tier alerts and automatic log compression

Deployed as: Cron job every hour
#!/bin/bash
# Disk usage monitor and auto-cleanup
# Cron: 0 * * * * /opt/scripts/disk_monitor.sh

set -euo pipefail

THRESHOLD_WARN=80
THRESHOLD_CRIT=90
LOG_DIR="/var/log/myapp"
ARCHIVE_DIR="/var/log/myapp/archive"
RETENTION_DAYS=30
SLACK_WEBHOOK="${SLACK_WEBHOOK_URL}"

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"; }

alert() {
    local level="$1" msg="$2"
    local icon=":warning:"
    [ "$level" = "CRITICAL" ] && icon=":rotating_light:"
    curl -s -X POST -H "Content-type: application/json" \
        --data "{\"text\":\"$icon *$level* $msg\"}" \
        "$SLACK_WEBHOOK" > /dev/null
}

clean_old_logs() {
    log "Cleaning logs older than $RETENTION_DAYS days"
    find "$LOG_DIR" -name "*.log" -mtime +"$RETENTION_DAYS" | while read -r f; do
        gzip -c "$f" > "$ARCHIVE_DIR/$(basename $f).gz"
        > "$f"
        log "Compressed: $f"
    done
    find "$ARCHIVE_DIR" -name "*.gz" -mtime +90 -delete
}

df -H | grep -vE 'Filesystem|tmpfs|udev' | while read -r line; do
    usage=$(echo "$line" | awk '{print $5}' | tr -d '%')
    mount=$(echo "$line" | awk '{print $6}')

    if [ "$usage" -ge "$THRESHOLD_CRIT" ]; then
        log "CRITICAL: $mount at $usage%"
        alert "CRITICAL" "Disk $mount at $usage% on $(hostname). Auto-cleaning."
        clean_old_logs
    elif [ "$usage" -ge "$THRESHOLD_WARN" ]; then
        log "WARNING: $mount at $usage%"
        alert "WARNING" "Disk $mount at $usage% on $(hostname). Investigate."
    else
        log "OK: $mount at $usage%"
    fi
done
Script 3 - Kubernetes Pod Restart Monitor

Alert when pods are restarting frequently

Deployed as: Kubernetes CronJob every 15 minutes
#!/bin/bash
# Kubernetes pod restart monitor
# Run as K8s CronJob every 15 minutes

set -euo pipefail

NAMESPACE="${NAMESPACE:-production}"
RESTART_THRESHOLD=5
SLACK_WEBHOOK="${SLACK_WEBHOOK_URL}"

log() { echo "[$(date '+%Y-%m-%d %H:%M:%S')] $1"; }

alert_slack() {
    curl -s -X POST -H "Content-type: application/json" \
        --data "{\"text\":\"$1\"}" "$SLACK_WEBHOOK" > /dev/null
}

check_pod_restarts() {
    log "Checking pod restarts in namespace: $NAMESPACE"

    high_restart_pods=$(kubectl get pods -n "$NAMESPACE" --no-headers \
        -o custom-columns=NAME:.metadata.name,RESTARTS:.status.containerStatuses[0].restartCount \
        2>/dev/null | awk -v t="$RESTART_THRESHOLD" '$2 > t {print $1 " restarts:" $2}')

    if [ -n "$high_restart_pods" ]; then
        log "High restart pods: $high_restart_pods"

"""
API Health Monitor - auto-creates ServiceNow incidents on failure
Deploy as AWS Lambda triggered by EventBridge every 1 minute
"""
import requests, time, logging, os, json
from dataclasses import dataclass
from typing import Optional

logging.basicConfig(level=logging.INFO,
    format="%(asctime)s %(levelname)s %(message)s")
logger = logging.getLogger(__name__)

ENDPOINTS = [
    {"name": "payments-api",  "url": "https://api.example.com/health",   "timeout_ms": 2000},
    {"name": "auth-service",  "url": "https://auth.example.com/health",  "timeout_ms": 1000},
    {"name": "fraud-service", "url": "https://fraud.example.com/health", "timeout_ms": 3000},
]

@dataclass
class HealthResult:
    name:       str
    url:        str
    status:     Optional[int]
    latency_ms: Optional[float]
    healthy:    bool
    error:      Optional[str] = None


def check_endpoint(name, url, timeout_ms) -> HealthResult:
    try:
        start = time.time()
        resp = requests.get(url, timeout=timeout_ms / 1000)
        latency_ms = (time.time() - start) * 1000
        return HealthResult(
            name=name, url=url,
            status=resp.status_code,
            latency_ms=round(latency_ms, 2),
            healthy=resp.status_code == 200 and latency_ms < timeout_ms
        )
    except requests.exceptions.RequestException as exc:
        return HealthResult(name=name, url=url,
            status=None, latency_ms=None,
            healthy=False, error=str(exc))


def create_snow_incident(result: HealthResult) -> str:
    snow_url = os.environ["SNOW_INSTANCE"] + "/api/now/table/incident"
    payload = {
        "short_description": f"Health check FAILED: {result.name}",
        "description": (
            f"URL: {result.url}\n"
            f"Status: {result.status}\n"
            f"Latency: {result.latency_ms}ms\n"
            f"Error: {result.error}"
        ),
        "urgency": "1", "impact": "1",
        "assignment_group": "SRE-Platform"
    }
    resp = requests.post(snow_url, json=payload,
        auth=(os.environ["SNOW_USER"], os.environ["SNOW_PASS"]),
        headers={"Content-Type": "application/json"}, timeout=30)
    resp.raise_for_status()
    number = resp.json()["result"]["number"]
    logger.info("Created incident %s for %s", number, result.name)
    return number


def lambda_handler(event, context):
    failed = []
    for ep in ENDPOINTS:
        r = check_endpoint(ep["name"], ep["url"], ep["timeout_ms"])
        if r.healthy:
            logger.info("OK  | %s | %sms", r.name, r.latency_ms)
        else:
            logger.error("FAIL| %s | %s", r.name, r.error or r.status)
            failed.append(r)
            create_snow_incident(r)
    return {"statusCode": 200,
            "body": json.dumps({"checked": len(ENDPOINTS), "failed": len(failed)})}

"""
AWS Resource Tagging Compliance Checker
Finds EC2 and RDS instances missing required tags
Run as Lambda daily or as part of CI/CD compliance gate
"""
import boto3, json, logging
from datetime import datetime
from typing import List, Dict

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

REQUIRED_TAGS = ["Environment", "Owner", "Cost-Center", "Data-Classification"]
REGION = "ap-south-1"


def get_missing_tags(tags: list) -> List[str]:
    existing = {t["Key"] for t in (tags or [])}
    return [tag for tag in REQUIRED_TAGS if tag not in existing]


def check_ec2_instances() -> List[Dict]:
    ec2 = boto3.client("ec2", region_name=REGION)
    violations = []
    paginator = ec2.get_paginator("describe_instances")
    for page in paginator.paginate():
        for reservation in page["Reservations"]:
            for inst in reservation["Instances"]:
                if inst["State"]["Name"] == "terminated":
                    continue
                missing = get_missing_tags(inst.get("Tags", []))
                if missing:
                    violations.append({
                        "resource_type": "EC2",
                        "resource_id":   inst["InstanceId"],
                        "instance_type": inst["InstanceType"],
                        "state":         inst["State"]["Name"],
                        "missing_tags":  missing
                    })
    return violations


def check_rds_instances() -> List[Dict]:
    rds = boto3.client("rds", region_name=REGION)
    violations = []
    paginator = rds.get_paginator("describe_db_instances")
    for page in paginator.paginate():
        for db in page["DBInstances"]:
            tags = rds.list_tags_for_resource(
                ResourceName=db["DBInstanceArn"])["TagList"]
            missing = get_missing_tags(tags)
            if missing:
                violations.append({
                    "resource_type": "RDS",
                    "resource_id":   db["DBInstanceIdentifier"],
                    "engine":        db["Engine"],
                    "missing_tags":  missing
                })
    return violations


def lambda_handler(event, context):
    all_v = check_ec2_instances() + check_rds_instances()
    logger.info("Total violations: %s", len(all_v))

    s3 = boto3.client("s3", region_name=REGION)
    key = f"tagging/{datetime.now().strftime('%Y/%m/%d')}/report.json"
    s3.put_object(Bucket="company-compliance-reports", Key=key,
        Body=json.dumps({"date": datetime.now().isoformat(),
                         "total": len(all_v), "violations": all_v}, indent=2),
        ContentType="application/json")
    return {"violations": len(all_v), "report_key": key}

Script 3 - Kubernetes Deployment Health Checker

Post-deployment validation that triggers pipeline rollback on failure

Deployed as: Jenkins or Harness post-deploy step
"""
Kubernetes Deployment Health Checker
Post-deployment validation - exits code 1 to trigger pipeline rollback
Called from Jenkins or Harness as post-deploy step
"""
import subprocess, json, time, sys, logging, argparse

logging.basicConfig(level=logging.INFO,
    format="%(asctime)s %(levelname)s %(message)s")
logger = logging.getLogger(__name__)


def kubectl(args: list) -> dict:
    cmd = ["kubectl"] + args + ["-o", "json"]
    r = subprocess.run(cmd, capture_output=True, text=True, timeout=30)
    if r.returncode != 0:
        raise RuntimeError(f"kubectl failed: {r.stderr}")
    return json.loads(r.stdout)


def wait_for_ready(deployment: str, namespace: str,
                   expected: int, timeout: int = 300) -> bool:
    deadline = time.time() + timeout
    logger.info("Waiting for %s/%s to have %s ready replicas",
                namespace, deployment, expected)
    while time.time() < deadline:
        try:
            data  = kubectl(["get", "deployment", deployment, "-n", namespace])
            ready = data["status"].get("readyReplicas", 0)
            total = data["status"].get("replicas", 0)
            logger.info("Ready: %s/%s", ready, total)
            if ready >= expected:
                logger.info("Deployment %s is healthy", deployment)
                return True
        except Exception as exc:
            logger.warning("Check error: %s", exc)
        time.sleep(10)
    logger.error("Timed out after %ss", timeout)
    return False


def check_no_crash_loop(namespace: str) -> bool:
    r = subprocess.run(
        ["kubectl", "get", "pods", "-n", namespace, "--no-headers"],
        capture_output=True, text=True, timeout=30)
    crash_pods = [l.split()[0] for l in r.stdout.splitlines()
                  if "CrashLoopBackOff" in l or "Error" in l]
    if crash_pods:
        logger.error("CrashLoopBackOff pods: %s", crash_pods)
        for pod in crash_pods:
            subprocess.run(["kubectl", "logs", pod, "-n", namespace,
                            "--previous", "--tail=20"], timeout=30)
        return False
    return True


def main():
    p = argparse.ArgumentParser()
    p.add_argument("--deployment", required=True)
    p.add_argument("--namespace",  required=True)
    p.add_argument("--replicas",   type=int, default=3)
    p.add_argument("--timeout",    type=int, default=300)
    args = p.parse_args()

    ok = (wait_for_ready(args.deployment, args.namespace,
                         args.replicas, args.timeout)
          and check_no_crash_loop(args.namespace))

    if ok:
        logger.info("All health checks PASSED. Deployment is healthy.")
        sys.exit(0)
    else:
        logger.error("Health checks FAILED. Pipeline will trigger rollback.")
        sys.exit(1)


if __name__ == "__main__":
    main()

Called from the Jenkins or Harness pipeline as a post-deployment step. If it exits with code 1 the pipeline triggers a rollback automatically. Checks both replica readiness and CrashLoopBackOff. Captures previous container logs automatically for fast debugging. We also set a hard timeout so it never blocks the pipeline indefinitely if something unexpected happens.

"""
SLO Error Budget Tracker
Queries Prometheus, calculates budget consumed per service, posts Slack report
Run as K8s CronJob daily at 8am
"""
import requests, os, json, logging
from datetime import datetime

logging.basicConfig(level=logging.INFO)
logger = logging.getLogger(__name__)

PROMETHEUS_URL = os.environ.get("PROMETHEUS_URL", "http://prometheus:9090")
SLACK_WEBHOOK  = os.environ["SLACK_WEBHOOK_URL"]

SERVICES = [
    {"name": "payments-service", "slo": 0.999},
    {"name": "auth-service",     "slo": 0.9995},
    {"name": "fraud-service",    "slo": 0.998},
]


def promql(query: str) -> float:
    resp = requests.get(f"{PROMETHEUS_URL}/api/v1/query",
                        params={"query": query}, timeout=30)
    resp.raise_for_status()
    results = resp.json()["data"]["result"]
    return float(results[0]["value"][1]) if results else 0.0


def calc_budget(name: str, slo: float) -> dict:
    total   = promql(f'increase(http_requests_total{{service="{name}"}}[30d])')
    success = promql(
        f'increase(http_requests_total{{service="{name}",status!~"5.."}}[30d])')
    if total == 0:
        return {"name": name, "error": "No data"}

    actual_reliability  = success / total
    budget_total        = 1 - slo
    errors_allowed      = total * budget_total
    errors_actual       = total - success
    consumed_pct        = round(errors_actual / errors_allowed * 100, 1) if errors_allowed else 0
    remaining_pct       = round(100 - consumed_pct, 1)
    status = ("CRITICAL" if remaining_pct < 10
              else "WARNING" if remaining_pct < 25
              else "OK")

    return {
        "name":          name,
        "slo":           f"{slo * 100}%",
        "actual":        f"{actual_reliability * 100:.4f}%",
        "total_requests": int(total),
        "total_errors":   int(errors_actual),
        "consumed_pct":   consumed_pct,
        "remaining_pct":  remaining_pct,
        "status":         status
    }


def send_slack_report(budgets: list):
    lines = [f"*SLO Error Budget Report - {datetime.now().strftime('%Y-%m-%d')}*\n"]
    for b in budgets:
        if "error" in b:
            lines.append(f"  {b['name']}: {b['error']}")
            continue
        icon = (":red_circle:"    if b["status"] == "CRITICAL"
                else ":yellow_circle:" if b["status"] == "WARNING"
                else ":green_circle:")
        lines.append(
            f"{icon} *{b['name']}*\n"
            f"    SLO: {b['slo']} | Actual: {b['actual']}\n"
            f"    Budget consumed: {b['consumed_pct']}%"
            f" | Remaining: {b['remaining_pct']}%"
        )
    requests.post(SLACK_WEBHOOK, json={"text": "\n".join(lines)}, timeout=10)
    logger.info("Slack report sent")


def main():
    budgets = [calc_budget(s["name"], s["slo"]) for s in SERVICES]
    for b in budgets:
        logger.info("%s", json.dumps(b))
    send_slack_report(budgets)


if __name__ == "__main__":
    main()

Key Principles to Explain in Interview

1. Multi-stage: build stage is fat with compilers and SDKs. Runtime stage is minimal - only what is needed to RUN the app.
2. Layer caching: copy dependency manifests BEFORE source code. The dependency download layer is only invalidated when pom.xml or package.json changes. Source changes do not re-download dependencies.
3. distroless: no shell, no package manager, no utilities. Dramatically reduces attack surface. Security scanners find far fewer CVEs.
4. Non-root user: USER 65532 or USER 1001. Many security policies reject root containers in production.
5. HEALTHCHECK: tells Docker and K8s how to determine if the container is healthy. Without it, a container that is running but not serving traffic looks healthy.

Key Points to Explain in Interview

Topology spread constraints: ensures pods are distributed across AZs. Without this, all 3 replicas may land in AZ-a and fail together when AZ-a goes down.
preStop sleep 5: gives the load balancer time to stop routing traffic to the pod before the process exits. Without it, you get brief 502 errors during rolling deployments.
readOnlyRootFilesystem: security hardening. Forces you to mount emptyDir for anything the app needs to write. Prevents malware from persisting on the filesystem.
startupProbe vs livenessProbe: startup probe gives the app time to initialise (slow startup apps). Liveness probe then takes over. Without startup probe, liveness probe kills slow-starting apps before they are ready.
PDB minAvailable=2: ensures at least 2 pods are always serving traffic during voluntary disruptions like node drain during K8s upgrades.

Key Points to Explain in Interview

one_nat_gateway_per_az: one NAT Gateway per AZ not a single shared NAT. Single NAT is a single point of failure. If that AZ fails, all private subnet traffic loses internet access.
enable_irsa: creates an OIDC provider for the EKS cluster. Required for IRSA - the mechanism that lets pods assume AWS IAM roles without access keys.
deletion_protection and prevent_destroy: two layers of protection on the RDS instance. deletion_protection is enforced by AWS API. prevent_destroy is enforced by Terraform. Both together means accidental destruction is very hard.
performance_insights_enabled: free for 7 days retention. Shows you slow queries, wait events, top SQL. Essential for diagnosing database performance issues in production.
capacity_type SPOT: spot instances for the application node group saves 60-70% on EC2 costs. For a stateless K8s workload this is safe because pods are rescheduled when spot instances are reclaimed.

                --values ./helm/${SERVICE_NAME}/values-prod.yaml \
                --wait --timeout 10m

              # Remove canary after full rollout
              helm uninstall ${SERVICE_NAME}-canary -n ${NAMESPACE_PROD} || true
            """
          }
        }
      }
      post {
        success {
          slackSend channel: env.SLACK_CHANNEL,
            color: "good",
            message: "PROD DEPLOYED: ${SERVICE_NAME} ${IMAGE_TAG} by ${env.DEPLOY_APPROVER}"
        }
        failure {
          slackSend channel: env.SLACK_CHANNEL,
            color: "danger",
            message: "PROD DEPLOY FAILED: ${SERVICE_NAME} ${IMAGE_TAG}. Rolling back."
          container("kubectl") {
            withKubeConfig([credentialsId: "kubeconfig-prod"]) {
              sh "helm rollback ${SERVICE_NAME} 0 -n ${NAMESPACE_PROD} || true"
            }
          }
        }
      }
    }
  }

  post {
    always {
      cleanWs()
    }
    failure {
      slackSend channel: env.SLACK_CHANNEL,
        color: "danger",
        message: "Build FAILED: ${SERVICE_NAME} #${env.BUILD_NUMBER} ${env.BUILD_URL}"
    }
  }
}

Key Points to Explain in Interview

Kubernetes agents: each build gets a fresh pod with exactly the tools it needs. No state carries over between builds. Auto-scales with the cluster. No idle agents waiting for work.
Parallel stages: Test, SonarQube, and OWASP scan run simultaneously. In a sequential pipeline this would take 20 minutes. In parallel: 7 minutes. This is the single biggest pipeline optimisation.
IMAGE_TAG = GIT_COMMIT: using the git commit SHA as the image tag creates an immutable reference. You can always trace any running container back to the exact code commit and build that created it.
Canary deploy then check: deploy to 20% of traffic, wait 5 minutes, query Prometheus for error rate. If above 1% the pipeline fails and rollback is triggered automatically. Human approval is required for prod but the CANARY decision is automated.
post always cleanWs(): always clean the workspace after the build. Without this, old build artifacts accumulate and fill the node disk. A full disk silently breaks future builds.

Jenkins is running fine but suddenly builds are queuing and agents are not picking up jobs. How do you debug this?

▼
This is a real production problem with Jenkins at scale. I follow this sequence:

Step 1 - Check agent status: Go to Manage Jenkins > Nodes or run: curl -s http://jenkins/computer/api/json | jq '.computer[].displayName + " " + .computer[].offline'. Are agents showing as offline? Disconnected? Busy?

Step 2 - Check agent logs: For Kubernetes agents, kubectl get pods -n jenkins and kubectl logs AGENT-POD. Common causes: pod evicted due to node memory pressure, JNLP connection refused, PVC full.

Step 3 - Check master disk: Jenkins master needs disk for workspace, artifacts, logs. df -h on the Jenkins master node. A full disk silently prevents agents from connecting.

Step 4 - Check JNLP port: Kubernetes agents connect to the Jenkins master on port 50000 (JNLP). Check that the Jenkins service exposes this port and firewall rules allow it.

Step 5 - Check resource quotas: In K8s, if the jenkins namespace has a ResourceQuota that is full, new agent pods cannot be scheduled. kubectl describe quota -n jenkins.

Root fix: In production I set up Prometheus monitoring on Jenkins itself - number of queued builds, agent count, build duration. Alert when queue depth exceeds 10 for more than 5 minutes. Catch it before engineers notice.

A Jenkins pipeline is passing locally but failing in CI with 'permission denied' errors on Docker. What is happening and how do you fix it?

▼
Classic Docker-in-Docker permission problem. Multiple possible causes:

Cause 1 - Socket permission: The Jenkins agent container is trying to use the host Docker socket at /var/run/docker.sock. If the socket is owned by root and the container runs as a non-root user, you get permission denied.

Fix: Add the jenkins user to the docker group in the agent image: usermod -aG docker jenkins. Or mount the socket with the right permissions. Or use DinD (Docker-in-Docker) with privileged mode.

Cause 2 - DinD container not ready: If using a separate DinD sidecar, the build container may start running Docker commands before the DinD daemon is ready. Fix: Add a wait-for-docker step: until docker ps; do sleep 1; done.

Cause 3 - Security context too restrictive: The K8s pod security policy or OPA admission controller may block privileged containers needed for DinD. Fix: Create a specific PodSecurityPolicy that allows privileged mode only for the docker container in jenkins namespace.

Production recommendation: Avoid Docker-in-Docker entirely. Use Kaniko for building images inside Kubernetes - it builds Docker images without needing Docker daemon or privileged access. Much more secure for regulated environments.

How do you manage secrets in Jenkins pipelines across 50 microservices without repeating credentials everywhere?

▼
At scale, you need a centralised secret strategy. Here is what I implement:

1. Vault integration as the source of truth: HashiCorp Vault stores all secrets. Jenkins has a Vault plugin. Pipelines authenticate to Vault using the Jenkins service account token (Kubernetes auth method). Vault returns short-lived dynamic secrets.

2. Shared Library for Vault access: Write a Jenkins Shared Library function once that handles Vault authentication and secret retrieval. All 50 pipelines call this function. One place to update when Vault path changes.

// vars/getSecret.groovy in shared library
def call(String secretPath, String key) {
    def vaultSecrets = [
        [path: secretPath, secretValues: [
            [envVar: 'SECRET_VALUE', vaultKey: key]
        ]]
    ]
    withVault([vaultSecrets: vaultSecrets]) {
        return env.SECRET_VALUE
    }
}
3. Never print secrets: Jenkins masks credentials automatically if you use the credentials() binding. Never echo or print secret values. Add set +x before commands that use secrets in shell scripts.

4. Audit trail: Vault logs every secret access. You can see which pipeline accessed which secret and when. Critical for SOX compliance.

5. Secret rotation: Dynamic secrets from Vault rotate automatically. Jenkins gets a fresh token per build. Even if a build log is compromised, the token is already expired.

Your Docker image builds successfully locally but when pushed to production Kubernetes, pods fail with ImagePullBackOff. How do you debug this?

▼
ImagePullBackOff means K8s cannot pull the image. Follow this sequence:

Step 1 - Get the exact error: kubectl describe pod POD-NAME and look at Events. The error message tells you exactly what failed: auth error, network error, image not found, etc.

Step 2 - Verify the image exists: aws ecr describe-images --repository-name payments-service --image-ids imageTag=1a2b3c4d. Is the tag you are deploying actually in the registry?

Step 3 - Check pull secret: K8s needs credentials to pull from a private registry. Verify the imagePullSecret exists: kubectl get secret ecr-pull-secret -n production. ECR tokens expire every 12 hours - you need a cron job to refresh them. Or better, use the ECR credential helper or IRSA with the EKS IAM permissions.

Step 4 - Check node network: The node must be able to reach the registry endpoint. kubectl debug node/NODE-NAME -it --image=alpine and try wget https://123456.dkr.ecr.ap-south-1.amazonaws.com.

Step 5 - Check imagePullPolicy: If set to Never, K8s will never pull and will fail if the image is not locally cached. If set to Always and network is slow, it may timeout.

Real cause I have seen in production: The CI/CD pipeline pushed to ECR in us-east-1 but the EKS cluster is in ap-south-1. Different registry URL. The image literally did not exist in the right region.

How do you reduce a Docker image from 2GB to under 200MB? Walk through your approach.

▼
Image size reduction is a multi-step process I follow in order from most to least impactful:

Step 1 - Multi-stage builds (biggest win): The build stage needs compilers, package managers, SDKs. The runtime stage needs NONE of that. A Java app only needs the JRE, not the JDK. A Go binary is statically compiled and needs nothing except possibly libc.

Step 2 - Use a minimal base image:
- ubuntu: 77MB (not bad but has apt, bash, coreutils you do not need)
- alpine: 7MB (much smaller, uses musl libc - watch for compatibility issues)
- distroless: 20MB (no shell, no package manager, minimal attack surface - BEST for production)
- scratch: 0MB (for Go statically compiled binaries)

Step 3 - Combine RUN commands: Each RUN creates a layer. Multiple RUNs = multiple layers = wasted space even if you delete files in a later layer.
Bad: RUN apt-get update then RUN apt-get install curl then RUN rm -rf /var/lib/apt/lists
Good: RUN apt-get update && apt-get install -y curl && rm -rf /var/lib/apt/lists/*

Step 4 - Use .dockerignore: Without it, every file in the build context is sent to the Docker daemon. node_modules alone can be 500MB. Add .dockerignore first.

Step 5 - Remove build artifacts in the same layer: RUN mvn package && rm -rf ~/.m2

Step 6 - Use dive to inspect layers: dive IMAGE shows you exactly what each layer contains and what is wasting space.

Real example: Java Spring Boot app went from 750MB (openjdk:17) to 85MB (distroless/java17:nonroot) using multi-stage build.

A container is consuming 100% CPU on the host and other containers are suffering. How do you identify it and what do you do immediately?

▼
Production emergency response for container CPU runaway:

Step 1 - Identify the culprit immediately:
docker stats --no-stream shows CPU% per container. Or on the host: top -c and look for the container process name, or ps aux --sort=-%cpu | head -5.

Step 2 - In Kubernetes, identify the pod:
kubectl top pods -A --sort-by=cpu | head -10. Or in Grafana, check the CPU per pod panel.

Step 3 - Immediate mitigation:
If K8s: The pod should already be CPU-throttled by the resource limit. If it has no limit (never acceptable in production), patch it immediately:
kubectl patch deployment myapp --patch='{"spec":{"template":{"spec":{"containers":[{"name":"myapp","resources":{"limits":{"cpu":"1000m"}}}]}}}}'

If this is killing other pods, cordon the node to prevent new pods being scheduled there:
kubectl cordon NODE-NAME
Then drain and investigate: kubectl drain NODE-NAME --ignore-daemonsets

Step 4 - Root cause investigation:
- Get a CPU profile: for Java use jstack or async-profiler. For Python use py-spy. For Go use pprof.
- Check application logs for any infinite loop indicators (same log line repeated millions of times).
- Check if a recent deploy introduced the runaway code.

Long-term fix: EVERY container must have CPU limits in production. This is non-negotiable. Use OPA to reject deployments without resource limits.

You deploy a new version and the rollout gets stuck with some pods in Running and some in Pending. How do you diagnose and fix this without causing downtime?

Stuck rollout with mixed Running and Pending pods is a common production scenario. Methodical approach:

Step 1 - Check rollout status:

Step 2 - Describe the pending pods:
kubectl describe pod PENDING-POD -n production

- "0/3 nodes have sufficient memory" → resource request too high or nodes full
- "didn't match pod affinity" → topology spread constraint cannot be satisfied
- "PVC not bound" → storage issue
- "node had taints that the pod didn't tolerate" → wrong node selector

kubectl rollout undo deployment/payments-service -n production

If resource issue: reduce memory request in the new version. If affinity issue: check if you added a new node selector or topology constraint that cannot be met. Fix the Helm values and re-deploy.

How do you implement zero-downtime database migrations in a Kubernetes deployment?

▼
DB migrations during K8s rolling deployments is one of the most dangerous operations. The wrong approach causes data loss or application crashes. Here is the safe pattern I use:

The core problem: During a rolling deploy, old pods (using old schema) and new pods (expecting new schema) run SIMULTANEOUSLY. If the migration drops a column or renames it, old pods crash. If the migration is not backward compatible, you have a production incident.

The Expand-Contract (3-phase) pattern:

Phase 1 - Expand (backward compatible migration): Deploy a migration that only ADDS things. Add the new column with a default value. Add the new table. Add the new index. Old code continues to work. New code uses the new column.
Run migration as a K8s Job or Init Container that runs before pods start.

Phase 2 - Deploy new application version: New pods use the new column. Old pods that may still be running also work because the old column still exists.

Phase 3 - Contract (cleanup migration): After ALL old pods are gone (verified by checking deployment rollout complete), run a second migration to remove the old column. Deploy this as a separate Job after verifying the new version is stable.

Implementation: I use Flyway or Liquibase for migration management. The migration Job runs in a K8s Job with backoffLimit=3. If it fails, the deployment pipeline fails and no new pods are started.

Never do this: Never run DROP COLUMN or RENAME COLUMN as part of the same deployment that introduces new code. That is how you get 3am incidents.

Your cluster has 100 nodes and you need to upgrade Kubernetes from 1.27 to 1.29. How do you do it with zero downtime?

▼
K8s cluster upgrade is a high-risk operation. Zero-downtime requires planning and a specific sequence:

Pre-upgrade checklist:
- Check API deprecations: kubectl deprecations-check or use Pluto tool. Any deprecated APIs in your manifests will break after upgrade.
- Verify add-ons compatibility: CoreDNS, kube-proxy, cluster-autoscaler, metrics-server all have K8s version requirements.
- Test in staging first: always upgrade staging 1 week before production.
- Snapshot etcd and all PVs as a last resort backup.

Upgrade sequence (EKS managed example):

Step 1 - Upgrade the Control Plane first:
AWS EKS: aws eks update-cluster-version --name my-cluster --kubernetes-version 1.28
EKS upgrades the control plane with no downtime. API server may be briefly unavailable during the upgrade (about 1-2 minutes). Existing pods keep running - kubelet manages them independently.

Step 2 - Upgrade add-ons: After control plane, upgrade CoreDNS, kube-proxy, VPC CNI to compatible versions.

Step 3 - Upgrade node groups one at a time:
For each node group, update the launch template AMI to the new K8s version, then do a rolling replacement:
aws eks update-nodegroup-version --cluster-name my-cluster --nodegroup-name application
EKS cordon and drains old nodes, starts new nodes. PodDisruptionBudgets are respected during drain.

Step 4 - Verify after each node group:
kubectl get nodes - all nodes should show new version.
Run smoke tests against your services.
Check that all pods are Running and not Restarting.

Total time for 100 nodes: Roughly 2-4 hours for a rolling upgrade if you upgrade node groups in batches of 20.

Explain the Kubernetes architecture for a senior SRE interview.

Your pod is in CrashLoopBackOff. Walk me through how you debug it.

▼
CrashLoopBackOff means the container is starting and crashing repeatedly.
Kubernetes backs off exponentially before each restart.

Step 1 - Get the exit code
kubectl describe pod POD-NAME -n NAMESPACE
Look for 'Last State' -> 'Exit Code':
Exit code 1 = application error (check logs)
Exit code 137 = OOM kill (increase memory limit)
Exit code 127 = binary not found (wrong entrypoint or image)
Exit code 1/255 = configuration or permission error

Step 2 - Get logs from the previous (crashed) container
kubectl logs POD-NAME --previous -n NAMESPACE
This is critical: the current container may have no logs yet.

Step 3 - If logs are too short (container crashes in milliseconds)
Override the entrypoint to keep it alive:
kubectl run debug --image=myapp:v1.2.3 --command -- sleep 3600
kubectl exec -it debug -- /bin/sh
Manually run the application entrypoint and watch what fails.

Step 4 - Check events
kubectl get events --sort-by=.lastTimestamp -n NAMESPACE
Look for OOMKilled, ImagePullBackOff, or liveness probe failures.

Common root causes:
- Missing required environment variable (DB_PASSWORD, API_KEY)
- Database connection refused on startup (add init container with wait)
- OOM kill: memory limit too low for the application
- Liveness probe too aggressive: killing the pod before it finishes startup
- Wrong command or entrypoint in the Docker image

How does Kubernetes networking work and how do you troubleshoot connectivity issues?

▼
Kubernetes networking model:
Every pod gets its own IP address.
All pods can communicate with all other pods without NAT.
Services provide stable DNS and VIP in front of a set of pods.
kube-proxy manages iptables rules that route Service traffic to pod IPs.

DNS resolution in the cluster:
Service DNS format: service-name.namespace.svc.cluster.local
CoreDNS resolves this to the Service ClusterIP.

Networking troubleshooting steps:

Step 1 - Can the pod resolve DNS?
kubectl exec -it POD -- nslookup kubernetes.default
If this fails: CoreDNS is broken. Check: kubectl get pods -n kube-system

Step 2 - Can the pod reach the target service IP?
kubectl exec -it POD -- curl http://SERVICE-NAME.NAMESPACE:PORT/health

Step 3 - Can the pod reach the target pod IP directly?
Get pod IP: kubectl get pod TARGET-POD -o wide
kubectl exec -it POD -- curl http://POD-IP:PORT/health
If direct works but service does not: check Service selector labels.

Step 4 - Check Network Policy
kubectl get networkpolicy -n NAMESPACE
A default-deny policy blocks all traffic unless explicitly allowed.
Check if the calling pod's labels are in the ingress rules.

Step 5 - Check Security Groups (EKS)
EKS with VPC CNI uses pod-level Security Groups.
Verify the pod SG allows inbound from the calling pod SG on the target port.

How do you perform a zero-downtime rolling deployment in Kubernetes?

▼
Kubernetes rolling update is the default deployment strategy.
To guarantee zero downtime, these settings must be correct:

Deployment spec:
strategy:
type: RollingUpdate
rollingUpdate:
maxUnavailable: 0 # never kill old pod before new one is Ready
maxSurge: 1 # allow one extra pod during the rollout

Readiness probe (mandatory):
readinessProbe:
httpGet:
path: /health
port: 8080
initialDelaySeconds: 10
periodSeconds: 5
failureThreshold: 3
Kubernetes will not route traffic to the new pod until readiness passes.
Without this, traffic hits the pod before the app is ready -> errors.

PreStop hook (for graceful shutdown):
lifecycle:
preStop:
exec:
command: ['sleep', '5']
Gives the load balancer time to stop routing traffic before the pod stops.

PDB (Pod Disruption Budget):
Ensures minimum pods are always available during voluntary disruptions:
spec:
minAvailable: 2

Verification:
kubectl rollout status deployment/myapp
kubectl rollout history deployment/myapp

How does Kubernetes HPA work and when would you use KEDA instead?

▼
HPA (Horizontal Pod Autoscaler):
Scales pod count based on observed metrics vs target.
Default metric: CPU utilization percentage.
Custom metrics: memory, HTTP RPS, queue depth (via Prometheus Adapter).

How it works:
HPA controller polls Metrics Server every 15 seconds.
desired replicas = ceil(current replicas * (current metric / target metric))

Example: target CPU 70%, current CPU 140% on 3 pods
desired = ceil(3 * (140 / 70)) = ceil(6) = 6 pods

kubectl autoscale deployment api --cpu-percent=70 --min=3 --max=20

HPA limitations:
- Cannot scale to zero (minimum is 1 pod)
- Only scales on CPU, memory, or custom metrics that are always present
- Latency in scale-up: metrics poll + pod start time = 1-3 minutes

KEDA (Kubernetes Event Driven Autoscaling):
Scales based on external event sources:
- Kafka consumer lag
- SQS queue depth
- RabbitMQ queue length
- Azure Service Bus message count
- Cron schedule
- Can scale to ZERO when there are no events

When to use KEDA:
- Batch processing jobs that should scale to zero when idle
- Event-driven microservices consuming from a queue
- Significant cost saving: scale to zero overnight

In production (example):
Payments notification service scales from 0 to 50 pods
based on SQS queue depth. Costs near zero when queue is empty.

You need to refactor a large Terraform monolith into separate state files without destroying any resources. How do you do this safely?

▼
Splitting a Terraform monolith is one of the most dangerous Terraform operations. The safe approach:

The danger: If you move a resource to a new state file by deleting it from the old config and adding it to the new one, Terraform will DESTROY the resource in the old state and CREATE a new one. For a production database, this means data loss.

Safe approach - terraform state mv:

Step 1 - Plan the split: Decide which resources move to which new state file. Group by lifecycle (infrastructure vs application) or by team (payments vs auth vs notifications).

Step 2 - For each resource being moved:
# Remove from old state without destroying it terraform state rm 'aws_rds_cluster.payments' # Pull the state from old state file to a JSON file terraform state pull > old_state.json # In the new config directory, import the resource terraform import aws_rds_cluster.payments payments-prod-cluster

Step 3 - Verify with plan: Run terraform plan in both old and new configs. Old config should show the resource as removed (no changes, just gone from state). New config should show NO changes (resource imported correctly). If new config shows changes, your resource config does not match reality - fix it before proceeding.

Step 4 - Update CI/CD pipelines: The old pipeline that managed everything now needs to be split into separate pipelines for each state file. Update and test in staging before doing production.

Pro tip: Do this as a 2-step process over 2 sprints. Sprint 1: add the resource to the new state (import) without removing from old state. Sprint 2: once new state is stable, remove from old state. Briefly managed in both places but much safer.

Terraform plan shows no changes but your production infrastructure is different from what the code describes. Why can this happen?

▼
This is terraform drift that Terraform itself is not detecting - a dangerous scenario. Multiple causes:

Cause 1 - Resource is managed outside Terraform: Someone created a security group rule manually in the AWS console. Terraform only knows about what it created. It cannot detect manually added rules because it does not track the full resource - only what it created.
Fix: Import the resource or use terraform plan -refresh-only which forces refresh of actual state.

Cause 2 - State file is stale: The state file has not been refreshed. In Terraform 0.15+, plan automatically refreshes. In older versions you needed terraform refresh manually.

Cause 3 - Ignoring changes with lifecycle ignore_changes:
lifecycle { ignore_changes = [tags, ami] }
If someone added this to ignore tag changes, Terraform will never show a diff even if someone manually changed all the tags. Very easy to forget this is in place.

Cause 4 - Provider bug or API inconsistency: Sometimes the AWS provider normalises values differently than the API returns them. For example, CIDR blocks: you wrote 10.0.0.1/24 but the API normalises to 10.0.0.0/24. Terraform computes the diff as equal even though the values look different.

Cause 5 - Data sources are cached: Data sources are computed at plan time. If the data changed since the last plan, you get stale data.

Detection strategy: Use AWS Config rules and CloudTrail to detect configuration drift independently of Terraform. Alert on any AWS API call that is not initiated by your CI/CD system. This catches the cases Terraform misses.

Grafana and Prometheus - Real-World Interview Questions

Your Grafana dashboard is showing gaps in metrics every 2-3 hours. Prometheus data is there but Grafana is not displaying it. What is wrong?

▼
Metric gaps in Grafana while Prometheus has the data usually come from a few specific causes:

Cause 1 - Grafana session/token expiry: If the Prometheus data source uses a token for authentication, and that token expires every few hours, Grafana will fail to query. Check the data source health in Grafana > Configuration > Data Sources. Look for auth errors.

Cause 2 - Prometheus retention or compaction: Prometheus compacts data in the background. During compaction, recent data may not be queryable for a brief window. The gap pattern (every 2-3 hours) matches the default compaction interval. Check Prometheus logs for compaction events.

Cause 3 - Prometheus scrape target down: The application pod restarted every 2-3 hours (OOM kill, liveness probe failure). Check: kubectl get pod --field-selector=status.phase=Running and compare restart count over time. If the pod restarts, the counter resets and there is a gap in the metric.
Fix: check WHY the pod is restarting. Do not fix Grafana - fix the underlying restart.

Cause 4 - Staleness handling: Prometheus marks a metric as stale if the target has not been scraped for 5 minutes. After that, Grafana shows a gap. This happens if the scrape interval is 30s but the target becomes unavailable for more than 5 minutes.

Cause 5 - Clock skew between Prometheus and K8s nodes: If node time is drifting, Prometheus timestamps are inconsistent. Grafana queries by time range - if the timestamps do not match, you see gaps. Fix: ensure NTP is synchronised across all nodes.

Debugging approach: Query the metric directly in Prometheus UI for the time range where the gap appears. If data exists in Prometheus but not Grafana, it is a data source connection issue. If data does not exist in Prometheus, the scrape target was down.

Chaos Engineering - Testing Reliability Before Users Find the Bugs
Chaos engineering is deliberately injecting failures to discover weaknesses before they cause real incidents. In financial services, run chaos experiments in staging first, then production during low-traffic windows with a kill switch ready.

Chaos Experiment 1 - Pod Kill (Most Common)

# Using Chaos Mesh (open source, Kubernetes-native)
apiVersion: chaos-mesh.org/v1alpha1
kind: PodChaos
metadata:
  name: payments-pod-kill
  namespace: production
spec:
  action: pod-kill
  mode: one               # kill one pod at a time
  selector:
    namespaces: [production]
    labelSelectors:
      "app": "payments-service"
  scheduler:
    cron: "@every 10m"    # kill a random pod every 10 minutes

What to watch: Does the service recover in under 30 seconds? Does HPA respond? Do alerts fire correctly? If the service is down for 2 minutes after a single pod kill, you have a reliability problem to fix.

Chaos Experiment 2 - Network Latency Injection

kind: NetworkChaos
metadata:
  name: payments-network-delay
  namespace: production
spec:
  action: delay
  mode: all
  selector:
    namespaces: [production]
    labelSelectors:
      "app": "payments-service"
  delay:
    latency: "500ms"      # add 500ms to all outbound network calls
    jitter: "100ms"
  duration: "5m"
  direction: to

What to watch: Does the application have proper timeouts? Does it handle slow downstream gracefully? Does it return 503 instead of hanging indefinitely? Are circuit breakers firing?

kind: StressChaos
metadata:
  name: payments-cpu-stress
spec:
  mode: one
  selector:
    namespaces: [production]
    labelSelectors:
      "app": "payments-service"

  duration: "5m"

What to watch: Does HPA scale out before latency degrades? Does the K8s CPU limit throttle the container? Do readiness probes fail and remove the pod from service?

Chaos Engineering Golden Rules

1. Always have a kill switch: be able to stop the experiment instantly
2. Define steady state BEFORE the experiment: what does healthy look like? Error rate under 0.1%, P99 under 200ms
3. Start small: one pod, not the whole deployment. One service, not the whole cluster
4. Run in staging first: prove the experiment works and the recovery works
5. Document every experiment: what you tested, what you expected, what actually happened, what you fixed

Your EKS pods can reach the internet but cannot reach the RDS database in the private subnet. How do you diagnose this?

▼
Classic VPC connectivity debugging. Systematic approach:

Step 1 - Verify the basic network path exists:
kubectl exec -it POD -- nc -zv DB-ENDPOINT 5432
If this fails, it is a network issue not an app config issue.

Step 2 - Check Security Groups: The most common cause. The RDS Security Group must have an inbound rule allowing port 5432 from the EKS pod security group (or CIDR). Check:
aws ec2 describe-security-groups --group-ids SG-ID
With VPC CNI, pods get real VPC IPs but may use a different security group than the node. Check the pod's ENI: kubectl describe pod POD | grep -i annotation - look for the ENI security group annotation.

Step 3 - Check Route Tables: The private subnet where pods live needs a route to reach the data subnet where RDS lives. If they are in the same VPC, this is automatic. If different VPCs, check VPC peering or Transit Gateway routes.

Step 4 - Check NACLs: NACLs are stateless. If there is a NACL on the RDS subnet, it must explicitly allow both inbound 5432 from the pod CIDR AND outbound ephemeral ports (1024-65535) back to the pod. This is the NACL trap people forget.

Step 5 - Check RDS Public Accessibility:
aws rds describe-db-instances --query DBInstances[*].PubliclyAccessible
Should be false for production. If false and pods are in the same VPC, the routing must work through private IPs.

Step 6 - Enable VPC Flow Logs and check: Flow logs show ACCEPT or REJECT for every packet. This is the definitive proof of where the traffic is being blocked.

Explain the difference between Security Groups and NACLs. When does the NACL trap catch people?

▼
This is one of the most common AWS networking interview questions:

Security Groups (SG):
- Applied at the ENI (Elastic Network Interface) level - per instance or per pod
- STATEFUL: if you allow inbound port 443, the return traffic is automatically allowed without an explicit outbound rule
- Rules are ALLOW only - no explicit deny. If a rule is not present, traffic is denied by default
- Can reference other SGs as source/destination - this is powerful for micro-segmentation

NACLs (Network Access Control Lists):
- Applied at the SUBNET level - affects all traffic in/out of the entire subnet
- STATELESS: you must explicitly allow BOTH directions
- Rules support both ALLOW and DENY - useful for blocking known malicious IPs
- Evaluated in order by rule number (lower number wins)

The NACL Trap people always fall into:
You add an inbound rule to allow port 5432 on the database subnet NACL. The connection still fails.
Why? Because NACL is stateless. The database sends the response back on an ephemeral port (1024-65535). The return NACL rule for outbound ephemeral ports is missing. The response packets are dropped.

Fix: Add an outbound rule on the database subnet NACL:
Rule: Allow TCP 1024-65535 to the application subnet CIDR

Mental model: Security Group = a smart bouncer at the door (remembers who came in and lets them out). NACL = a checkpoint at the neighbourhood entrance (checks every car in both directions, remembers nothing).

What is IRSA and how does it work? Why is it better than using IAM access keys in pods?

▼
IRSA = IAM Roles for Service Accounts. It is the mechanism that lets EKS pods assume AWS IAM roles without any static credentials.

How it works (the full flow):
1. EKS cluster has an OIDC (OpenID Connect) identity provider configured
2. A Kubernetes Service Account has an annotation pointing to an IAM role ARN
3. The IAM role has a trust policy that allows the OIDC provider to assume the role
4. When a pod starts with that Service Account, the EKS Pod Identity webhook automatically mounts two environment variables: AWS_ROLE_ARN and AWS_WEB_IDENTITY_TOKEN_FILE
5. The AWS SDK automatically reads these and calls STS AssumeRoleWithWebIdentity to get temporary credentials
6. The temporary credentials expire after 15 minutes and are automatically refreshed

Trust policy on the IAM role:
{"Effect":"Allow","Principal":{"Federated":"arn:aws:iam::123:oidc-provider/..."},"Action":"sts:AssumeRoleWithWebIdentity","Condition":{"StringEquals":{"oidc.eks.region.amazonaws.com/id/CLUSTER:sub":"system:serviceaccount:production:payments-service-sa"}}}

Why better than access keys:
- No secret to rotate, store, or leak in code
- Credentials are short-lived (15 min) - even if stolen, they expire quickly
- Fine-grained: each pod gets only the permissions it needs (not one shared key for everything)
- Auditable: CloudTrail shows which pod assumed which role and when
- Revocable: remove the trust policy and the pod loses access immediately - no key rotation needed

How do you design AWS infrastructure for disaster recovery with RTO under 15 minutes and RPO under 30 seconds?

▼
For RTO 15 min and RPO 30 seconds, you need active-active or warm standby with synchronous replication:

Architecture choice: Warm Standby (cost-effective for most financial services)

Data replication (RPO = 30 seconds):
- Aurora Global Database: primary in ap-south-1, secondary in ap-southeast-1. Replication lag is typically under 1 second. RPO = ~1 second, well within 30 second requirement
- DynamoDB Global Tables: active-active, sub-second replication. RPO = seconds
- ElastiCache: Redis Global Datastore for cross-region replication
- S3 Cross-Region Replication: near-real-time object replication

Compute standby (RTO = 15 minutes):
- Secondary region has a scaled-down EKS cluster running (minimum nodes, minimum replicas)
- Auto Scaling Groups configured but at 20% of production capacity
- When failover triggered: scale up ASG to full capacity (takes 3-5 minutes for spot instances)
- EKS pods are already deployed and running - just scaling from 2 to 20 replicas

Failover automation:
- Route 53 health checks on the primary ALB endpoint
- If health check fails for 3 consecutive checks (90 seconds): automatic DNS failover to secondary region
- DNS TTL set to 60 seconds for fast propagation
- Lambda function triggered by health check failure to scale up secondary region

Total RTO breakdown: Health check detection: 90s + DNS propagation: 60s + Compute scale-up: 180s + Pod startup: 60s = approximately 6.5 minutes. Well within 15 minutes.

Testing: Quarterly DR drill: failover to secondary, validate all transactions work, fail back. Archive drill report for business continuity audit evidence.

What is the difference between AKS System Node Pool and User Node Pool and why does it matter?

▼
This is an AKS-specific question that tests real production experience:

System Node Pool:
- Runs kube-system pods: CoreDNS, kube-proxy, metrics-server, Container Insights agent
- Has a taint: CriticalAddonsOnly=true:NoSchedule
- Must always have at least 1 node running (cannot scale to zero)
- Use Standard_D2s_v3 or similar - does not need to be large

User Node Pool:
- Runs your application workloads
- Can scale to zero (cost saving for dev/staging during off-hours)
- Can be Spot instances (60-70% cost saving for stateless workloads)
- Can have multiple pools with different VM sizes (GPU pool for ML, high-memory for analytics)

Why the separation matters in production:
Without separation, your payment service and CoreDNS share the same nodes. If a misbehaving payment pod causes OOM on the node, CoreDNS gets evicted too. Now ALL pods in the cluster lose DNS resolution. Everything breaks.

With system pool tainted: your app pods cannot be scheduled on the system pool even by accident. CoreDNS is protected. System components stay stable regardless of what application workloads do.

Setting it up:
az aks nodepool add --cluster-name mycluster --resource-group rg --name apppool --node-count 3 --node-vm-size Standard_D4s_v3 --mode User --spot-max-price -1

How does Azure Workload Identity work and how is it different from Pod Managed Identity (the old way)?

▼
Workload Identity is the modern, secure way for AKS pods to authenticate to Azure services without credentials:

Old way - Pod Managed Identity (AAD Pod Identity):
- A DaemonSet intercepts IMDS requests from pods and swaps identities
- Required privileged DaemonSet with access to Azure AD
- Had a known security issue: any pod on the node could potentially steal another pod's identity
- Microsoft deprecated this approach

New way - Workload Identity:
1. AKS cluster has OIDC issuer URL configured
2. You create a managed identity in Azure
3. You create a federated credential linking the managed identity to a specific K8s ServiceAccount
4. Pod runs with that ServiceAccount
5. Azure AD Workload Identity webhook injects AZURE_CLIENT_ID, AZURE_TENANT_ID, AZURE_FEDERATED_TOKEN_FILE into the pod
6. Azure SDK uses these to call Azure AD and get short-lived tokens

Setup commands:
az identity create --name payments-identity --resource-group rg az aks update --enable-oidc-issuer --enable-workload-identity az identity federated-credential create --name payments-federated --identity-name payments-identity --issuer $(az aks show --query oidcIssuerProfile.issuerUrl -o tsv) --subject system:serviceaccount:production:payments-sa

Why better: No DaemonSet, no privileged access. Each pod gets its own identity. Token exchange happens at the Azure AD level, not on the node. Completely audit-able. Same principle as AWS IRSA.

Explain Harness Continuous Verification (CV). How is it different from just watching Grafana after a deploy?

▼
Continuous Verification is what makes Harness worth the cost in regulated environments:

What CV actually does:
1. Before the deployment, CV establishes a baseline by analysing the last N deployments and their metrics (error rate, P99 latency, log patterns)

3. It uses ML-based anomaly detection to compare canary metrics against the baseline

- Manual watching requires an engineer to be available and alert during every deployment - not scalable
- Manual watching is subjective - "is this error spike normal or concerning?" requires experience
- CV is automated, consistent, and documented. Every verification result is stored as evidence
- CV fires the rollback in 2 minutes. A human checking Grafana might take 10 minutes to notice and act
- In financial services: CV output becomes part of the change record evidence for SOX audit

CV data sources Harness supports:
Splunk, Prometheus, New Relic, Datadog, AppDynamics, Dynatrace, Stackdriver, CloudWatch

Interview Questions - ArgoCD and GitOps

Explain GitOps and how ArgoCD implements it. What is the pull model and why is it more secure than a push model?

▼
GitOps is the practice of using Git as the single source of truth for both application code AND infrastructure state:

The core principle: The desired state of your production cluster is fully described in Git. Any change to production is a Git commit. The cluster continuously reconciles itself to match what is in Git.

Push model (traditional CI/CD, Jenkins/Harness):
Pipeline runs → pipeline has credentials to access the cluster → pipeline pushes changes to cluster
Security risk: Your CI system needs cluster credentials stored as secrets. If the CI system is compromised, an attacker can deploy anything to production.

Pull model (ArgoCD/Flux):
ArgoCD agent runs INSIDE the cluster → agent watches the Git repo → agent pulls and applies changes
Security advantage: No cluster credentials stored in CI. The CI pipeline only has permission to write to Git. ArgoCD running inside the cluster has cluster-admin, but that is normal for a workload running in the cluster. CI never touches the cluster directly.

ArgoCD reconciliation loop:
Every 3 minutes (default): ArgoCD compares the live cluster state to the desired state in Git
If different (drift): ArgoCD either auto-syncs (if configured) or shows OutOfSync status in UI
If auto-sync: ArgoCD applies the change, records who made the Git commit, and stores the diff in the application history

Full audit trail: Every production change is a Git commit with author, timestamp, and commit message. This is perfect for SOX audit: "who changed production and when?" - just check Git log.

What is the App of Apps pattern in ArgoCD and when do you use it?

▼
The App of Apps pattern solves multi-cluster and multi-service management at scale:

The problem without App of Apps:
You have 30 microservices across 3 clusters. That is 90 ArgoCD Application CRDs to manage manually. When you add a new environment (staging-asia), you have to create 30 more Application CRDs by hand. Error-prone and does not scale.

App of Apps solution:
Create one root ArgoCD Application that points to a Git directory containing other Application manifests. When ArgoCD syncs the root app, it creates and manages all the child Applications automatically.

Directory structure:
apps/ root-app.yaml # ArgoCD watches this dir payments-service.yaml # ArgoCD Application for payments auth-service.yaml # ArgoCD Application for auth fraud-service.yaml # etc. services/ payments/ # actual Helm chart values auth/

Adding a new service: Just add a new Application YAML to apps/ directory. ArgoCD picks it up automatically on the next sync. No manual ArgoCD UI work needed.

Adding a new cluster: Point a new ArgoCD instance at the root apps/ directory with a different values file. All 30 services are deployed to the new cluster automatically.

Ordering with sync waves: Use argocd.argoproj.io/sync-wave: "1" annotations to control deployment order. Deploy the database before the application. Deploy the ConfigMap before the Deployment.

How do you handle secrets in GitOps? You cannot commit secrets to Git but ArgoCD needs them in the cluster.

▼
This is the fundamental GitOps security challenge. Three approaches used in production:

Approach 1 - Sealed Secrets (Bitnami):
- SealedSecrets controller runs in the cluster. It has a private key.
- You encrypt the secret with the public key: kubeseal --format yaml secret.yaml > sealed-secret.yaml
- Commit the SealedSecret (encrypted blob) to Git safely
- ArgoCD deploys the SealedSecret. The controller decrypts it inside the cluster.
- Risk: if you lose the private key, all sealed secrets are unrecoverable. Backup the key.

Approach 2 - External Secrets Operator (recommended):
- Secrets stay in AWS Secrets Manager / Azure Key Vault / HashiCorp Vault
- You commit an ExternalSecret CRD to Git: "create a K8s Secret from this path in Secrets Manager"
- External Secrets Operator reads from the secret store and creates the K8s Secret in the cluster
- Rotation: when the secret rotates in Secrets Manager, ESO automatically updates the K8s Secret
- Nothing sensitive in Git. The ExternalSecret CRD is just a pointer, not the value.

Approach 3 - Vault Agent Injector:
- HashiCorp Vault runs separately
- Vault Agent sidecar is injected into pods
- The sidecar authenticates to Vault using the pod's service account token
- Vault injects secrets directly into the pod's filesystem (not as K8s Secrets)

Production recommendation: External Secrets Operator with AWS Secrets Manager or Azure Key Vault. It is the cleanest approach: secrets stay in a managed secret store, rotation is automatic, and nothing sensitive ever touches Git.

Interview Questions - Ingress and Networking

What is the difference between a Kubernetes Service (ClusterIP), NodePort, and LoadBalancer? When do you use Ingress instead?

▼
Understanding K8s networking layers is fundamental:

ClusterIP (default):
- Creates a stable virtual IP accessible only inside the cluster
- kube-proxy maintains iptables rules mapping the ClusterIP to pod IPs
- DNS: payments-service.production.svc.cluster.local resolves to the ClusterIP
- Use for: service-to-service communication inside the cluster
- Problem: not accessible from outside the cluster

NodePort:
- Exposes the service on a static port (30000-32767) on EVERY node
- Traffic to any node IP on that port → forwarded to the service
- Use for: development, testing, when you do not have a cloud load balancer
- Problem: exposes ports on all nodes, not production-grade

LoadBalancer:
- Provisions a cloud load balancer (AWS ELB, Azure LB) automatically
- The cloud LB gets a public IP and forwards to NodePort behind the scenes
- Use for: exposing a single service to the internet
- Problem: one load balancer per service = expensive (each LB costs ~$20/month on AWS)

Ingress:
- ONE load balancer (Ingress Controller) handles routing for MANY services
- Path-based routing: /api/payments → payments-svc, /api/auth → auth-svc
- Host-based routing: api.example.com → service-a, admin.example.com → service-b
- TLS termination at the Ingress Controller level
- Much more cost-effective: one ALB instead of 30
- Use for: production, multiple services behind one entry point

A request is reaching your ALB but returning 502. How do you systematically debug this?

▼
502 from ALB means the ALB reached the backend (pod) but got an invalid response. Systematic debug:

Step 1 - Check ALB target group health:
aws elbv2 describe-target-health --target-group-arn ARN
Are targets showing as healthy? If unhealthy: ALB cannot forward to the pod. The pod health check is failing.

Step 2 - Check what health check the ALB is using:
ALB checks a specific path (e.g. /health or /actuator/health). Check:
- Is the path correct in the IngressClass/TargetGroupBinding annotation?
- Is the app returning 200 on that path? Try: kubectl exec -it POD -- curl localhost:8080/health
- Is the health check port correct? Must match containerPort

Step 3 - If targets are healthy but 502 persists:
The pod is responding to the health check but not to the actual request. Causes:
- Pod is starting up and not fully ready (race condition between readiness and health check)
- Pod is returning 502 itself (it has a downstream error - check pod logs)
- Connection is being reset by the pod (check for connection limits, thread pool exhaustion)
- Timeout: ALB default idle timeout is 60 seconds. If the app takes longer, ALB returns 502. Check ALB access logs for elb_status_code=502 and target_status_code=- (never reached pod) vs target_status_code=200 (pod returned 200 but ALB timed out)

Step 4 - Enable ALB access logs:
alb.ingress.kubernetes.io/load-balancer-attributes: access_logs.s3.enabled=true,access_logs.s3.bucket=my-bucket
Access logs show: client IP, target IP, target port, request time, status codes. Gold for debugging.

Step 5 - Check Security Group:
ALB Security Group must allow outbound to the pod port. Pod Security Group must allow inbound from ALB SG.

What is a service mesh and when do you actually need one? What problem does Istio solve?

▼
Service meshes are often over-used. Here is an honest assessment:

What Istio actually does:
- Injects an Envoy proxy sidecar into every pod (you did not ask for it, it is injected automatically)
- All traffic between pods goes through the sidecars
- The sidecars handle: mTLS, retries, circuit breaking, traffic shifting, observability

Problems Istio solves:
1. mTLS everywhere: Without Istio, pod-to-pod traffic is unencrypted inside the cluster. With Istio, every connection is mutually authenticated and encrypted. Required for PCI-DSS in some interpretations.
2. Traffic management without code changes: Canary 10% to new version via Istio VirtualService - no code change needed in the app.
3. Circuit breaker at the infrastructure level: If payment-gateway is slow, Istio detects it and opens the circuit. No Hystrix library needed in your Java code.
4. Observability: Golden signal metrics for every service-to-service call, automatically, without code instrumentation.

When NOT to use Istio:
- Small cluster (under 10 services): the complexity overhead outweighs the benefits
- Istio adds ~50ms to pod startup and ~10% CPU overhead per pod from the Envoy sidecar
- Debugging becomes harder: is the 502 from your app or from Envoy?
- Learning curve is steep: CRDs (VirtualService, DestinationRule, ServiceEntry) are complex

Real recommendation: Use Cilium for network policy enforcement and basic encryption. Add Istio only when you need: traffic shifting for canary, circuit breaking, or strict mTLS for compliance reasons.

Interview Questions - Linux and Unix for SRE

A Linux server has load average of 24 but only 4 CPUs and CPU usage shows only 15%. What is happening and how do you investigate?

▼
This is the classic IO-wait vs CPU confusion question:

What this means: Load average counts ALL processes that are running OR waiting to run OR waiting for IO. CPU percentage only counts processes actively computing. 24 load with 15% CPU = 20+ processes are blocked waiting for something, not computing.

Step 1 - Confirm with vmstat:
vmstat 1 5
Look at the "wa" column (IO wait). If wa is above 30%, the server is stuck waiting for disk or network IO.

Step 2 - Identify the IO culprit:
iostat -x 1 5
Look for a disk with await above 100ms or %util at 100%. A disk at 100% util is the bottleneck.
iotop -o shows which processes are generating the IO.

Step 3 - Find D-state processes:
ps aux | awk '$8 ~ /D/ {print}'
D state = uninterruptible sleep = process is waiting for IO and cannot be killed. If many processes are in D state, you have severe IO bottleneck.

Common causes:
- Application writing too many logs to a slow disk (fix: reduce log verbosity, move to faster storage)
- NFS mount that has gone stale (fix: unmount and remount, or the NFS server is down)
- Failing disk causing retries: dmesg | grep -i error will show SCSI/ATA errors
- Elasticsearch or database doing heavy indexing

On a Kubernetes node: This exact scenario causes node pressure → K8s starts evicting pods. You will see eviction events in kubectl describe node NODE-NAME before the node becomes NotReady.

How do you find which process is holding a port and how do you handle zombie processes?

▼
Two common SRE operational questions:

Finding which process holds a port:
ss -tulnp | grep :8080 - modern way, fast
netstat -tulnp | grep :8080 - older systems
lsof -i :8080 - also shows process name
fuser 8080/tcp - just shows the PID

The output shows you the PID and process name. Then:
ps aux | grep PID - see what the process is
kill -15 PID - send graceful shutdown signal
kill -9 PID - force kill if -15 does not work

Real scenario: Application fails to start with "Address already in use". The previous instance crashed but did not release the port. This port conflict is resolved by killing the zombie process.

Zombie processes:
A zombie is a process that has finished executing but its parent has not read its exit status. It stays in the process table as "Z" (defunct).
ps aux | grep "Z" or ps aux | awk '$8 == "Z"'

Why zombies are usually not a problem: they consume no CPU, no memory, just one row in the process table. One or two zombies is harmless.

Why zombies become a problem: if the application creates thousands of zombie children (a bug in the parent process not reaping children), the process table fills up. New processes cannot be created. The system becomes effectively frozen.

Fix: you cannot kill a zombie directly - it is already dead. You must kill or fix the PARENT process so it reaps its children. Send SIGCHLD to the parent. If the parent is the application, fix the code to call wait()/waitpid().

In Kubernetes: this is why PID 1 in containers should use an init process (like tini). Tini reaps zombie child processes automatically.

2. Diagnose slow database connections:
tcpdump -i eth0 host DB-IP port 5432 -n -w /tmp/db.pcap
Open in Wireshark. Look at the time delta between the query being sent and the response arriving. If the delta is 2 seconds, the DB is slow, not the app.

3. Prove that traffic is being dropped by a firewall:
tcpdump -i eth0 'tcp[tcpflags] & (tcp-rst) != 0'
RST packets mean the connection is being actively rejected. If you see RST from the server, the service is not listening. If you see RST from a firewall, the traffic is being blocked.

4. Diagnose DNS resolution failures:
tcpdump -i eth0 port 53 -n
Is the DNS query being sent? Is the response coming back? What is the response? (NXDOMAIN = name does not exist, SERVFAIL = DNS server issue)

5. Verify TLS handshake:
tcpdump -i eth0 'tcp port 443 and (tcp[((tcp[12] & 0xf0) >> 2)] = 22)'
Filters TLS handshake packets. If you see ClientHello but no ServerHello, the server is not responding to TLS.

Important: On a production server with high traffic, add filters immediately. Unfiltered tcpdump can generate gigabytes of data per minute and fill the disk.

A Linux server has very high load average but low CPU usage. What is going on and how do you investigate?

▼
High load average + low CPU = processes waiting on IO, not computing.
Load average counts all running AND waiting (blocked) processes.
CPU % only counts actively computing processes.

This pattern usually means:
- Disk IO bottleneck: processes blocked waiting for disk reads/writes
- Network IO: processes blocked on slow network calls
- Uninterruptible sleep (D state): usually disk wait

Investigation steps:

Step 1 - Confirm with vmstat
vmstat 1 5
Look at 'b' column (blocked processes) and 'wa' column (IO wait %)
High wa (above 30%) confirms IO wait is the problem.

Step 2 - Find the IO culprit
iostat -x 1 5
Look for device with high await (ms per IO) and high util%.
A disk at 100% util is saturated.

iotop -o
Shows which processes are causing the IO.

Step 3 - Find D-state processes
ps aux | awk '$8 ~ /D/ {print}'
D state = uninterruptible sleep = waiting for IO.

Common causes in production:
- Application writing too many logs to a slow disk
- Elasticsearch or database doing heavy indexing
- NFS mount that has gone stale (worst case: D state hangs the process)
- Failing disk causing retries (check dmesg for disk errors)

dmesg | grep -i error
Check for SCSI errors or EBS volume degradation warnings.

How do you investigate a server that has run out of disk space?

▼
Disk full is a common P2 or P1 event depending on the service affected.

Step 1 - Confirm and identify which filesystem is full
df -h
Look for 100% or Use% above 95.

Step 2 - Find what is consuming the space
du -sh /* 2>/dev/null | sort -rh | head -20
Drill down into the largest directory:
du -sh /var/log/* | sort -rh | head -10

Step 3 - Common culprits
- Application logs not rotated: /var/log/myapp/*.log
- Core dump files: find / -name 'core.*' -type f
- Docker layers: docker system df then docker system prune
- Old journal logs: journalctl --disk-usage
Clear with: journalctl --vacuum-size=500M
- Temp files: ls -lh /tmp

Step 4 - Immediate relief
Compress and move old logs:
gzip /var/log/myapp/app.log.2024-01-14
mv *.gz /mnt/archive/

Delete confirmed unnecessary files.
Do NOT blindly delete files you are unsure about.

Step 5 - Root cause fix
Add logrotate config if missing.
Add CloudWatch disk alert at 80% (before it hits 100%).
In K8s: set resource requests on PVCs and add StorageClass limits.
Long-term: evaluate whether the volume needs to be resized (AWS: extend EBS volume).

DevOps Principles - Mental Model
DevOps is not a technology - it is a way of working. The three core insights: (1) small, frequent changes are safer than large, infrequent ones; (2) the team that builds a service should run it in production; (3) feedback loops between production and development should be as short as possible. Everything else - CI/CD tools, monitoring, on-call - is in service of these three ideas.

Elite: Multiple times per day
High: Once per week to once per month

Interview Questions - DevOps

What is DevOps and how is it different from what teams did before it existed?

▼
DevOps is not a tool or a job title - it is a cultural and engineering movement:

Before DevOps (the wall of confusion):
Development team wrote code. When ready, they threw it "over the wall" to Operations. Operations team deployed it to production. When it failed in production (which it often did because Ops environment was different from Dev), Dev blamed Ops for breaking it. Ops blamed Dev for writing bad code. Release cycles were quarterly or annual because each release was a major risky event.

What DevOps changed:
1. You build it, you run it: The team that writes the service is also responsible for running it in production. Development engineers carry the pager. This immediately incentivises writing reliable, observable code.
2. Continuous Delivery: Small, frequent deployments instead of large, infrequent ones. Smaller changes are easier to test, easier to roll back, and fail in smaller ways.
3. Infrastructure as Code: Operations work treated like software development: version controlled, tested, reviewed in pull requests.
4. Feedback loops as tight as possible: Developers see production metrics within minutes of deployment. If something breaks, they know immediately and fix it immediately.

The Three Ways (Gene Kim's model):
Flow (fast left-to-right delivery), Feedback (fast right-to-left signals), Continuous Learning (experiment and learn from failure)

How SRE implements DevOps: SRE is Google's specific implementation of DevOps principles. Error budgets make reliability quantitative. Postmortems make learning from failure systematic. Eliminating toil makes automation the default.

What is the difference between CI, CD (Continuous Delivery), and CD (Continuous Deployment)?

▼
These three terms are often confused. The distinction matters:

CI - Continuous Integration:
Every developer integrates their code into the shared main branch at least daily. Every integration is automatically built and tested. The goal is to detect integration problems as early as possible. Without CI, developers work in isolation for weeks and then have a painful merge.
Tools: Jenkins, GitHub Actions, GitLab CI

Continuous Delivery:
Code is always in a deployable state. Every successful CI build produces an artifact that COULD be deployed to production. The actual deployment requires a human approval (button click, pull request merge, etc.).
Key insight: The decision to deploy is a business decision, not a technical one. The technical capability to deploy safely is always there.

Continuous Deployment:
Every successful CI build is automatically deployed to production - no human approval needed. This requires very high confidence in your automated tests and deployment pipeline.
Used by: Netflix, Amazon, companies with mature testing and feature flags.
Not appropriate for: Financial services (regulated change management requires approval), safety-critical systems

Which to use in financial services:
CI: always
Continuous Delivery: deploy frequently (multiple times per day to staging), but require CAB/ITSM approval for production (ServiceNow CHG)
Continuous Deployment: not used for regulated production environments - change management regulations require documented approval

The important nuance: You can have very fast Continuous Delivery. Harness + ServiceNow can auto-create, route for approval, and auto-close a CHG in 15 minutes. That is Continuous Delivery that also satisfies SOX requirements.

What is DevSecOps and how do you embed security into a CI/CD pipeline without slowing teams down?

▼
DevSecOps = security shifts left, from the end of the release cycle to the beginning:

The problem with traditional security:
Security review happened at the end: build the feature, test it, hand it to the security team, wait 2 weeks for pen test results, then discover you need to rewrite half the authentication layer. By that time, the developer has moved to a different project.

Security at each CI/CD stage (with approximate time cost):

Pre-commit (developer's laptop):
- Git hooks: detect secrets before they are committed (gitleaks, trufflehog) - 1 second
- IDE plugins: SonarLint highlights security issues as you type - 0 seconds overhead

Pull Request / CI build (fast gates, under 5 minutes):
- SAST (Static Application Security Testing): SonarQube, Semgrep - scan for SQL injection, XSS, hardcoded secrets - 2-3 minutes
- Dependency scan: OWASP Dependency Check, Snyk - find CVEs in your libraries - 1-2 minutes
- Secret scanning: detect any secrets that slipped through - 30 seconds

Docker build stage:
- Container vulnerability scan: Trivy, Grype - scan the image for OS and library CVEs - 1-2 minutes
- Image signing: Cosign - prove this image was built by your CI pipeline - 30 seconds

Deployment stage:
- OPA/Kyverno admission policies: reject deployments without resource limits, without non-root user, with :latest tags
- Infrastructure scan: Checkov scans Terraform before apply - 30 seconds

Production (continuous):
- Falco: runtime security - detects suspicious behaviour in running containers (shell spawned, unexpected network connection)
- AWS Security Hub / Defender for Cloud: continuous compliance assessment

How to avoid slowing teams down:
Make security gates fast (under 5 minutes total for all checks). Fail on CRITICAL and HIGH CVEs only (not medium - too many false positives). Allow teams to accept risk for specific CVEs with documented justification. Give developers actionable fix instructions, not just "you have a vulnerability".

Chapter 1 — SRE Fundamentals and Culture

What is the difference between SRE and DevOps? Are they the same thing?

▼
They are related but not the same:

DevOps is a cultural movement and set of practices: break the wall between Dev and Ops, automate everything, deliver frequently, take ownership of what you build. It is a philosophy, not a specific job role.

SRE is Google's specific implementation of DevOps principles. It is a prescriptive engineering discipline with specific practices: SLOs, error budgets, toil elimination, blameless postmortems. SRE answers the question: "How do you operationalise DevOps with rigour and measurement?"

Key difference in practice:
DevOps teams may define reliability goals loosely ("we want high availability"). SRE teams define them mathematically (99.9% availability = 43.8 minutes downtime/month error budget). Error budget policy then drives engineering decisions: if budget is at risk, stop shipping features and fix reliability.

What to say in an interview: "DevOps is the cultural philosophy — build it, run it, automate it. SRE is the engineering discipline that makes reliability measurable and actionable. I apply SRE practices — SLOs, error budgets, blameless postmortems, and toil elimination — within a DevOps cultural framework where my team owns the full lifecycle of the services we build."

How do you build a culture of reliability in a team that has never practised SRE?

▼
This is asked to test whether you can lead change, not just implement tools:

Step 1 — Measure first, preach later: Do not walk in talking about SLOs. Measure what is actually happening: how many incidents per week, average MTTR, how many alerts per day, how many hours the team spends on manual toil. Show the data. Data creates urgency without blame.

Step 2 — Start with one small win: Pick the most painful recurring incident. Fix the root cause permanently. Document what you did. Show the before/after: "We used to get paged 3 times a week for this. Now: zero." This builds credibility and trust.

Step 3 — Introduce blameless postmortems: After the next incident, run a blameless RCA meeting. Focus on system conditions, not people. For many teams, this is the single most culturally significant SRE practice. When engineers stop fearing blame, they report problems earlier and honestly.

Step 4 — Make SLOs real: Define one SLO for the most important service. Track it on a dashboard everyone can see. When the error budget is at 50%, have a conversation: should we ship the planned feature or fix reliability? This makes reliability a business conversation, not an engineering complaint.

Step 5 — Reduce on-call pain: Fix the noisiest alerts. Teams that trust their SRE lead are ones whose on-call rotation gets better, not worse, over time.

What is toil and how do you decide what to automate first?

▼
Toil definition (Google SRE book): Manual, repetitive operational work that scales linearly with service growth, has no lasting value, and could be automated. Running a script to restart a service every Monday morning is toil. Writing the script that automatically detects and restarts it is eliminating toil.

How to identify toil: Track every operational task your team does for two weeks. Log: what was done, how long it took, and whether the same thing happened before. Anything appearing more than once per week is a toil candidate.

Prioritisation framework (2x2 matrix):
High frequency + High time cost = Automate immediately (e.g. manual incident ticket creation)
High frequency + Low time cost = Automate next (e.g. daily disk check email)
Low frequency + High time cost = Automate eventually (e.g. quarterly SSL cert renewal)
Low frequency + Low time cost = Probably not worth automating

What I automated in my last role: Our team was spending 6 hours/week manually creating ServiceNow incident tickets from monitoring alerts. I built a Python Lambda: CloudWatch alert → SNS → Lambda → auto-create ServiceNow INC with pre-populated severity, CI from CMDB, and runbook link. MTTA dropped from 8 minutes to under 2 minutes. 300 hours/year reclaimed for actual engineering work. Two other teams adopted the same pattern.

Explain the concept of error budget and how you use it to make release decisions.

▼
Error budget math: SLO = 99.9% availability → Error budget = 0.1% = 43.8 minutes/month. If your service has already been unavailable for 44 minutes this month, the error budget is exhausted.

How error budget drives decisions (this is the key insight):
Error budget bridges the gap between engineering (who wants to ship features fast) and reliability (who wants to prevent incidents). Instead of "ops says no" and "dev says ship", you have a shared budget that both teams agree governs decisions.

Budget healthy (above 50% remaining): Ship features aggressively. Experiment. Take calculated risks. Reliability is fine.
Budget warning (below 50% with 2 weeks left): Reliability review meeting. Be cautious about risky releases. Start working on known reliability issues.
Budget exhausted: Feature releases FREEZE. Every engineer focuses on reliability. No exceptions. This is the policy that needs business leadership sign-off before SRE can enforce it.

Real example to share: "In a previous role, our payments service error budget was at 90% consumed by the 20th of the month. The product team wanted to release a new payment flow feature on the 22nd. We used the error budget conversation to explain: if this release causes any incidents, we breach the SLA we have committed to customers. We delayed by one sprint, used that time to fix two known reliability issues, and the next month we had budget to spare."
Chapter 2 — Monitoring, Observability and Alerting

What is the difference between monitoring and observability? Why does it matter?

▼
Monitoring: Collecting predefined metrics and alerting when they breach thresholds. You know what questions to ask in advance. You build dashboards for things you already know can go wrong.

Observability: The ability to understand the internal state of a system from its external outputs (metrics, logs, traces), without needing to know in advance what might go wrong. An observable system lets you ask NEW questions about its behaviour during an incident.

Why the distinction matters in practice:
With monitoring only: your error rate alert fires. You check the dashboards you already built. If the problem matches a pattern you anticipated, you find the cause. If it is a new failure mode, you are blind.

With observability: during the same incident, you can query across logs, metrics, and traces ad-hoc. "Show me all requests that took over 2 seconds AND had a DB query count above 10 AND came from European users." This question was not anticipated when the dashboards were built. Observability makes it answerable.

The three pillars:
Metrics: tell you something is wrong. Error rate up.
Logs: tell you what happened. Stack trace showing which exception.
Traces: tell you where in the system it happened. Which microservice, which DB query, which downstream call.

Interview answer tip: "We had monitoring in place but not observability. We had dashboards for the scenarios we anticipated. When a novel failure mode appeared — a specific combination of user region, payment method, and time of day — our dashboards showed nothing wrong because we had not built that panel. I introduced distributed tracing with New Relic APM, which let us trace individual user sessions and find the exact code path that was failing. Monitoring tells you the house is on fire. Observability tells you which room, why, and how the fire started."

Your P99 latency is rising but error rate is still zero. How do you investigate?

▼
P99 rising without errors is a canary signal of an emerging problem. Structured approach:

Step 1 — Characterise the latency spike: What percentile? P50 flat but P99 rising = outlier requests are slow, not all requests. Which endpoints? histogram_quantile(0.99, sum(rate(http_request_duration_seconds_bucket[5m])) by (le, handler)) When did it start? Correlate with the Grafana deployment annotations.

Step 2 — Correlate with deployments: Check Harness or Jenkins for any deployment in the last 2 hours. If timing matches, consider rollback before investigating further. "If in doubt, roll it out."

Step 3 — Check the database: 70% of latency issues trace to the database. PostgreSQL: SELECT pid, now()-query_start AS duration, left(query,100) FROM pg_stat_activity WHERE state != 'idle' ORDER BY duration DESC; Look for queries over 1 second. Look for lock waits.

Step 4 — Check downstream dependencies: Is any external API (payment gateway, fraud service) experiencing latency? Check your service mesh (Istio) or New Relic APM for latency per dependency.

Step 5 — Check thread pool / connection pool: If thread pool queue is growing, requests are waiting for a thread. This shows as P99 latency without errors (the request eventually gets a thread and succeeds, just slowly). Check JVM thread pool metrics or hikaricp_connections_active in Prometheus.

Step 6 — Check GC pressure (Java): A full GC pause of 500ms causes P99 to spike without errors. Check GC logs: jcmd PID GC.heap_info or look at jvm_gc_pause_seconds_sum in Prometheus.

How do you design SLOs for a service you have never worked with before?

▼
SLO design for an unknown service — a common scenario when joining a new team:

Step 1 — Understand the user journey: What does the user actually do? For a payment service: user clicks pay → transaction is initiated → result is returned. The SLI must measure what the user experiences, not what the infrastructure is doing.

Step 2 — Look at historical data: Before setting a target, know what you are actually delivering. Query Prometheus for the last 90 days of availability and latency. Do not set an SLO of 99.9% if you have been delivering 98.5% — you will immediately be in breach.

Step 3 — Consult with the business: What does the customer contract (SLA) commit to? Your internal SLO should be tighter than the external SLA to give a buffer. If SLA is 99.5%, internal SLO might be 99.9%.

Step 4 — Define good events clearly: A good request is one that: returned HTTP 200 AND completed in under 500ms AND the response contained valid data. Make this concrete. Edge cases matter: is a 200 with an error in the body a good event? Probably not.

Step 5 — Start conservative: Begin with achievable SLOs. It is better to have a 99.5% SLO you always meet than a 99.99% SLO you breach every month. Tighten the SLO over time as reliability improves.

Step 6 — Get stakeholder agreement: SLOs only have power when engineering, product, and business all agree on the error budget policy. Without that agreement, the SLO is just a metric, not a decision-making tool.

What is Prometheus scrape interval and how does it affect alert accuracy?

▼
Scrape interval: How often Prometheus pulls metrics from a target. Default is 15 seconds. Every 15 seconds, Prometheus sends GET to /metrics and stores the response.

How it affects alert accuracy:
If your error rate spikes for 10 seconds and recovers, Prometheus may miss it entirely if the spike happens between scrapes. This is the fundamental sampling tradeoff: shorter scrape interval = more accurate detection = more storage and CPU load.

The rate() function and scrape interval: rate(http_requests_total[5m]) calculates per-second rate over the last 5 minutes. The window must be at least 4x the scrape interval for statistical stability. With 15s scrape interval, the minimum reliable window is 60s (4x15s). Using [30s] with a 15s scrape interval will give noisy, inaccurate results.

$__rate_interval in Grafana: Always use this instead of a hardcoded window. Grafana automatically calculates the correct interval based on the scrape interval configured in Prometheus. This means your dashboards work correctly whether the scrape interval is 15s or 30s without manual editing.

For_clause in alerts: for: 5m means the condition must be true for 5 consecutive minutes before firing. This prevents flapping on brief spikes. The tradeoff: a real outage takes 5 minutes to trigger the alert. For financial services, we use for: 2m for critical alerts to balance false positives with detection speed.

Recording rules for performance: Complex PromQL queries run on every scrape interval across all alert rules. For expensive queries used in dashboards and alerts, create recording rules that pre-compute the result every 30 seconds. This dramatically reduces Prometheus query load at scale.
Chapter 3 — Incident Management and Production Support

Walk me through your exact process when you get paged for a P1 at 3am.

▼
This is the most common senior SRE interview question. The interviewer wants a structured, confident answer:

T+0 — Acknowledge (within 2 minutes): PagerDuty acknowledgement stops the escalation chain. Open the incident Slack channel immediately: "IC: [my name], Scribe: [assign colleague], T+0: [time]. Investigating." Even if you know nothing yet, you have established command.

T+2 — Quick triage (2 minutes): One look at Grafana. I need three numbers: error rate (how bad?), traffic rate (is this a traffic spike?), and which service (what is the blast radius?). I do NOT start reading logs yet. Logs come after I understand the scope.

T+5 — Communicate to stakeholders: Status page update: "We are investigating an issue affecting [service]. Updates every 15 minutes." Business bridge notification if it is revenue-impacting. This happens WHILE technical investigation continues on a parallel track.

T+10 — Narrow to a hypothesis: One check: was there a deployment in the last 2 hours? If yes, rollback is my first mitigation action. I do not wait to confirm the deployment caused it — the cost of a false rollback is low, the cost of not rolling back if it is the cause is high.

T+15 — Mitigate, not fix: The goal is to stop the bleeding. Rollback, scale out, failover, feature flag off. I am not finding the root cause yet. I am restoring service.

T+20 — Confirm resolution: Grafana shows error rate returning to baseline. Manual end-to-end test. Status page update: "Issue resolved. Post-mortem within 48 hours."

Throughout: Scribe documents everything in real time. Every command run, every finding, every decision. This becomes the raw material for the postmortem.

What is a blameless postmortem and how do you run one effectively?

▼
Why blameless: If engineers fear punishment for mistakes, they hide problems. Hidden problems escalate. You want engineers to report failures immediately and honestly. Blameless postmortems create that safety. Blameless does NOT mean consequence-free — it means the focus is on systemic causes, not on finding a scapegoat.

What a good postmortem contains:
1. Incident summary: what broke, for how long, customer impact in business terms, not technical terms
2. Timeline: exact timestamps from logs and monitoring. Not "around 9am" but "09:14:32 UTC"
3. Root cause: technical cause. Use 5 Whys to get to the actual root, not just the immediate trigger
4. Contributing factors: things that made the incident worse or harder to detect
5. What went well: rollback worked fast, on-call responded in 90 seconds, runbook was accurate
6. What went poorly: alert fired 10 minutes late, runbook was outdated, DB team unreachable
7. Action items: specific, owned, dated. "Add connection pool alert" owned by Saroj, due in 3 days — not "improve monitoring"

Running the meeting: Separate the data collection (fact gathering) from the interpretation (why analysis). First, build the timeline collaboratively using logs. Everyone contributes what they saw and did. Second, run the 5 Whys analysis. Third, brainstorm action items without assigning ownership in the meeting — assign async afterwards.

Common mistake: Listing 20 action items that never get done because they have no owner and no deadline. Three action items that actually get completed are worth more than twenty that get forgotten.

How do you handle a situation where a teammate made a mistake that caused a production incident?

▼
This is a leadership and culture question disguised as a technical one:

In the moment (during the incident): Focus entirely on resolution. Never name the person in the incident bridge. Never say "X did this" or "X broke it." Say "a deployment" or "a configuration change." The incident is not the time for accountability — it is the time for recovery.

In the postmortem: Redirect any blame language immediately. If someone says "X deployed without testing," reframe: "The system allowed a deployment without automated testing gates. How do we add that gate?" The postmortem question is always: what system condition allowed this to happen, regardless of which human triggered it?

After the postmortem: Privately check in with the teammate. Incidents that involve a teammate's change can be emotionally difficult even in blameless cultures. A simple "these things happen to all of us, the system had gaps that made this possible" goes a long way.

The systemic fix conversation: Use the incident as evidence for a process or tooling improvement. "Our CI pipeline did not catch this type of issue. Let us add an integration test." This turns the individual mistake into a team learning without blame.

What NOT to do: Never use the postmortem to demonstrate that you personally would not have made this mistake. Never say "if they had followed the runbook this would not have happened" as a conclusion — instead ask "why was the runbook not followed?" and fix the process that caused that.
Chapter 4 — Kubernetes Production Scenarios

You have 3 replicas across 3 AZs. An AZ fails. What happens exactly and how long does recovery take?

▼
This tests understanding of K8s failure handling at the infrastructure level:

Immediate impact (T+0): One AZ fails. The node in that AZ becomes unreachable. Your load balancer (ALB) has health checks. ALB detects the unhealthy target within 2-3 health check intervals (~30-45 seconds) and stops sending traffic to pods on that node. So traffic is immediately degraded but 2 of 3 pods (in the other AZs) are still serving.

K8s node failure detection (T+0 to T+40s): kubelet stops sending heartbeats. API server waits node-monitor-grace-period (default 40 seconds) before marking the node NotReady.

Pod eviction (T+40s to T+5m30s): Node marked NotReady. K8s waits pod-eviction-timeout (default 5 minutes) before evicting pods — this prevents unnecessary pod shuffling for brief network hiccups.

ReplicaSet controller acts (T+5m30s): Sees 2 running pods, desired is 3. Creates a new pod. Scheduler tries to assign it to a healthy node in the remaining AZs.

New pod starts (T+5m30s to T+7m): Pod starts, passes startup probe, passes readiness probe, is added to ALB target group. Full capacity restored.

Total impact window: 30-45 seconds of reduced capacity (2/3 pods) from when the AZ fails until ALB removes the unhealthy target. Full capacity restored after ~7 minutes.

How to make this faster: Reduce pod-eviction-timeout to 30 seconds if you need faster recovery. Add topology spread constraints so the new pod is guaranteed to go to a different AZ. Ensure Cluster Autoscaler is running so new nodes are provisioned if the remaining AZs do not have enough capacity.

How do you implement and test K8s resource quotas and limit ranges in a multi-tenant cluster?

▼
Multi-tenant K8s resource management is a common senior SRE question:

ResourceQuota — namespace-level limits: apiVersion: v1 kind: ResourceQuota metadata: name: payments-quota namespace: production spec: hard: requests.cpu: "20" requests.memory: 40Gi limits.cpu: "40" limits.memory: 80Gi pods: "100" services: "20" persistentvolumeclaims: "10"
This prevents any single team from consuming all cluster resources. If the payments namespace tries to schedule a pod that would exceed these quotas, the pod stays Pending.

LimitRange — per-pod defaults and boundaries: apiVersion: v1 kind: LimitRange metadata: name: payments-limits namespace: production spec: limits: - type: Container default: # applied if container has no limits cpu: "500m" memory: "512Mi" defaultRequest: # applied if container has no requests cpu: "100m" memory: "128Mi" max: # no container can exceed this cpu: "4" memory: "8Gi"

Why LimitRange is critical: Without it, a developer deploys a container with no resource limits. It consumes all available CPU on the node. Other pods are throttled or evicted. LimitRange ensures every container has at minimum the default values.

Testing quotas: Deploy a pod that requests more than the quota allows. Verify it stays Pending. Check events: kubectl get events -n production | grep "exceeded quota". Alert when namespace quota reaches 80%: query kube_resourcequota{type="used"} / kube_resourcequota{type="hard"} > 0.8 in Prometheus.

What happens when a Kubernetes node runs out of memory and what is the eviction sequence?

▼
This is a common production scenario that tests deep K8s knowledge:

Node memory pressure sequence:

Stage 1 — kubelet detects memory pressure: kubelet monitors node resource usage every 10 seconds. When available memory drops below eviction-hard threshold (default 100Mi or 10%): node is tainted with NoSchedule. No new pods scheduled on this node. Existing pods continue.

Stage 2 — kubelet starts evicting pods: Pods are evicted in priority order based on QoS class:
First evicted: BestEffort pods (no requests or limits set). These have the lowest priority.
Second evicted: Burstable pods (requests less than limits) that are using more than their requested memory.
Last evicted: Guaranteed pods (requests equal limits). These are protected until the node is truly out of memory.

Within each QoS class: Pods using the most memory ABOVE their request are evicted first. A pod using 2x its memory request is evicted before a pod using 1.1x.

OOM Killer: If kubelet cannot evict fast enough and the kernel runs out of memory, the OOM killer activates. It kills processes (not pods gracefully) — often killing the container with exit code 137. This is the worst case and leaves containers in a bad state.

Prevention:
1. Always set memory limits on every container. This makes all pods Guaranteed or Burstable. BestEffort pods are dangerous in production.
2. Set memory limits slightly above memory requests (10-20% headroom). This allows the JVM/runtime to use extra memory for GC without immediately hitting the OOM killer.
3. Monitor node memory with alert at 80%. Give yourself time to scale before hitting eviction territory.
4. Use VPA recommendations to right-size memory requests based on actual usage.
Chapter 5 — CI/CD and Release Engineering

How do you handle database schema migrations in a zero-downtime deployment pipeline?

▼
DB migrations during zero-downtime deployments is one of the hardest engineering problems. The wrong approach causes data loss or downtime:

The core problem: During a rolling deploy, old pods (using old schema) and new pods (expecting new schema) run simultaneously. If you run a migration that drops a column while old pods are running, those old pods crash when they try to read the dropped column.

The Expand-Contract pattern (3 phases across 3 deployments):

Phase 1 — Expand (backward compatible migration): Only ADD things. Add the new column with a nullable type or sensible default. Example: ADD COLUMN new_payment_method VARCHAR(50) NULL Old code: ignores the new column (it does not know about it yet). No crash. New code: can be deployed and uses the new column. Migration runs as a K8s Job before the new pods start (init container pattern).

Phase 2 — Deploy new application version: New pods use the new_payment_method column. Old pods that may still be running during rolling update also work because the old column still exists. Both old and new code work with the expanded schema.

Phase 3 — Contract (cleanup migration, next sprint): After confirming all old pods are gone and the new version is stable (verify rollout complete, monitor for 24-48 hours): run the cleanup migration. DROP old column. ALTER column to NOT NULL if needed. This runs as a separate deployment, not the same one as Phase 1.

Migration tools: Flyway and Liquibase manage migration history and ordering. They track which migrations have run and prevent re-running. Run them as K8s Jobs with backoffLimit: 3 in the CI/CD pipeline before pod rollout.

Never do this: RENAME COLUMN or DROP COLUMN in the same deployment that introduces the new code. That is a guaranteed production incident.

How do you implement feature flags and how do they relate to your CI/CD pipeline?

▼
Feature flags are the bridge between code deployment and feature release:

What feature flags enable: Decouple deployment (putting code in production) from release (making features available to users). You can deploy code to production daily, but control who sees new features via flag configuration — no deployment needed to toggle a feature.

Flag types and use cases:
Release flags: new feature in development, disabled by default, enabled when ready. Zero-risk deployment.
Experiment flags: A/B test — 50% of users see variant A, 50% see variant B. Measure conversion.
Ops flags: kill switch for a problematic feature. Disable instantly without deployment.
Permission flags: enable feature for specific users/accounts (early access, beta testers).

Implementation options:
LaunchDarkly (SaaS): most mature, real-time flag evaluation, percentage rollouts, user targeting. Enterprise cost.
Unleash (open source): self-hosted, full-featured, Prometheus metrics built-in.
AWS AppConfig: native AWS, integrates with CloudWatch for automatic rollback if error rate increases after flag change.
Simple approach for small teams: config file in S3, Lambda reads it on a schedule, pods reload config without restart.

In CI/CD pipeline context: All new features behind flags by default. Pipeline deploys freely to production. Feature enabled via flag for 1% of users first → monitor → 10% → 50% → 100%. If metrics degrade at any stage, disable the flag (seconds, not a deployment). Harness Continuous Verification can monitor after a flag change, not just after a deployment.

Important: Flag cleanup is technical debt. Every flag added must have a cleanup ticket. Flags live at most 2 sprints. Old flags that no one dares remove are a codebase smell.
Chapter 6 — AWS Deep Dive

Explain the difference between horizontal and vertical scaling on AWS and when to use each.

▼
Vertical scaling (scale up): Replace the instance with a larger one. t3.medium → r6i.4xlarge. More CPU, more RAM, more network bandwidth.

When vertical scaling makes sense: - Stateful workloads that cannot easily be distributed (single-node legacy databases, monolithic apps)
- When the application is not designed for parallelism (single-threaded workloads)
- Quick fix: if you need more capacity immediately and cannot quickly change the architecture

Problems with vertical scaling: - Hard limit: you cannot scale beyond the largest instance type
- Downtime: changing instance type requires stopping the instance (unless using AWS Graviton and live resize)
- Single point of failure: all traffic on one (larger) instance

Horizontal scaling (scale out): Add more instances of the same size. 1 × r6i.4xlarge → 10 × r6i.xlarge.

When horizontal scaling makes sense: - Stateless applications (web servers, API servers, microservices)
- When you need high availability (multiple instances across AZs)
- When traffic is variable (Auto Scaling Groups scale in and out)
- Cost: 10 × smaller instances are often cheaper and more available than 1 × huge instance

On EKS specifically: Horizontal = HPA (add pods) + Cluster Autoscaler (add nodes). Vertical = VPA (resize pod CPU/memory requests). For most microservices on Kubernetes, horizontal scaling is the right answer because pods are stateless and designed to run in parallel.

Database exception: Read replicas are a form of horizontal scaling for reads. The primary database often needs vertical scaling as it cannot be horizontally scaled for writes without sharding.

What is the difference between AWS S3 storage classes and how do you use lifecycle policies to manage cost?

▼
S3 storage classes and use cases:

S3 Standard: Default. 3 AZ redundancy. Millisecond access. Most expensive. Use for: active data accessed frequently — application assets, current data, frequently accessed logs.

S3 Standard-IA (Infrequent Access): Same 3 AZ redundancy. Same millisecond access. Cheaper storage cost but retrieval fee per GB. Use for: backups you might need, older data still needed occasionally. Break-even vs Standard at about 30 days average access frequency.

S3 One Zone-IA: Single AZ only. Cheaper than Standard-IA. Data loss risk if AZ fails. Use for: reproducible data you can regenerate (thumbnails, derived analytics), non-critical backups where cost matters more than availability.

S3 Glacier Instant Retrieval: Archive storage. Millisecond retrieval. Very cheap. Use for: archive data accessed a few times per year. Quarterly reports, compliance archives.

S3 Glacier Flexible Retrieval: Hours to retrieve (expedited: minutes for a premium). Cheapest long-term archive. Use for: compliance data retention (7 year retention in financial services). You know you will almost never retrieve it.

S3 Intelligent-Tiering: Automatically moves objects between access tiers based on usage patterns. Small monitoring fee per object. Best for: data with unpredictable access patterns where you do not want to manually manage lifecycle rules.

Lifecycle policy example for financial logs: Days 0-30: S3 Standard (active investigation access) Days 30-90: S3 Standard-IA (occasional access for audits) Days 90-365: S3 Glacier Instant Retrieval (rare access, compliance holds) Days 365+: S3 Glacier Flexible Retrieval (7-year legal retention, almost never accessed)

This single policy on an S3 bucket can reduce storage costs by 80-90% for log data without any manual management.

How does AWS IAM Policy evaluation work? What is the difference between identity-based and resource-based policies?

▼
IAM policy evaluation is a common interview question that many candidates get wrong:

Identity-based policies: Attached to IAM users, groups, or roles. They define what that identity can do.
Example: An EC2 instance role with a policy allowing s3:GetObject on a specific bucket.

Resource-based policies: Attached to AWS resources (S3 bucket policy, SQS queue policy, KMS key policy). They define who can access that resource. They can grant access to principals from OTHER accounts (cross-account access).

Policy evaluation order (this is what interviewers test):
1. Explicit DENY wins everything. If any policy at any level has an explicit Deny, access is denied. Period.
2. SCPs (Service Control Policies): if the account is in an AWS Organization, SCP must allow the action. Even if the user's IAM policy allows it, if the SCP denies it, access is denied.
3. Permission boundaries: if set on a role, the role can only do what both the identity policy AND the permission boundary allow.
4. Identity-based policies: what the user/role is allowed to do.
5. Resource-based policies: for S3 cross-account, if the bucket policy grants access to an external account, that account can access even without an IAM policy on their side.

Default is DENY: Everything is denied unless explicitly allowed. You do not need an explicit Deny for things you have not allowed.

Common mistake: Assuming that because an IAM role has s3:* permissions, it can access any S3 bucket. But if there is an SCP at the organisation level that restricts S3 access to buckets in ap-south-1 only, the role cannot access buckets in us-east-1 even though the identity policy would allow it.
Chapter 7 — Terraform Production Scenarios

How do you test Terraform code before applying it to production?

▼
Terraform testing is often overlooked but critical in regulated environments:

Level 1 — Static analysis (seconds, runs in CI on every commit): terraform validate — checks syntax and internal consistency terraform fmt -check — enforces consistent formatting tflint — linting rules (deprecated resources, AWS naming conventions, unused variables) checkov or tfsec — security scanning: flag unencrypted S3, public security groups, missing deletion protection

Level 2 — Plan review (minutes, runs in CI on pull requests): terraform plan -out=tfplan stored as artifact Plan is posted as PR comment automatically via Atlantis or Terraform Cloud Humans review the plan before merging. Any unexpected destroy action = mandatory manual review.

Level 3 — Integration testing (minutes to hours, runs in staging): Terratest (Go library): write Go tests that apply Terraform, test the outputs, then destroy. Example: apply the module, verify the RDS endpoint is accessible, verify security groups are correct, then destroy.

Level 4 — Policy as Code (runs in CI, blocks apply if violated): OPA or Sentinel policies: - All S3 buckets must have encryption enabled - All EC2 instances must have deletion_protection or lifecycle prevent_destroy - All resources must have mandatory tags (Environment, Owner, Cost-Center) If policy fails, the CI pipeline blocks the merge. No human review needed for known violations.

Level 5 — Drift detection (runs daily in production): Scheduled pipeline: terraform plan -refresh-only and alert if output shows any changes. Any deviation from state = a notification to the SRE team to investigate.

What is Terraform state locking and what happens if a lock is stuck?

▼
Why state locking exists: Terraform state tracks what infrastructure exists. If two engineers run terraform apply simultaneously, they can corrupt the state file: both read the current state, make conflicting changes, and write back — the second write overwrites the first, losing infrastructure tracking.

How locking works with S3+DynamoDB backend: When any Terraform operation that modifies state begins (plan with refresh, apply, destroy): Terraform writes a lock record to the DynamoDB table. The record contains: who holds the lock, when it was acquired, which operation.
If another operation tries to acquire the lock while it is held: it fails immediately with "Error acquiring the state lock" and shows who holds it.
When the operation completes: lock is released from DynamoDB.

What to do if a lock is stuck: Locks get stuck when: a Terraform process crashes mid-run, network drops during apply, engineer force-kills the terminal.

# First: verify that NO active Terraform operation is running # Check with the engineer whose name is in the lock # Check CI/CD systems for running jobs # Only after confirming it is safe: terraform force-unlock LOCK-ID

Finding the lock ID: The error message when you try to run Terraform shows the lock ID. Or query DynamoDB directly:
aws dynamodb get-item --table-name terraform-state-lock --key '{"LockID":{"S":"prod/payments/terraform.tfstate"}}'

In financial services: Never force-unlock without verbal confirmation from the person who holds the lock or from their manager. A stuck lock may indicate an apply is still running in a non-terminal state. Unlocking while an apply is in progress corrupts the state file.
Chapter 8 — Docker and Container Security

What is the container build cache and how do you design Dockerfiles to maximise it?

▼
Build cache is what makes Docker builds fast or slow:

How the cache works: Docker evaluates each instruction in the Dockerfile. If the instruction and all its inputs (file contents, environment) are identical to a previous build, Docker reuses the cached layer. If any input changes, that layer and all layers below it are invalidated.

The critical rule — put things that change least at the top:
Wrong order (cache breaks on every source code change): COPY . . # copies everything including source - changes constantly RUN mvn install # re-downloads all dependencies every build (10+ minutes!) Correct order: COPY pom.xml . # copies only the dependency manifest - rarely changes RUN mvn dependency:go-offline # downloads deps, cached unless pom.xml changes COPY src ./src # copies source - changes frequently but deps already cached RUN mvn package # only compilation, not download - fast

Why this matters in practice: Without proper cache ordering: developer changes one line of code → Docker invalidates the cache at COPY . . → re-downloads all Maven/npm dependencies → 15-minute build. With correct ordering: dependency cache is preserved → build takes 90 seconds.

Multistage build cache sharing: Docker BuildKit allows using the cache from previous builds via registry: docker build --cache-from=myregistry/myapp:cache .. This shares the cache across different CI agents instead of each agent rebuilding from scratch.

Invalidating cache intentionally: Sometimes you want fresh dependencies (security patches). Use docker build --no-cache . or pass a build argument that changes: ARG CACHE_BUST=$(date +%s)

How do you scan Docker images for vulnerabilities in CI/CD and what do you do with the results?

▼
Container vulnerability scanning in a production CI/CD pipeline:

Tool choice — Trivy (open source, recommended): Fast (~30 seconds for a 500MB image), zero config, scans OS packages AND language libraries (Maven, npm, pip), integrated with Docker, free. trivy image --exit-code 1 --severity HIGH,CRITICAL myapp:1.2.3 exit-code 1 = pipeline fails on HIGH or CRITICAL CVEs. This is your quality gate.

What to do with scan results:
Immediately actionable: CVE in a library you directly depend on → update the library version in pom.xml/package.json, rebuild and rescan.
OS-level CVEs: update the base image. If using ubuntu:22.04, pin to the latest patched version. Use Dependabot or Renovate to automate base image updates.
CVE in a transitive dependency (library your library depends on): harder to fix. Options: update to a version of your library that uses a safe transitive version, or suppress with documented justification.

Suppression policy (important in regulated environments): Not all CVEs require immediate action. Create a .trivyignore file for CVEs that:
- Have no fix available yet
- Do not affect the codepath you use from that library
- Have been reviewed and accepted by the security team
Each suppression must have a documented justification and expiry date. This prevents .trivyignore from becoming a graveyard of unreviewed suppressions.

Beyond Trivy — Snyk in the IDE: Developers see vulnerabilities as they type in their IDE, not after a 30-minute build. Shift-left security: fix problems before they reach the pipeline.

Image signing with Cosign: After scanning and pushing a clean image, sign it: cosign sign --key cosign.key myregistry/myapp:1.2.3. In production, configure OPA/Kyverno to reject any pod spec that references an unsigned image. This ensures only images built and scanned by your CI pipeline can run in production.
Chapter 9 — Networking, Security and Compliance

How do you implement zero-trust networking in a Kubernetes cluster?

▼
Zero-trust = trust nobody by default, verify every request, assume breach:

Layer 1 — Network Policy (L3/L4 firewall): Default deny all: create a NetworkPolicy with podSelector: {} (all pods) and policyTypes: [Ingress, Egress] but no rules. Now no pod can talk to any other pod by default. Then explicitly allow only required communication paths: payments-service can call fraud-service on port 8080 payments-service can call postgres on port 5432 auth-service can call payments-service on port 8080 Nothing else is allowed. Any new service is denied by default until a policy is written.

Layer 2 — mTLS with Istio or Cilium: Mutual TLS means both the client and server authenticate each other using certificates. Without mTLS, even within the cluster, pod-to-pod traffic is plaintext. With mTLS, traffic is encrypted and authenticated — even a compromised pod cannot intercept or impersonate another service's traffic.

Layer 3 — Service Account and RBAC: Every pod runs with a specific Kubernetes ServiceAccount. RBAC policies define exactly what that ServiceAccount can do in the K8s API (which secrets it can read, which configmaps, etc). Ensure no pod has cluster-admin. Apply least-privilege: a payments service should not be able to read secrets from the auth namespace.

Layer 4 — IRSA/Workload Identity (cloud permissions): Each pod assumes a specific AWS IAM role/Azure Managed Identity with only the permissions it needs. Payments service: read from secrets manager, write to SQS. Auth service: read from DynamoDB, write to CloudWatch. No shared service account with broad permissions.

Layer 5 — Runtime security with Falco: Falco detects suspicious behaviour at runtime: a shell spawned inside a container, a binary written to disk, unexpected network connections. Alert on these events via PagerDuty. Even if a container is compromised, Falco detects anomalous behaviour.

How do you handle PCI-DSS or SOX compliance requirements in your infrastructure?

▼
Compliance requirements turn into specific engineering controls:

PCI-DSS key controls and their implementation:

Requirement: Protect cardholder data (CHD) Implementation: Tokenisation — never store PANs in your application. Use a PCI-certified tokenisation vault (Spreedly, Basis Theory, or build your own in a hardened CDE environment). Your application only ever sees tokens, never card numbers. Macie on S3 to detect if card numbers accidentally end up in logs.

Requirement: Restrict access to system components Implementation: Separate AWS account for the Cardholder Data Environment. VPC with no internet access. All access through PrivateLink or Direct Connect only. No SSH keys — use AWS Systems Manager Session Manager for emergency access (all commands logged to CloudTrail).

Requirement: Monitor and log all access Implementation: CloudTrail on all accounts with immutable log storage (S3 with MFA delete). VPC Flow Logs. All application logs shipped to centrally managed SIEM. Log retention: minimum 12 months immediately accessible, 3 years total.

Requirement: Regular security testing Implementation: Quarterly penetration testing by an approved scanning vendor (ASV). Trivy in CI for continuous CVE scanning. AWS Config rules for continuous compliance assessment.

SOX key controls:
Change management: every production change has a ServiceNow CHG with approvers documented. Harness auto-creates and closes CHGs.
Segregation of duties: developers cannot deploy to production. SRE deploys. Separate approval required.
Evidence collection: Terraform plan outputs, Harness pipeline logs, CloudTrail API calls — all preserved for auditor review.

Audit evidence in practice: Auditors will ask "show me every change made to production system X in the last 90 days." Your answer: pull CloudTrail logs filtered by the resource ARN. Pull ServiceNow CHG records. Pull Harness pipeline execution history. All three corroborate each other.

Vault — Secret Management

How does HashiCorp Vault work and how do you integrate it with Kubernetes?

▼
Vault is the enterprise standard for secrets management in regulated environments:

Core concepts: Vault is a secrets management system that stores, rotates, and controls access to secrets. Unlike K8s Secrets (base64 encoded, effectively plaintext), Vault encrypts secrets at rest and provides fine-grained access control, audit logging, and dynamic secrets.

Kubernetes authentication method: Pods authenticate to Vault using their Kubernetes Service Account token. Vault calls the K8s API to verify the token is valid and which service account it belongs to. Based on the service account, Vault returns a Vault token with specific policies attached.

Two integration patterns:
1. Vault Agent Injector (sidecar): Vault Agent is injected as a sidecar container via webhook. It authenticates to Vault and writes secrets to a shared volume that the main container reads. Secrets are never stored in K8s Secrets — they exist only in-memory in the shared volume.
annotations: vault.hashicorp.com/agent-inject: "true" vault.hashicorp.com/agent-inject-secret-db: "secret/payments/db" vault.hashicorp.com/role: "payments-service"

2. External Secrets Operator (ESO): Reads from Vault and creates K8s Secrets automatically. Simpler integration. Secrets DO exist as K8s Secrets (encrypted if etcd encryption is configured). Easier for teams not fully invested in Vault's native integrations.

Dynamic secrets (Vault's killer feature): For databases, Vault can generate a unique username/password per application request, valid for a specific TTL. Each pod gets its own DB credentials that expire when the pod stops. Compromise of one pod's credentials does not affect other pods, and credentials automatically expire.

Audit log: Every secret access is logged: which service, which secret path, which token, when. For SOX/PCI audits, this is the privileged access audit trail.
Service Reliability — Patterns

What is a circuit breaker and when do you use it? Have you implemented one?

▼
Circuit breaker prevents cascading failures across microservices:

The problem without circuit breakers: Payment service calls fraud-check service. Fraud-check is slow (300ms instead of 50ms). Payment service has 100 threads. Each thread is now blocked for 300ms instead of 50ms. Throughput drops by 6x. The fraud-check problem has cascaded into a payments problem. Every downstream service experiencing issues cascades upstream.

How circuit breaker works (3 states):
Closed (normal): Requests flow through. Failure rate is monitored.
Open (tripped): Failure rate exceeded threshold (e.g., 50% errors in 10 seconds). All requests immediately fail fast without calling the downstream service. Returns a fallback response. Error is short-lived: the caller continues serving other requests.
Half-open (recovery): After a timeout (e.g., 30 seconds), one test request is allowed through. If it succeeds, circuit closes and normal operation resumes. If it fails, circuit stays open.

Implementation options:
Resilience4j (Java library): add to pom.xml, annotate method with @CircuitBreaker. Zero infrastructure change needed.
Istio (service mesh): configure circuit breaking in DestinationRule — no code change in the application. Better for polyglot environments.
AWS App Mesh: similar to Istio, AWS-native.

Fallback strategies: Return cached last-known-good response. Return a degraded response (approve all transactions under $50 without fraud check). Return a clear error to the user so they can retry later. The strategy depends on the business requirements.

Interview answer: "We integrated Resilience4j circuit breaker on the fraud-check client in our payments service. Threshold: trip when 5 of the last 10 calls fail. Half-open after 30 seconds. Fallback: for transactions under $100, approve without fraud check and flag for manual review. This prevented the March incident where fraud-check had a 5-minute outage from cascading into a payments outage."

What is the Saga pattern and when do you use it for distributed transactions?

▼
The Saga pattern solves distributed transactions across microservices without two-phase commit:

The problem: In a monolith, a database transaction spanning multiple operations (debit account + create payment record + send notification) is atomic — all succeed or all rollback. In microservices, these operations span multiple databases and services. There is no distributed transaction coordinator that is both reliable and performant at scale.

Saga pattern: A sequence of local transactions. Each local transaction updates one service's database and publishes an event or message. The next service listens for the event and executes its local transaction. If any step fails, compensating transactions undo the previous steps.

Two implementations:
Choreography (event-driven): No central coordinator. Each service publishes events and subscribes to others' events. Simple, decentralised, but harder to trace and debug. Good for simple sagas with 2-3 steps.
SQS/Kafka: payment-initiated event → fraud-check service processes → fraud-cleared event → payment processor processes → payment-completed event → notification service sends confirmation.

Orchestration (centralised): A Saga Orchestrator service coordinates the saga. It tells each participant what to do and handles failures by triggering compensating transactions. Easier to observe and debug. Better for complex sagas with many steps.

Compensating transactions (the hard part): If payment fails after the account was debited: run compensating transaction to credit the account back. Compensating transactions must be idempotent — if they run twice due to a retry, the result must be the same.

When to use Saga: When you have business transactions that span multiple services and you cannot tolerate a 2-phase commit performance penalty. Common in: order processing, payment flows, booking systems.

How do you approach cloud cost optimisation as an SRE? What have you specifically done to reduce costs?

▼
Cost is increasingly part of the SRE mandate — reliability at sustainable cost:

Visibility first: You cannot optimise what you cannot see. Start with tagging: every resource tagged with team, environment, service, cost-center. AWS Cost Explorer filtered by these tags shows you which team and which service is spending what.

Request vs limit ratio: over-provisioned requests waste node capacity. Pod scheduled on a node based on requested CPU/memory. If request is 2x actual usage, you need 2x as many nodes.

How do you troubleshoot a memory leak in a production Java application?

▼
Memory leaks in Java typically show as gradually increasing heap usage followed by OOM kill.

Detection signals:
- Grafana: JVM heap used metric rising consistently over hours
- Pod restart count increasing (exit code 137 = OOM kill)
- GC logs showing longer and longer Full GC pauses
- kubectl describe pod POD-NAME shows OOMKilled in last state

Immediate mitigation:
Rolling restart to buy time: kubectl rollout restart deployment/myapp
This restores service but does not fix the root cause.

Investigation:

1. Capture a heap dump before OOM kill:
Add JVM flag: -XX:+HeapDumpOnOutOfMemoryError -XX:HeapDumpPath=/tmp/
kubectl cp POD-NAME:/tmp/heapdump.hprof ./heapdump.hprof

2. Analyse with Eclipse Memory Analyzer (MAT) or VisualVM:
- Look for the top retained heap object
- Common culprits: unbounded cache, event listener not deregistered,
static collection growing forever, ThreadLocal not cleared

3. Compare two heap dumps 30 minutes apart:
Objects growing between dumps = the leak.

4. Check GC logs:
-Xlog:gc*:file=/tmp/gc.log:time,uptime
Increasing Full GC frequency = heap filling up faster than it is cleared.

Long-term fix:
- Fix the leak in code
- Set K8s memory limit = JVM max heap + 200MB overhead (prevent OOM of the node)
- Add Grafana alert: heap usage above 80% for more than 10 minutes

How do you handle database connection pool exhaustion in production?

▼
Connection pool exhaustion means all connections are taken.
New requests queue up or fail immediately with 'too many connections' error.

Detection:
- Application logs: 'could not acquire connection within timeout'
- RDS CloudWatch: DatabaseConnections at max_connections limit
- Grafana: connection pool available = 0

Immediate mitigation:

Step 1 - Find what is holding connections
PostgreSQL:
SELECT pid, usename, application_name, state, query_start,
now()-query_start AS duration, query
FROM pg_stat_activity
WHERE state != 'idle'
ORDER BY duration DESC;

Kill connections idle for over 10 minutes:
SELECT pg_terminate_backend(pid)
FROM pg_stat_activity
WHERE state = 'idle'
AND now()-state_change > interval '10 minutes';

Step 2 - Rolling restart of application pods
kubectl rollout restart deployment/myapp
This releases all connections held by the application.

Short-term fix:
Increase max_connections on RDS (requires reboot - plan carefully).
Add PgBouncer or RDS Proxy as a connection pooler in front of the DB.
PgBouncer can multiplex thousands of app connections into tens of DB connections.

Root cause fixes:
- Fix connection leak in application code (connection not closed after use)
- Reduce connection pool size per pod if you have too many pods
- Add connection pool utilization alert at 80% to catch it before it exhausts

How do you store Kubernetes and AWS credentials securely in GitHub Actions?

▼
Best practice — OIDC with IAM Role (no static credentials):

GitHub Actions supports OIDC (OpenID Connect). GitHub acts as an identity provider. When the workflow runs, GitHub provides a short-lived OIDC token. AWS STS exchanges this token for temporary credentials to assume a specific IAM role.

Setup:
1. Create an IAM OIDC identity provider in AWS pointing to token.actions.githubusercontent.com
2. Create an IAM role with trust policy restricting to your specific repo and branch
3. In the workflow: permissions: id-token: write and use configure-aws-credentials action with role-to-assume

Trust policy condition:
"Condition": {"StringLike": {"token.actions.githubusercontent.com:sub": "repo:myorg/myrepo:ref:refs/heads/main"}}

Why this is better than GitHub Secrets with access keys:
No static credentials that can be rotated incorrectly or leaked. Credentials last 15 minutes (OIDC token lifetime). Each workflow run gets fresh credentials. Full audit trail in CloudTrail.

▼

What happens if the Kubernetes rollout fails mid-way through? How does the pipeline handle it?

▼
kubectl rollout status is the gate:
kubectl rollout status deployment/user-service -n production --timeout=5m
This command blocks until: all pods are Running and Ready (exit 0) or timeout is reached (exit 1).

On timeout/failure the pipeline step exits non-zero → subsequent steps do not run → GitHub Actions marks the job as failed → Slack notification fires with failure.

But K8s still has half-deployed pods. How do you handle this?
Add an automatic rollback step in the pipeline:
- name: Rollback on Failure if: failure() run: | kubectl rollout undo deployment/user-service -n production echo "Rolled back to previous version"

What K8s does internally during a failed rollout:
If maxUnavailable=0 is set, old pods keep serving traffic while new pods fail to become Ready. The old deployment is still functional. K8s does NOT auto-rollback — it waits. Your pipeline's rollback step triggers the explicit undo.

Kubernetes rollback vs pipeline rollback:
kubectl rollout undo: reverts to the previous ReplicaSet. Fast (seconds). Uses the stored previous manifest.
Pipeline re-run with old image tag: re-executes the full pipeline with a previous commit SHA. Slower but provides a full audit trail in CI.

Explain Readiness Probe vs Liveness Probe. What happens if you get them wrong?

▼
Readiness Probe — controls TRAFFIC:
If readiness probe fails: pod is removed from Service endpoints. Load balancer stops sending traffic to it. Pod keeps running. Container is NOT restarted.
Use for: checking if the app is fully initialised and ready to serve requests (DB connections established, caches warmed).

Liveness Probe — controls CONTAINER RESTART:
If liveness probe fails N times (failureThreshold): kubelet kills and restarts the container. Pod stays but container is restarted (restart count increments).
Use for: detecting deadlocks or states where the app is running but cannot recover without a restart.

What happens if you get them wrong:
Wrong 1 — Liveness probe too aggressive (short initialDelaySeconds): App starts, JVM takes 45 seconds to load. Liveness probe fires after 10 seconds, sees the app not responding, kills the container. CrashLoopBackOff. App never starts. Fix: use startupProbe for slow-starting apps.

Wrong 2 — Readiness probe on same path as liveness: If a dependency (downstream service) is slow, readiness probe fails → pod removed from load balancer. But pod is healthy. Now you have zero available pods even though all containers are running. Fix: readiness probe should only check local health, not downstream dependencies.

Wrong 3 — No probes at all: Pod becomes Ready immediately after container starts, before app is initialised. Load balancer sends traffic to a pod that cannot handle it yet. Users get 503s during every deployment. Fix: always set readiness probe with initialDelaySeconds appropriate for your app startup time.

Kafka — Producer, Consumer, Broker

Kafka Architecture Mental Model

Producer → publishes messages to a Topic
Topic → logically named channel. Divided into Partitions.
Partition → ordered, immutable sequence of messages. Each message has an offset. Partitions enable parallelism.
Broker → Kafka server that stores partition data. A cluster has multiple brokers for HA.
Consumer Group → group of consumers that divide partition consumption. Each partition is consumed by exactly ONE consumer in the group at a time.
Consumer Lag → difference between latest offset produced and latest offset consumed. High lag = consumers falling behind producers.

What is Kafka consumer lag and how do you monitor and fix it?

▼
What consumer lag means:
Consumer lag = latest offset in partition - consumer's current offset. If producer writes message 1000 and consumer has only read to 800, lag = 200. High lag means consumers cannot keep up with the production rate. Messages are piling up and processing is delayed.

Monitoring consumer lag:
Prometheus + kafka-exporter: kafka_consumer_group_lag metric. Alert when lag exceeds threshold for more than 5 minutes.
CLI check: kafka-consumer-groups.sh --bootstrap-server kafka:9092 --describe --group my-group
Shows: topic, partition, current-offset, log-end-offset, lag, consumer-id

Common causes and fixes:
Cause 1 — Consumer processing is too slow: A downstream API the consumer calls is slow. Fix: make the API call async or batch process.
Cause 2 — Insufficient consumer instances: 6 partitions but only 2 consumers. Fix: scale consumers to match partition count (maximum parallelism = number of partitions).
Cause 3 — Consumer is stuck: Consumer is in an infinite retry loop on a bad message (poison pill). Fix: implement a dead letter topic. After N retries, move the message to a DLT for later analysis.
Cause 4 — Producer burst: Traffic spike produced 10x normal volume. Fix: scale consumers temporarily with KEDA using Kafka lag as the scaling trigger. KEDA scales consumer pods based on lag metric directly.

What is a Kafka partition and how do you decide how many partitions to create?

▼
What partitions do:
A partition is the unit of parallelism in Kafka. Messages within a partition are strictly ordered. Messages across partitions have no order guarantee. One consumer per partition (in a consumer group) means partitions = maximum consumer parallelism.

How Kafka distributes messages across partitions:
If producer specifies a key: messages with the same key always go to the same partition (consistent hashing). All transactions for customer_id=12345 go to partition 3. Order guaranteed per customer.
If no key: round-robin distribution across partitions.

How to decide partition count:
Target throughput: partition count = max(target-throughput / throughput-per-consumer, target-throughput / throughput-per-producer)
Rule of thumb: start with 3-6 partitions for low-volume topics, 20-50 for high-volume. More partitions = more parallelism but more file handles, more replication overhead, longer leader election.

Can you change partition count after creation?
Yes, you can increase partitions. But this breaks key ordering for existing keys — previously, key X always went to partition 3. After increasing partitions, the same key may now hash to partition 7. This breaks ordering guarantees. For ordered processing, design the partition count upfront and do not change it.

A Kafka broker is down. What is the impact and what do you check?

▼
Immediate impact depends on replication factor:
Replication factor = 3 (standard): Each partition has 1 leader and 2 followers. If a broker goes down: partitions where that broker was the leader need a new leader elected. Kafka controller elects a new leader from the in-sync replicas (ISR). This takes seconds to minutes.
During leader election: producers writing to affected partitions get temporary errors (leader not available). Consumers may see brief interruption.

Checking broker health:
# Check all brokers are registered kafka-broker-api-versions.sh --bootstrap-server kafka:9092 # Check under-replicated partitions (critical alert) kafka-topics.sh --bootstrap-server kafka:9092 --describe --under-replicated-partitions # Check ISR state kafka-topics.sh --bootstrap-server kafka:9092 --describe --topic payments

Under-replicated partitions = critical alert:
If replication factor is 3 and a broker is down: partitions with that broker as a follower have ISR=2 instead of 3. They are under-replicated. If another broker dies now, you could have ISR=1 — data at risk. This is why you alert on under-replicated partitions and act fast to replace the dead broker.

Auto-recovery in Kubernetes:
Kafka brokers should run as a StatefulSet. If a pod dies, K8s restarts it with the same stable identity (kafka-0, kafka-1) and the same PVC. Kafka cluster rebalances automatically when the broker rejoins.
MongoDB — SRE Production Questions

Explain MongoDB Replica Set and what happens when the primary node fails.

▼
Replica Set architecture:
A replica set has: 1 Primary (handles all writes), 1+ Secondaries (replicate from primary, can handle reads if configured), optional 1 Arbiter (votes in elections but stores no data — used when you want odd number of votes without the cost of a full data node).

Replication: Primary writes to its oplog (operations log). Secondaries tail the oplog and apply the same operations. Replication is asynchronous by default. Lag between primary write and secondary apply = replication lag.

What happens when primary fails:
1. Remaining members detect primary is unreachable (heartbeat timeout ~10 seconds)
2. A new election is triggered. Any secondary can call for election.
3. A candidate secondary gets majority votes (majority of all members, not just alive members — this prevents split brain)
4. The winner becomes new primary. This process takes 10-30 seconds typically.
5. During election: writes are rejected (no primary available). Reads from secondaries may still work if read preference allows it.

Write concern for durability:
w: majority: write is acknowledged only after written to majority of replica set members. Prevents data loss if primary fails immediately after acknowledging a write that has not replicated yet.

Monitoring replica set health:
rs.status() # in mongo shell — shows all members, their state, and replication lag rs.printReplicationInfo() # oplog window size and replication lag

How do you troubleshoot slow MongoDB queries in production?

▼
Step 1 — Enable profiler to capture slow queries:
# In mongo shell db.setProfilingLevel(1, { slowms: 100 }) # Log queries over 100ms # Check slow query log db.system.profile.find().sort({ts:-1}).limit(10).pretty()

Step 2 — Use explain() to understand query plan:
db.payments.find({user_id: 12345, status: 'pending'}).explain('executionStats') # Look for: # COLLSCAN = full collection scan (bad for large collections) # IXSCAN = index scan (good) # nScanned >> nReturned = inefficient index or missing index

Step 3 — Add appropriate indexes:
# Create compound index for common query pattern db.payments.createIndex({user_id: 1, status: 1, created_at: -1}) # Background: true allows other operations during index build db.payments.createIndex({user_id: 1}, {background: true})

Step 4 — Check for index bloat:
Large indexes that do not fit in RAM cause disk reads on every query. Check with:
db.payments.stats().indexSizes # shows size of each index
If index size > available RAM: queries will be slow regardless of index design. Either reduce document count (TTL indexes to expire old data) or increase WiredTiger cache size.

Step 5 — Check for lock contention:
db.serverStatus().globalLock # shows lock queue and wait time db.currentOp() # shows currently running operations
Long-running writes cause readers to wait (document-level locking in WiredTiger, but some operations still block). Kill a long-running operation: db.killOp(opid)

A Java application has high CPU usage. Walk through your investigation.

▼
This is the most common Java support question for SREs. Structure your answer as an investigation workflow:

Step 1 — Confirm it is actually the JVM:
top -H -p PID # -H shows threads # Look for multiple threads with high CPU # Each thread's TID can be converted to hex for thread dump analysis

Step 2 — Capture a thread dump at peak CPU:
kill -3 PID # Sends SIGQUIT, JVM prints thread dump to stdout # Or: jstack PID > /tmp/thread_dump.txt # Or via JMX if running in container: jcmd PID Thread.print > /tmp/thread_dump.txt

Step 3 — Analyse the thread dump:
Look for: threads in RUNNABLE state consuming CPU. Specifically look for:
• GC threads running constantly → memory pressure, not enough heap
• Regex compilation in a hot loop → catastrophic backtracking
• Infinite loop in application code → usually a bug introduced in recent deploy
• JSON serialisation/deserialisation in tight loop → optimise with streaming parser

Step 4 — Check GC first (most common cause in microservices):
jstat -gcutil PID 1000 # GC stats every 1 second # If FGC (Full GC) count is climbing rapidly: # JVM is spending most time in garbage collection = thrashing # Heap is too small for workload # Fix: increase -Xmx or find memory leak

Step 5 — If recent deployment caused it:
Roll back first, investigate second. The root cause can wait 20 minutes. Service restoration cannot.

Explain Java heap, stack, metaspace, and common JVM memory errors.

▼
JVM Memory Regions:

Heap: Where objects live. Managed by GC. Divided into Young Generation (Eden + Survivor spaces) and Old Generation. Configured with -Xms (initial) and -Xmx (maximum). Most objects are short-lived and die in Young Gen.

Stack: Per-thread. Stores method call frames, local variables, references (not objects). Fixed size per thread (usually 256KB-1MB). StackOverflowError when recursive calls too deep.

Metaspace: Class metadata, method bytecode, static variables. Outside heap (native memory). Replaced PermGen in Java 8+. Grows dynamically by default. Can cause OutOfMemoryError: Metaspace if too many classes are loaded (common with dynamic class generation or classloader leaks).

Common Memory Errors:
java.lang.OutOfMemoryError: Java heap space
Heap exhausted. Either heap is too small (-Xmx too low) or there is a memory leak (objects accumulating without being GC'd).
Investigation: add -XX:+HeapDumpOnOutOfMemoryError. Analyse dump with Eclipse MAT or VisualVM.

java.lang.OutOfMemoryError: GC overhead limit exceeded
JVM spending more than 98% of time in GC but reclaiming less than 2% heap. Application is effectively frozen. Root cause: memory leak or heap too small.

java.lang.StackOverflowError
Recursive method call too deep. Usually a bug (infinite recursion). Increase stack size with -Xss only as temporary fix — find the recursion bug.

java.lang.OutOfMemoryError: unable to create new native thread
OS thread limit reached. Too many threads being created. Check with: ls /proc/PID/task | wc -l. Fix: thread pool configuration, check for thread leaks.
